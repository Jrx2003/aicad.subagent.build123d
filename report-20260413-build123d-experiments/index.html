01

Why Build123d helps this runtime

Explicit local frames

`Plane`, `Locations`, and centered-object geometry make it easier to map requirement-local coordinates into stable model-space placements.

This is exactly the contract that made `benchmark/runs/20260413_102600/L2_172` converge in one round.

Builder-native booleans

`Mode.SUBTRACT` and `Mode.INTERSECT` keep half-shell and annular-band logic inside one readable builder instead of depending on brittle chained state.

This is the clean repair lane behind `L1_218` and the half-shell tightening for `L2_130`.

Safe staging

`Mode.PRIVATE` lets us stage cavity or lip solids inside a live builder without mutating the host before we are ready.

This matters for shelled boxes, lids, and feature families with opening-face semantics.

06

Current judgment

The migration already has enough substance for a report: Build123d is improving the runtime in the exact places we care about most, namely explicit local frames, builder-visible booleans, safe staging, and deterministic preflight repair surfaces.

The honest boundary is that the repo is not "done" just because benchmark anchors pass. The enclosure family still needs more generalized body/lid/top-face reasoning.

07

Next plan

Keep the demos small

Use the new demo suite as the stable explanation surface for Build123d adoption.

Generalize enclosure reasoning

Focus on shell-opening semantics, lid/body decomposition, and top-face fastener placement.

Use benchmark as a signal, not a target

Continue accepting only fixes that improve shared runtime contracts across multiple unseen prompts.
