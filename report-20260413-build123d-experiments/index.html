01

为什么 Build123d 对这个 runtime 更有帮助

局部坐标系更显式

`Plane`、`Locations` 和 centered host geometry 让 requirement 中的局部坐标 更容易映射成稳定的模型空间 placement。

这正是让 `benchmark/runs/20260413_102600/L2_172` 一轮收敛的关键契约。

Builder 原生布尔更干净

`Mode.SUBTRACT` 和 `Mode.INTERSECT` 可以把 half-shell、annular band 这类逻辑 放在一个可读的 builder 内完成，而不是依赖脆弱的链式状态。

这正是 `L1_218` 的修复路径和 `L2_130` 半壳 contract 收紧的基础。

安全的 staging 能力

`Mode.PRIVATE` 让我们可以在活动中的 builder 内先 staging cavity 或 lip solid， 而不会在时机未成熟时提前污染 host。

这对 shelled box、lid，以及带 opening-face 语义的 feature family 都很重要。

工件链路仍然可讲、可复盘

prompts/模型这一轮到底看到了什么。

actions/Build123d 写入是否成功，还是命中了结构化 lint / repair surface。
queries/写后 validator、kernel refresh 和 probe 拿回了什么证据。
trace/每一轮的决策链路和 stop reason。
outputs/ 与 evaluation/最终 STEP 产物以及 benchmark 对照证据。
06

当前判断

这次迁移已经有足够扎实的内容支撑一次汇报：Build123d 确实在我们最关心的几个点上改善了 runtime，包括显式 local frame、builder 可见的 boolean、safe staging，以及可确定的 preflight repair surface。

需要诚实说明的是：benchmark 锚点通过，并不等于仓库已经“全部完成”。enclosure family 仍然需要更泛化的 body / lid / top-face reasoning。
