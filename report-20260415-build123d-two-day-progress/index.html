这里说的 canary，到底是什么

canary 不是“随手挑几题 smoke 一下”，而是一组固定的小切片回归集，用来代表当前最关键的 failure family 和两三个相对稳定的 guardrail case。 这组 case 固定写在 `benchmark/canary_case_sets.json`，当前包含 `L1_122`、`L1_148`、`L1_157`、`L1_159`、`L2_88`、`L2_130`、`L2_149`、`L2_172`。

  • 它覆盖了三类问题：`validator mismatch / read-stall`、`runner exception`、`half-shell / path-sweep / countersink`。
  • 它的作用不是替代 full L1/L2，而是作为每次主路径改动后的第一道固定回归入口。
  • 定义文档在 `docs/cad_iteration/CANONICAL_BASELINE.md`，运行入口是 `benchmark/run_prompt_benchmark.py` 的 `--case-set canary`。

为什么要先固定 canary

因为 runtime 的问题往往不是“这一题过没过”这么简单，而是会反复出现在同一种控制流或同一个 feature family 上。 先固定 canary，后续每一次改动都能用同一把尺子回答三个问题：first write 是否更稳、validator 是否更容易闭环、repair packet 是否真的帮到收敛。

这也是为什么页面里既展示 sampled L1/L2，也展示 canary：前者证明仓库整体可运行，后者负责稳定暴露主矛盾。

实现细节 A / 写后收口

`agent_loop_v2.py` 具体改了什么，为什么能压住 read-stall

这一层不是简单加一句 prompt，而是直接改写后控制流。`_latest_validation_is_fresh_for_write(...)` 负责判断当前 validation 是否真的是“刚刚这次写入”的新结果； `_has_repeated_validation_without_new_evidence_after_write(...)` 负责识别“没有新写入、没有新 kernel/probe 证据，却还在重复 validation”的空转。

  • 当 fresh validation 已经 `complete`，stop reason 会直接落成 `post_write_validated_complete`。
  • 当 fresh validation 是 `insufficient_evidence`，runtime 不再继续 generic read/validation ping-pong，而是强制回到 semantic refresh lane。
  • 这条修改直接对应 `L1_122` 从 `20260415_203500` 的 `read_stall_gap`，变成 `20260415_205500` 的单轮 `PASS`。
实现细节 B / Build123d 契约

`tool_runtime.py` 不是“报错更多了”，而是在更早的位置把错法拦住

这轮对 `tool_runtime.py` 的增强，核心是把以前会浪费完整一轮 sandbox 的常见错法提前升格成 preflight lint，并给出对应 repair recipe。 这样模型不会先把 session 写坏，再靠下一轮猜为什么几何不对。

  • `invalid_build123d_contract.active_builder_temporary_primitive_arithmetic`：拦截 active `BuildPart` 里直接拿临时 `Cylinder` / `Cone` / `Box` 做后续布尔算术的错法。
  • `invalid_build123d_contract.explicit_anchor_manual_cutter_requires_subtract_mode`：拦截 explicit-anchor hole family 里忘记 `mode=Mode.SUBTRACT` 的手写 cutter。
  • `invalid_build123d_keyword.countersink_*_alias`：拦截 `CounterSinkHole(...)` 的错误关键字，比如 `angle=`、`head_radius=`，要求改回 Build123d 的字面契约。
实现细节 C / 语义状态

`blocker_taxonomy.py` + `feature_graph.py` 负责把“修对了”真正反映回 prompt 和 repair

仅有 lint 还不够，如果旧 blocker 继续挂在 `general_geometry` 或旧 family 上，下一轮 prompt 仍然会被过期信息带偏。 这轮修改把 `head_diameter_*`、`cone_angle_*`、`through_hole_diameter_*`、`countersink`、`conical_recess`、`activate_the_hole_wizard_or_the_revolved_cut_tool` 这类 blocker 重新映射回 `explicit_anchor_hole`，同时把已被新 validation 覆盖的旧 feature instance 标成 `resolved`。

  • 效果是：新的 countersink blocker 不会再被错误塞进 `general_geometry`。
  • 旧的孔位 blocker 不会继续留在 `domain_kernel_digest` 里污染后续 repair。
  • 这一点在 `L2_172` 的新旧 run 对照里非常明显，后面的案例 D 会给出完整证据链。

页面里的“问题归类”具体怎么做

这份报告里的归类不是人工写备注，而是 benchmark 聚合时从 case 产物里算出来的，核心代码在 `benchmark/run_prompt_benchmark.py`。 它分三层：

  • `status / failure_category`：先回答“哪一层先坏了”，例如 `RUN_ERROR`、`NO_STEP`、`VALIDATOR_MISMATCH`、`INCOMPLETE`。
  • `failure_cluster`：再回答“这类失败在行为上属于哪一族”，例如 `read_stall_gap`、`code_path_family_gap`、`runtime_gap`、`tool_gap`。
  • `recommended_fix_layer`：最后给出优先修哪一层，比如 `kernel_binding_gap`、`tool_runtime`、`validator`、`prompt/code path`。

三层归类分别看什么证据

`status / failure_category` 主要来自 return code、STEP 产物是否存在、validator 与 evaluator 是否分歧、run 是否 converged； `failure_cluster` 主要继承 runtime summary 里的聚合判断；`recommended_fix_layer` 会结合 validation lanes、domain-kernel mismatch 和 stop reason。

  • 如果 runner 非零退出，就是 `runner_exception / RUN_ERROR`。
  • 如果 evaluator 过了但 validator 没闭环，就是 `validator_evaluator_disagreement / VALIDATOR_MISMATCH`。
  • 如果有 STEP 但 inspection-only rounds 很多，就会落到 `excessive_reinspection / INCOMPLETE`，再进一步看是不是 `read_stall_gap`。

06 / 证据链路

这轮工作最大的变化之一，是效果已经能被完整复盘，而不是只能靠截图或口头描述。

如果需要把任何一条结论展开，可以按下面顺序取证，不必从目录树盲翻。

1. 决策层

plans/round_*_response.json

用于说明模型这一轮为什么选 `execute_build123d`、`query_feature_probes` 或 `validate_requirement`。

2. 执行层

actions/round_*_execute_build123d.json

用于说明 preflight lint、实际写入成功与否、结构化错误以及 snapshot geometry。

3. 验证层

queries/round_*_validate_requirement*.json

用于说明 check、blocker、clause interpretation、coverage confidence 为什么高或低。

4. 收敛层

trace/stop_reason.json + summary.json

用于说明这条 run 最终是 `post_write_validated_complete`、`max_rounds_reached` 还是其他停止原因。

5. 几何评估层

evaluation/benchmark_eval.json

用于说明 runtime validator 与 evaluator 是否一致，以及几何差异具体落在哪些面或 anchor 上。

6. 汇总层

brief_report.md + run_diagnostics.md + summary.json

用于说明整个 run root 的状态分布、主导 failure cluster、baseline 指标和待修重点。

最值得直接打开的几个目录

  • `benchmark/canary_case_sets.json`：看 canary 的固定 case 切片到底包含哪些 case。
  • `docs/cad_iteration/CANONICAL_BASELINE.md`：看 canary 与 baseline 的正式定义。
  • `benchmark/run_prompt_benchmark.py`：看 `status / failure_category / failure_cluster / baseline_metrics` 是怎样算出来的。
  • `benchmark/runs/20260415_205500/L1_122`：看 closure 收口。
  • `benchmark/runs/20260416_001500/L1_148`：看 runner 稳定性问题是如何被压平的。
  • `benchmark/runs/20260415_122001/L2_149`：看 path-sweep 成功闭环。
  • `benchmark/runs/20260415_164609/L2_172`：看 countersink 修复链条怎样从 lint 走到 validator complete。
  • `benchmark/runs/20260416_002500`：看 canary 总表。

这套证据链为什么重要

因为这代表后续优化不再需要“先靠人记住上次发生了什么”。每次改动都可以回到同一套 `plans → actions → queries → trace → evaluation → report` 链路里复盘。

07 / 当前判断

现在最值得汇报的，不是“已经全部做完”，而是“系统已经进入正确的收敛形态”。

最后一页把成果、边界和下一步一起说清，避免只报喜或只报问题。

已经拿到的成果
  • Build123d 主路径已经在 sampled L1/L2 上完整运行，仓库可跑性有实证。
  • canary、baseline、brief report、summary 的口径已经统一。
  • L1 的 validator/read-stall 家族已经明显后退，`L1_122` 和 `L1_148` 都给出了清楚的改善证据。
  • path-sweep 家族已经出现“成功闭环 + 暴露误判 + 继续收紧”的完整治理链条。
仍需继续推进的重点
  • `L2_172` countersink family 仍未收敛完成，下一阶段应继续围绕 countersink geometry recognition 与 explicit-anchor hole contract 推进。
  • `L2_149` 虽然已有成功 run，但 canary 仍显示 first-turn bias 和 code path 稳定性不够，需要继续提高 path-sweep recipe 命中率。
  • 后续优化仍应坚持 family-first 泛化，不应该回到针对单个 case 堆硬规则的老路上。
一句话收束

这两天最大的进展，不是多过了几题，而是 Build123d 迁移后的 runtime 已经开始具备“发现问题、归类问题、收紧契约、复跑验证”的稳定闭环能力。
