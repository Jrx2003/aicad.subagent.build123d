02 / 长期上限证据

为什么说 Build123d 的长期天花板更高？不是因为它“更新”，而是因为它更适合被 runtime 编程。

这部分只看源码和运行时表面。结论要靠产品代码证明，而不是靠 CAD 库口碑。

Prompt Contract src/sub_agent/prompts/codegen.md 
- Curve helpers such as `Polyline(...)`, `Line(...)`, `CenterArc(...)`, and `RadiusArc(...)`
  belong inside `BuildLine`, not directly inside `BuildSketch`
- `revolve(...)` uses `revolution_arc=` ...
- `CounterSinkHole(...)` is a `BuildPart` operation ...
- `Plane.XZ.offset(d)` shifts along Y, not Z

这不是单纯“教模型写代码”，而是在把几何语义拆成可判定的契约：曲线归属哪个 builder、revolve 的合法签名、countersink 的合法宿主、Plane.offset 的真实含义。

Runtime Skill src/sub_agent_runtime/skill_pack.py 
- choose the workplane whose normal matches the drill direction
- `CounterSinkHole(...)` belongs in `BuildPart`, not `BuildSketch`
- A safe first-pass half-shell skeleton is
  `Cylinder(...) -> subtract inner -> intersect half-plane -> add pad/lugs -> bore -> holes`
- For Build123d `revolve(...)`, ... do not invent `angle=`

Skill pack 的价值在于：同一条 family 规则可以同时作用在 first write、repair、probe follow-up 上，而不是只存在于单次 prompt 里。

Preflight Lint src/sub_agent_runtime/tool_runtime.py 
invalid_build123d_context.curve_requires_buildline
invalid_build123d_keyword.revolve_angle_alias
invalid_build123d_contract.directional_drill_plane_offset_coordinate_mixup
invalid_build123d_context.countersinkhole_requires_buildpart

这才是“长期上限更高”的第一条硬证据：Build123d 失败可以前移为结构化 lint，再绑定成 repair recipe。CadQuery 也能修，但它更多依赖链式 Workplane 经验和运行后读错。

证据 1

Build123d 把几何状态拆成了更多显式对象。

`BuildPart / BuildSketch / BuildLine / Plane / Axis / Locations / Mode.PRIVATE` 这些对象，使 runtime 能准确判断“错在 builder 上下文、坐标 frame、还是布尔顺序”。

证据 2

这些对象已经被我们接成了产品级修复面。

本轮不是临时补 prompt，而是把 revolve、BuildLine、countersink、directional drill 这些错误转成了可测试、可复用、可前置拦截的 runtime 规则。

证据 3

规则一旦对，整个 family 会一起受益。

`L2_88` 的前后对照就是证据：我们没有改 case 数据，也没有改 benchmark evaluator，只补了通用契约，结果 fresh run 从失败变成 1 轮满分。

03 / L2_88 前后对照

这是最关键的一条证据链：一次通用修复，把 Build123d 的 axisymmetric revolve family 从 fresh run 失败拉回 1 轮满分。

如果不能证明“产品级规则能快速解锁一个 family”，那“长期上限更高”就只是观点。这一节专门证明这件事已经发生了。

修复前 / 20260414_093041

fresh run 的失败不是几何太难，而是 API/上下文契约还没被前移。

这两条错误都不是 case 私有知识，而是通用 Build123d 契约：`revolve` 的正确签名、curve helper 的 builder 归属。

修复后 / 20260414_130005

加入两条通用规则后，同题 fresh run 一轮收敛。

1 轮收敛 4,462 tokens `eval_score=1.0` 
with BuildSketch(Plane.XZ):
    with BuildLine():
        ...
    make_face()
revolve(axis=Axis.Z)

新 run 不需要 probe 链，也不需要补充 repair packet。模型首轮就写出了正确的 Build123d recipe，说明规则已经进入主路径。

为什么这个前后对照是“长期上限”证据，而不是一次偶然命中？

  1. 修复前，错误集中在 Build123d 通用契约：`revolve(angle=...)` 和 `Polyline` builder 误用。
  2. 本轮修改没有碰 benchmark 数据，只在 `codegen.md`、`skill_pack.py`、`tool_runtime.py` 里补了通用规则。
  3. 相关测试 `78 passed in 1.38s`，证明这些规则已经进入产品表面，而不是 notebook 里的临时经验。
  4. 修复后，同题 fresh run 一轮通过并拿到 `STEP geometric signatures are closely aligned`。
本页结论

`L2_88` 证明了一个关键事实：Build123d 的失败面更容易被产品化编码，而一旦规则进入 runtime，这个 family 的能力会以“整类修复”的方式提升。

06 / Artifact 链路

为什么这些 run 值得汇报？因为它们不是只有最终分数，而是完整保留了“模型为什么走到这一步”的链路。

这一节用 `L2_88` 修复后 run 作为模板，说明 benchmark 里每类中间产物分别证明什么。

prompts/

模型这一轮到底看到了什么。比如 20260414_130005/L2_88/prompts/round_01_user_prompt.txt 里明确写着 front plane、centerline、closed profile、360 度 revolve，这决定了 runtime 应该激活 axisymmetric revolve family，而不是去猜别的 body family。

plans/

模型的决策摘要和工具载荷。真正值得看的字段不是“这轮有没有调用工具”，而是 decision_summary 与 tool_calls[0].arguments.code。例如 20260414_130005/L2_88/plans/round_01_response.json 明确写出 “Following the explicit revolve profile recipe...”，并且代码里真的出现了 BuildSketch(Plane.XZ) -> BuildLine() -> make_face() -> revolve(axis=Axis.Z)。

actions/

真正执行的代码与执行结果。这里要同时看 stderr 和 snapshot.geometry。修复前 20260414_093041/L2_88/actions/round_01_execute_build123d.json 里明确报 revolve(..., angle=360) 非法；修复后 20260414_130005/L2_88/actions/round_01_execute_build123d.json 的快照直接给出 solids=1、faces=6、bbox=[50,50,20]，说明它不只是“没报错”，而是真的生成了目标族几何。

queries/

validator、probe、kernel state 的读回证据。这里最关键的是 validate_requirement 的 checks[].evidence 与 clause_interpretations[]。例如 L2_172 的 Build123d 最终验证里，feature_hole_position_alignment 已 pass，但 feature_countersink 明确 fail，证据字符串直接写着 cone_like_face_present=False。这能把“孔位做对了”与“沉头语义没做对”清楚拆开。

trace/

完整的 round timeline。这里最值得看的不是时间戳本身，而是 stop_reason.json 和各轮工具切换。比如 20260414_093041/L2_88 的 stop reason 是 max_rounds_reached，而修复后 20260414_130005/L2_88 在 round 1 就变成了 post_write_validated_complete。这条变化说明规则已经进入首轮主路径。

outputs/

最终 STEP 和 geometry_info。汇报时如果只想看最终产物，这一层最直观；但如果要解释为什么变强，必须和 plans/trace/queries 连起来看。单独看 STEP 只能证明“最后长这样”，不能证明“为什么这次是系统变强了，而不是恰好命中”。

evaluation/

和 ground truth 的最终几何对比。这里要盯的字段是 scores.final_score、metrics.feature_anchor_rel_max、metrics.face_type_iou 和 difference_notes。例如 L2_130 虽然 validator pass，但 evaluator 仍明确指出 local feature-anchor deviation 0.119 is high；而 L2_88 修复后则直接回到 final_score=1.0。

07 / 当前判断

最后的结论必须同时诚实和可行动。

这部分给出汇报时可以直接说的话，以及接下来继续投入 Build123d 的正确方向。

可以直接说

Build123d 的长期投资价值已经被证明。

证明方式不是“它今天 benchmark 平均分领先”，而是：它让 runtime 能用更显式的 Plane / builder / boolean / local-frame 契约，把失败前移成 lint 与 repair recipe，再用一次通用修复快速拉起整个 revolve family。

不能过度说

不能说 Build123d 现在已经全面超过 CadQuery。

至少在本轮 `L2_130` 和 `L2_172` 上，这个结论不成立。CadQuery 在 half-shell 与 countersink family 上依然更稳。

真正要投的方向

继续投 family recipe，而不是投 case patch。

  • axisymmetric_profile：本轮已经证明 generic fix 有效。
  • half_shell + directional hole：继续强化 first-turn same-builder skeleton。
  • explicit_anchor_hole / countersink：统一 helper、宿主平面、corner-frame 到 centered-frame 的归一化。
下一步计划

如果继续推进 Build123d，不该再靠零散 prompt 文案，而应该继续加厚三层产品面。

  1. 继续把 Build123d 的 family 失败前移成 preflight lint，而不是浪费 sandbox 重试。
  2. 针对 half-shell 和 countersink family，补 first-turn canonical recipe，而不是等多轮 repair 才逼近正确写法。
  3. 继续保留同题 CadQuery 对照，确保每次“Build123d 更强”的说法都有 fresh run 证据支撑。
汇报建议

最稳妥的说法是：Build123d 还没在平均分上赢，但已经在产品化潜力上给出更硬的工程证据。

这份证据主要来自三条链：源码层的显式契约对象、preflight lint 与 repair recipe 的可编程失败面，以及 L2_88 这种“通用规则直接解锁一个 family”的前后对照。汇报时如果需要更细的证据，请直接引用同目录下的 shareable-report.md 和 detailed-evidence.md。
