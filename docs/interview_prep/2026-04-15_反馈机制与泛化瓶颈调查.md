# Build123d Runtime 反馈机制与泛化瓶颈调查

## 1. 结论摘要

当前系统已经不是“LLM 盲写代码然后靠运气跑通”的形态，而是一个带有读工具、验证工具、语义状态、修复面和局部编辑面的迭代 runtime。

但当前最明确的瓶颈也很清楚：

1. `LLM` 并不是完全不理解任务目标，主要问题是它对 `build123d` 的可执行表面仍会产生库级幻觉，导致第一轮容易写出错误 API、错误 builder 结构、错误局部坐标或错误 boolean 组织。
2. 反馈机制已经能提供相当多的几何证据，但这些证据还没有稳定到足以让所有高风险 family 都在第一轮进入正确 canonical lane。
3. 验证与修复当前是“混合驱动”而不是“纯 LLM 驱动”：
   - 几何证据抽取和大量 clause interpretation 仍以确定性规则为主
   - `validation_llm` 只在 `insufficient_evidence` 的窄场景下做补充裁决
   - repair surface 目前仍明显偏规则化

因此，真正需要做的不是继续到处补个案硬规则，而是把现有 hard rule 收敛成三类可维护资产：

1. `API / contract governance`
2. `evidence / interpretation registry`
3. `family repair packet / recipe registry`

这比继续把 case logic 散落在 `tool_runtime.py`、`skill_pack.py`、`service.py` 的长函数里更可维护，也更有利于复杂新案例的生成与修复。

## 2. 现在 LLM 在过程中间能否“理解模型”

可以，但这种“理解”不是直接读取 B-Rep 内核，而是通过一组结构化读工具和语义摘要间接理解。

1. `query_snapshot`
2. `query_sketch`
3. `query_geometry`

7. `query_kernel_state`
8. `validate_requirement`

2. [`src/sub_agent_runtime/tool_runtime.py:824`](../../src/sub_agent_runtime/tool_runtime.py)

### 2.2 这种“理解”具体包含什么

`query_geometry` 和 `query_topology` 并不是只返回一句摘要，而是能返回结构化对象。

例如：

1. `FaceEntity` 暴露 `area / center / normal / axis_origin / axis_direction / radius / geom_type / bbox`
2. `EdgeEntity` 暴露 `length / center / axis_origin / axis_direction / radius / geom_type / bbox`
3. `QueryTopologyOutput` 还会返回 `matched_ref_ids / candidate_sets / applied_hints`

对应合同在：

1. [`src/sandbox_mcp_server/contracts.py:313`](../../src/sandbox_mcp_server/contracts.py)
2. [`src/sandbox_mcp_server/contracts.py:343`](../../src/sandbox_mcp_server/contracts.py)
3. [`src/sandbox_mcp_server/contracts.py:1341`](../../src/sandbox_mcp_server/contracts.py)
4. [`src/sandbox_mcp_server/contracts.py:1400`](../../src/sandbox_mcp_server/contracts.py)

这说明当前模型并不是只能知道“模型对了/错了”，而是已经可以读取：

1. 面法向
2. 边长度
3. 圆柱/圆边半径
4. face / edge 的 step-local 引用
5. requirement-aware 的候选面/边集合

### 2.3 `query_kernel_state` 提供的不是几何，而是语义层理解

当前 runtime 还有一张独立于几何内核的语义状态表：`DomainKernelState`。

它会维护：

1. feature nodes
2. blocker nodes
3. evidence bindings
4. feature instances
5. repair patches
6. `FamilyRepairPacket`

对应定义在：

1. [`src/sub_agent_runtime/feature_graph.py:188`](../../src/sub_agent_runtime/feature_graph.py)
2. [`src/sub_agent_runtime/feature_graph.py:802`](../../src/sub_agent_runtime/feature_graph.py)

也就是说，LLM 看到的不是单一几何快照，而是：

1. 几何读工具给出的事实
2. validator 给出的 clause/blocker
3. domain kernel 整理出来的 family / anchor / repair state

所以严格来说，当前系统已经具备“中间过程理解模型”的能力，只是这种能力仍然依赖 read-model 和 semantic kernel 的质量。

## 3. 现在能否选中相关面和边进行针对性修改

可以，但只在一类受控场景里成立：`稳定宿主已存在` 之后的 bounded local finishing。

### 3.1 当前已经支持面/边级锚定的路径

`apply_cad_action` 的结构化 action 已经支持显式引用 `face_ref` 和 `edge_refs`。

最直接的例子在 action registry：

1. `FILLET` 优先使用 `edge_refs`
2. `CHAMFER` 优先使用 `edge_refs`
3. `HOLE` 支持 `face_ref`
4. `SPHERE_RECESS` 支持 `face_ref`
5. `CREATE_SKETCH` 也支持 `face_ref`

对应位置：

1. [`src/sandbox_mcp_server/registry.py:588`](../../src/sandbox_mcp_server/registry.py)
2. [`src/sandbox_mcp_server/registry.py:611`](../../src/sandbox_mcp_server/registry.py)
3. [`src/sandbox_mcp_server/registry.py:634`](../../src/sandbox_mcp_server/registry.py)
4. [`src/sandbox_mcp_server/registry.py:669`](../../src/sandbox_mcp_server/registry.py)

`face_ref / edge_ref` 的合法格式也被显式约束在：

1. [`src/sandbox_mcp_server/registry.py:34`](../../src/sandbox_mcp_server/registry.py)
2. [`src/sandbox_mcp_server/registry.py:1267`](../../src/sandbox_mcp_server/registry.py)

### 3.2 当前 runtime 已经明确把这个能力放进 skill guidance

例如：

1. “final local fillet/chamfer step，prefer `query_topology` + explicit `edge_refs`”
2. “host solid 存在后再做 local finishing”

对应位置：

1. [`src/sub_agent_runtime/skill_pack.py:529`](../../src/sub_agent_runtime/skill_pack.py)
2. [`src/sub_agent_runtime/skill_pack.py:1081`](../../src/sub_agent_runtime/skill_pack.py)

### 3.3 现阶段的限制

虽然支持 face/edge 级 targeting，但还不能夸大成“已经是一个成熟的交互式几何编辑器”。

现阶段的限制包括：

1. 这条路主要适用于 fillet / chamfer / hole / recess 这类局部编辑
2. 复杂主体拓扑错误、路径扫掠组织错误、半壳宿主组织错误，通常仍然要回到 `execute_build123d` 重写
3. 面/边 ref 是 step-local 的，需要稳定 session 和 fresh topology query 支撑，不能假设它们在任意大改之后仍稳定

所以当前能力更准确的表述是：

1. 系统已经支持“在已有稳定几何上做 topology-aware local edit”
2. 但还不支持把所有复杂错误都降解成局部 face/edge patch

## 4. 当前反馈机制到底是什么结构

### 4.1 反馈并不是一层，而是四层

当前反馈链路可以拆成四层：

1. `preflight governance`
2. `geometry/topology reads`
3. `validation evidence + interpretation`
4. `domain kernel / repair packet`

### 4.2 第一层：preflight governance

在真正执行 `execute_build123d` 之前，runtime 会做大量 Build123d 专项 lint。

对应入口：

1. [`src/sub_agent_runtime/tool_runtime.py:562`](../../src/sub_agent_runtime/tool_runtime.py)
2. [`src/sub_agent_runtime/tool_runtime.py:2345`](../../src/sub_agent_runtime/tool_runtime.py)
3. [`src/sub_agent_runtime/tool_runtime.py:3869`](../../src/sub_agent_runtime/tool_runtime.py)

这一层主要处理：

1. CadQuery/Workplane 幻觉
2. 不存在的 helper
3. 错误 primitive keyword
4. builder-native boolean 误用
5. path-sweep 常见错误
6. explicit-anchor countersink / half-shell 等 family 的高频错误

这说明当前系统已经在主动把“Build123d 库幻觉”从 sandbox runtime error 前移到 deterministic lint。

### 4.3 第二层：geometry / topology read-model

如果写入成功，模型可以继续通过 `query_geometry`、`query_topology`、`query_feature_probes` 等读工具拿高分辨率证据。

其中最关键的是：

1. `query_topology` 给 face/edge refs 与 candidate sets
2. `query_feature_probes` 给 family-specific 几何摘要
3. `render_view` 给局部视觉确认

这层是“模型能否看懂当前实体”的基础。

### 4.4 第三层：validation

`validate_requirement` 当前已经不是单纯的硬编码 pass/fail。

它的结构是：

1. `RequirementEvidenceBuilder` 先抽 geometry / topology / process evidence
2. `interpret_requirement_clauses(...)` 把 requirement clause 映射为 `verified / contradicted / insufficient_evidence`
3. 只有在 `insufficient_evidence` 仍然存在时，才尝试 `ValidationLLMAdjudicator`

关键位置：

1. [`src/sandbox_mcp_server/validation_evidence.py:131`](../../src/sandbox_mcp_server/validation_evidence.py)
2. [`src/sandbox_mcp_server/validation_interpretation.py:206`](../../src/sandbox_mcp_server/validation_interpretation.py)
3. [`src/sandbox_mcp_server/validation_llm.py:41`](../../src/sandbox_mcp_server/validation_llm.py)
4. [`src/sandbox_mcp_server/service.py:1367`](../../src/sandbox_mcp_server/service.py)
5. [`src/sandbox_mcp_server/service.py:1498`](../../src/sandbox_mcp_server/service.py)

### 4.5 第四层：domain kernel / repair packet

validator 和 probe 的结果会被同步进 `DomainKernelState`，再产出 feature instance、repair patch 和 `FamilyRepairPacket`。

关键位置：

1. [`src/sub_agent_runtime/feature_graph.py:802`](../../src/sub_agent_runtime/feature_graph.py)
2. [`src/sub_agent_runtime/tool_adapters.py:108`](../../src/sub_agent_runtime/tool_adapters.py)

这意味着当前系统已经有一个比“普通错误消息”更高级的修复层：

1. 知道哪一个 family 在阻塞
2. 知道 anchor keys 是什么
3. 知道 target anchor summary 是什么
4. 在少数 family 上甚至可以直接编译 deterministic repair code

## 5. 关于 Build123d 幻觉：当前真正的问题是什么

### 5.1 不是“LLM 完全不知道要干什么”

从当前 benchmark 现象看，更多时候不是 requirement 没懂，而是下面几类错误：

1. 把 CadQuery/Workplane 直觉带进 Build123d
2. 对 builder 生命周期理解错误
3. 对局部 workplane / local frame 理解错误
4. 把 Vector / Plane / primitive keyword 用错
5. 同一几何目标用了错误的 canonical lane

也就是说，问题更接近：

1. 几何意图理解正确
2. `build123d` 执行语法、builder 结构、局部坐标组织错误

### 5.2 当前系统已经在怎样解决这类幻觉

当前主要靠三种机制：

1. `preflight lint`
2. `skill_pack` 的 family guidance
3. `repair packet / recipe`

这三者的作用分别是：

1. lint：把已知无效 API surface 前移拦截
2. skill：给模型 first-turn bias
3. repair packet：把已知 family 的修复收窄成 deterministic lane

### 5.3 这条路为什么还不够

因为如果继续无限扩张 if/else 式 lint 和 skill 文案，会出现三个问题：

1. 规则越来越难维护
2. family 之间会互相抢路由
3. 新案例一旦稍微偏离既有 wording，规则命中率就会下降

所以，长期不能靠“继续补更多零散 hard rule”。

## 6. 当前验证与修复逻辑，是不是主要靠硬规则

是，但需要更精确地说。

### 6.1 validation 不是纯硬规则，也不是纯 LLM

当前 validation 更接近“规则主导 + LLM 补洞”的混合系统。

规则主导体现在：

1. evidence bundle 是确定性构建
2. clause interpretation 大部分是确定性逻辑
3. blocker taxonomy 也是结构化生成

LLM 补洞体现在：

1. 只有 unresolved clause 仍为 `insufficient_evidence` 时才会调用
2. 只允许高置信度结果生效
3. 对需要明确几何 grounding 的 clause，仍然会被拦回“继续取证”

这个限制在 [`src/sandbox_mcp_server/service.py:1585`](../../src/sandbox_mcp_server/service.py)。

### 6.2 repair 目前更明显是规则主导

当前 repair surface 的主体仍然是规则化：

1. preflight lint -> repair recipe
2. blocker taxonomy -> recommended repair lane
3. skill_pack -> rule-selected guidance
4. repair packet -> deterministic compiler

所以，如果问“当前修复是不是主要靠硬规则”，答案是：

1. 是，尤其在 repair 侧更明显
2. 但这些规则并不全是坏事，因为 API-invalid surface 本来就应当 deterministic 拦截

真正的问题不是“有规则”，而是“哪些规则应该存在于哪一层”。

## 7. 对复杂新案例的风险判断

如果继续沿着“看到一个 case 就补一条分散规则”的方式走，复杂新案例上确实会有明显风险：

1. 规则覆盖不到新组合型几何
2. prompt/skill 会越来越长
3. family 路由会越来越脆
4. 同一个新 case 可能同时触发多个修复分支，导致控制面发散

这也是为什么当前最应该收敛的是“治理形态”，不是单纯继续堆 patch。

## 8. 远期怎样改，才更有利于做成维护性强的系统

### 8.1 把 hard rule 分层，不要再混写

建议把现有规则正式拆成三层 registry：

1. `API governance rules`
   - 负责识别 Build123d 无效写法
   - 这层可以强规则化
2. `evidence interpretation rules`
   - 负责把 geometry/topology 证据解释成 clause/blocker
   - 这层应该 evidence-first，而不是 prompt-first
3. `family repair recipes / packets`
   - 负责把 repair 收敛成 narrow deterministic lane
   - 这层应以结构化数据资产存在，而不是散落长函数

### 8.2 让 validator 负责“描述问题”，不要过度负责“发明修法”

validator 最适合做的是：

1. 给出 contradiction
2. 给出 insufficient_evidence
3. 给出 required vs realized anchors
4. 给出 clause-level decision hints

不适合把 repair 逻辑继续长进 validator 本体。

### 8.3 让 repair packet 成为中间 DSL，而不是 case patch

`FamilyRepairPacket` 已经是很好的雏形。

远期更好的方向是：

1. packet 只携带 family / host / anchor / parameter / recipe slots
2. compiler 负责把 packet 编译成 deterministic Build123d 代码
3. LLM 更多负责：
   - 选择工具
   - 判断是否继续取证
   - 决定 whole rewrite 还是 packet repair

这样系统会比“每个 case 都现场写新代码”更稳定。

### 8.4 增强 read-model，而不是只增强 prompt

对复杂新案例更有价值的不是继续往 prompt 里塞更多注意事项，而是增强：

1. `query_topology` 的 candidate ranking
2. `query_feature_probes` 的 family coverage
3. `query_kernel_state` 的 anchor / blocker / packet surface
4. validator 的 local evidence grounding

因为复杂新 case 的难点，本质上是：

1. 模型需要知道当前实体哪里错了
2. 错在宿主、局部坐标、还是特征集合
3. 下一步该 whole rebuild 还是 local fix

这三件事主要依赖 read-model 和 semantic kernel，而不是 prompt 文案长度。

## 9. 当前最值得继续推进的方向

按优先级排序，当前更值得做的是：

1. 把 `tool_runtime.py` 里的 Build123d lint/repair recipe 继续抽成 registry 资产
2. 扩充 validator 的 evidence grounding，特别是 local anchor / host relation / path-sweep / half-shell 这类 family
3. 继续强化 `query_topology -> explicit face_ref/edge_refs -> apply_cad_action` 这条局部修边链
4. 扩大 `FamilyRepairPacket` 的覆盖面，但保持 packet 结构化，不退回 case-specific patch
5. 让 `query_kernel_state` 成为真正稳定的语义决策面，而不是只做摘要展示

## 10. 最终判断

如果只用一句话概括当前状态：

当前系统的主要短板已经不是“没有反馈”或“没有工具”，而是：

1. `build123d` canonical lane 的首轮进入稳定性还不够
2. validator / kernel 对局部几何语义的 grounding 还要继续加强
3. repair 逻辑虽然有效，但必须尽快从分散 hard rule 收敛为 registry + packet + evidence-driven 的可维护系统

这条收敛路径比继续补零散 case patch 更适合复杂新案例，也更接近一个长期可维护的 Build123d runtime。
