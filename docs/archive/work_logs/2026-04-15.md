### 6. 把 explicit-radius elbow 从 `TangentArc/JernArc` 高风险 lane 拉回 `CenterArc` canonical lane

1. round 1 不再走旧的 `Solid.sweep(...)` / annular-face-splitting lane，而是被 `path.line` preflight 拦住

4. round 5 最终仍然靠第三次 code repair 收敛

- 对于 requirement 明确给出“90-degree tangent arc + explicit radius”的 rail，模型仍会尝试 `TangentArc(...)` / `JernArc(...)` 这种更脆弱的构造 lane

1. `src/sub_agent_runtime/skill_pack.py`

   - 同时把这条 lint 纳入 `build123d_path_sweep_contract` repair recipe，避免 runtime 还要先吃一轮真实 sandbox construction error 才知道 arc lane 选错

### 14. path-sweep 新增一条 runtime-error 前置拦截

继续盯 `L2_149` 的复跑时，又定位到一条更具体但很泛化的 Build123d 错法：

1. 先在一个 annular `BuildSketch` 里画外圆 + `mode=Mode.SUBTRACT` 的内圆

1. `ValueError: Null TopoDS_Shape object`

1. 文件：`src/sub_agent_runtime/tool_runtime.py`
2. 新规则：`invalid_build123d_contract.annular_profile_face_extraction`
3. 行为：
   - 当检测到“从同一个 subtractive annular sketch 抽 `face()`，再把它直接送入 `sweep(...)`”时
   - 直接在 preflight 拦下
   - 并继续落回 `build123d_path_sweep_contract`
   - 明确要求优先使用 `sweep(profile.sketch, path=path_wire)`
