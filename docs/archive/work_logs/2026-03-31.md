## 继续推进 `L2_130` 的两条通用根因

### 1. `center + start_angle/end_angle` path segment 存在连续性契约缺口

最新证据来自：

1. `benchmark/runs/20260331_200400/L2_130`

现象不是单纯的“半圆难画”，而是 planner 会把这类 segment 当成绝对角度弧来写，默认以为：

1. `start_angle/end_angle`
2. `center`

已经足够定义弧段起终点，不再要求当前 cursor 正好落在 arc start。

但我们当前 runtime 的真实 contract 是：

1. arc 仍然从 current point 出发
2. `start_angle/end_angle` 只是补充 metadata

这就会出现一个可复用的错位模式：

1. 前一条 line 落在 arc 的终点
2. current point 与 arc implied start 不一致
3. helper 按当前点去造弧，导致：
   - `GC_MakeArcOfCircle::Value() - no result`
   - 或 silent geometry drift

今天已落地的修复：

1. `src/sub_agent_runtime/runner.py`
2. `tests/unit/sub_agent_runtime/test_runner_contracts.py`

修复内容：

1. `center + start_angle/end_angle` rewrite 不再只补 `to/end`
2. 现在会显式计算 implied start / implied end
3. 如果前一条 line 实际落在 arc end，就把它改写到 arc start
4. 如果没有可复用的前一条 line，就自动补一条桥接线
5. 同时把极小的 `-4.59e-15` 这类端点噪声吸到 `0.0`

这条修复的意义不是为 `L2_130` 定制，而是让 planner 常见的“absolute-angle arc mental model”能被 runtime 收口成真正连续的 path。

### 2. `query_sketch` 与真实 replay 对“闭合 path”说了两种语言

同样来自：

1. `benchmark/runs/20260331_200400/L2_130`

round 4 暴露出新的底层问题：

1. `query_sketch` 认为当前 window 已经有闭合 profile
2. planner 因此输出 `extrude`
3. 真实 replay 却报：
   - `ValueError: No pending wires present`

这不是 planner 再次乱来，而是 contract 本身不一致。

定位后发现：

1. planner/runtime 大量实际发的是 `add_path.close=true`
2. `query_sketch` 的 sketch-state 推断会把它当作闭合 path
3. 但 `service._action_to_code(ADD_PATH)` 只读 `closed`
4. 于是真实 replay 没有调用 `path.close()`
5. 最终形成：
   - 观测层认为 profile 已闭合
   - 执行层却没有 pending wire 可供 extrude

今天已落地的修复：

1. `src/sandbox_mcp_server/registry.py`
2. `src/sandbox_mcp_server/service.py`
3. `tests/unit/sandbox_mcp_server/test_action_params_contract.py`

修复内容：

1. `ADD_PATH.closed` 显式增加别名：
   - `close`
2. `service` 新增统一的 `_path_closed_flag(...)`
3. 让：
   - path entity build
   - profile metrics
   - area inference
   - `ADD_PATH` code generation
   全部共享同一套 `close/closed` 判定

这条问题的可复用价值很高，因为它解释了为什么某些 case 里：

1. `query_sketch` 看起来已经 ready
2. `extrude/cut_extrude` 却仍然拿不到 pending wires

4. 结果形成：
   - view plane 已识别

2. `tests/unit/sub_agent_runtime/test_runner_contracts.py`

   - `preferred_datum_plane or preferred_view_plane`

   - 且 axis-compatible plane 与当前 preferred plane 冲突
   - 或当前没有 preferred plane
   就直接优先 axis plane
3. 新增两条单测：
   - `Front Plane + along the Z-axis`
   - 只有 explicit extrude axis，无显式 plane

这条修复是通用的，因为任何“plane hint + axis hint”同时出现的 pre-solid case 都会受影响。

### 12. `L2_164` 暴露出执行层参数 alias 缺口，不是 planner 单例失误

在新的 full `L2` 权威回归里，`benchmark/runs/20260401_013645_l2_full_refresh/L2_164` 第一轮就出现了异常：

1. planner 给出的 base washer 动作本身并不离谱
   - `add_circle(r=15)`
   - `add_circle(r=7, mode=subtract)`
   - `extrude(height=3.0)`
2. 但执行后 `query_geometry` 直接变成：
   - `bbox=[30, 30, 20]`
   - 顶面 `z=20`
3. 这说明问题不在 tooth/pattern，而在更早的 base extrusion 执行层

我沿执行链定位后确认，根因是 `src/sandbox_mcp_server/service.py` 的 `EXTRUDE` / `CUT_EXTRUDE` translator 只读取 `distance`，却没有接 planner 常用的 `height` / `depth` alias：

1. `extrude` 代码模板之前是：
   - `distance = params.get("distance", 20)`
2. 所以 planner 写 `height=3.0` 时，会直接落回默认值 `20`
3. 这不是 `L2_164` 特有问题，而是所有 planner/provider 只产出 `height` 风格参数时都会触发的 shared execution defect

这轮落了三类修复：

1. 工具面文档 contract
   - `docs/cad_iteration/TOOL_SURFACE.md`
   - 显式写入：
     - `extrude` 支持 `distance` 和 `height`
     - `cut_extrude` 支持 `distance` 和 `depth` / `height`
2. 机器可读系统记录
   - `docs/cad_iteration/SYSTEM_RECORD.json`
   - 新增：
     - `extrude.height_alias_for_distance`
     - `cut_extrude.depth_height_aliases_for_distance`
3. 执行层与 registry
   - `src/sandbox_mcp_server/registry.py`
     - `extrude.distance` 新增 alias: `height`, `length`
     - `cut_extrude.distance` 新增 alias: `depth`, `height`, `length`
   - `src/sandbox_mcp_server/service.py`
     - 新增 `_resolve_linear_span_param(...)`
     - `EXTRUDE` / `CUT_EXTRUDE` translator 改为统一从 alias 集合取值，不再直接退回默认 `20/5`

新增单测：

1. `tests/unit/sandbox_mcp_server/test_action_params_contract.py`
2. 新覆盖：
   - `normalize_action_params` 能把 `extrude.height -> distance`
   - `normalize_action_params` 能把 `cut_extrude.depth -> distance`
   - `service._action_to_code(...)` 能直接消费 `height` alias 而不是落回默认 span

当前验证结果：

### 14. `L2_149` 的最终根因是 `create_sketch(path_ref=...)` 只在查询层附着成功，执行层并未真正平移到 endpoint origin

`benchmark/runs/20260401_022800_l2_149_add_path_dialect_fix/L2_149` 的 query 证据表明：

1. rail 已经正确
2. `profile:5:PR_2` 也显示：
   - `attached_path_ref=path:5:P_1`
   - `plane=XY`
   - current sketch `origin=[80,0,80]`

但最终几何却仍然是旧的错误大实体：

1. `bbox=[62.97,16,168]`
2. `faces=5`
3. `volume=25266`

这说明 query/state 层“认为” profile 在 path endpoint，但 replay 执行出来的真实 sketch 并没有真的在那个 3D 位置。继续沿代码查到两个共享缺陷：

1. `create_sketch(path_ref=...)` 虽然把 endpoint 世界坐标写进了 `origin`
   - 但 `_action_to_code(CREATE_SKETCH)` 在 no-solid sketch 分支里并不会把 workplane 平移到该 `origin`
2. `service._latest_sketch_frame(...)` 和 `create_sketch` codegen 之前都优先读取 `position`
   - 这会把 path-attached sketch 的局部 `position=[0,0]`
   - 错当成整个 sketch 的世界 origin

这轮落地修复：

1. `src/sandbox_mcp_server/service.py`
   - `_resolve_action_params_against_history(...)`
     - path-attached sketch 改成：
       - `origin = endpoint world origin`
       - `position = [0,0]` 保留为局部 profile 坐标
   - `_latest_sketch_frame(...)`
     - 优先读取 `origin`，而不是先读 `position`
   - `_action_to_code(CREATE_SKETCH)`
     - 对 `resolved_from_path_ref` / 显式 `origin` 的 no-solid sketch
     - 真实执行 `Workplane(...).transformed(offset=tuple(origin))`
3. 文档 contract：
   - `docs/cad_iteration/TOOL_SURFACE.md`
   - `docs/cad_iteration/SYSTEM_RECORD.json`
   - 明确写入 `create_sketch(path_ref=...)` 必须把 workplane rebasing 到 endpoint 的 world origin，而局部坐标仍保持局部
4. 新单测：
   - `test_resolve_action_params_rebases_path_attached_sketch_to_endpoint_origin`
   - `test_action_to_code_create_sketch_uses_world_origin_for_path_attached_frames`

最终 fresh live rerun：

1. `benchmark/runs/20260401_024200_l2_149_path_origin_fix/L2_149`
2. `summary.json`
   - `converged=true`
   - `validation_complete=true`
3. `queries/round_03_action_01.json`
   - `bbox=[90,20,90]`
   - `faces=8`
   - `edges=14`
   - `volume=16639.3199`
4. `evaluation/benchmark_eval.json`
   - `passed=true`
   - `final_score=1.0`

这条修复的意义比 `L2_149` 单题更大：

1. 所有 `create_sketch(path_ref=..., path_endpoint=...)` 的 sweep / loft / path-end local feature
2. 以后不再出现“query 说附着对了，但实际执行没在那个位置”的隐蔽错位
3. 这是真正的执行契约修复，不是 prompt 层 case patch
