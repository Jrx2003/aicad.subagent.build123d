### 2. evaluation 里的渲染确实有误导性，不是你主观感觉问题

现象：
- 同一平面会出现三角扇形/斜向切分的不同色块，看起来像几何被切裂。
- FreeCAD 里看起来正常，但 evaluation preview 看起来“面被分成很多块”。

根因：
- runtime 自定义 `render_view` 和 benchmark preview 都在用 tessellation triangles 直接做逐三角着色。
- `plot_trisurf(..., edgecolor=..., shade=True)` 会把同一 B-Rep 平面按三角网格分别着色，还会把三角边本身画出来。
- 这会把“渲染三角形边”伪装成“真实拓扑边”，对人工对图和 LLM 都有误导。

本次修正：
- 统一改成 flat fill：关闭逐三角阴影、去掉三角 edgecolor。
- 改为单独叠加 B-Rep edge overlay，让可见边来自真实边界，而不是三角剖分。

预期效果：
- preview 观感会更接近 FreeCAD。
- 同一平面的 patchwork 误导会明显下降。

### 3. `L1_148` 的核心问题不是单个 case，而是 plane-local 维度歧义处理错误

今日最重要的结论：
- 我前一轮为了避免 runtime 乱改 planner 的 `XZ/YZ` 宽高顺序，把 side-plane 的 `10x80` 过度视为“字面顺序绝对不可交换”。
- 这在 `L1_148` 上会导致第二、第三个“新杆”其实都退化成已有杆的重复表达，几何根本不变。

新结论：
- 对多平面正交 union 这类 requirement，`XZ/YZ` 上的 `A x B` 如果只是普通写法（例如 `10x80`），本质上只是“平面内两个 span 的无序对”。
- 只有当 requirement 明确把数值绑定到命名轴（例如 `10 along X, 80 along Z`）时，才应该把该顺序锁死，禁止 runtime 按 unmet-span 去重排。

这不是针对 `148` 的 case 拟合，而是在修一个通用语义问题：
- `plane-local width/height` 和 `global axis coverage` 之间存在歧义。
- 需要显式区分“模糊 span pair”和“axis-bound literal order”。

### 1. `L1_159` 的“看图明显不一致但 evaluator 仍给满分”已定位并修复

### 2. `L1_218` 的通用根因不是 groove 缺失，而是“height anchor 解释错位”

症状：
- 模型已经能生成 annular groove。
- 但 groove 的轴向位置偏了 `2.5 mm`，导致 front/iso 视角看起来明显不一致。

根因：
- prompt 写的是 `at a height of 30.0 mm`。
- 旧逻辑把它当作 groove profile 的中心坐标。
- GT 实际语义是 groove 的肩部/边界落在该高度，而不是 profile center 落在该高度。

通用修正：
- requirement 如果写 `at a height of H`，默认解释成 groove shoulder / axial edge anchor。
- 只有 requirement 明确说 `centered at a height of H` 时，才把 `H` 当作 profile center。
- runtime 可以保持 sketch origin 在 `H`，但会给 local profile 自动加 axial offset，让最后切出的 groove shoulder 对齐 `H`。
- validator 也同步按同一套 anchor 语义判断，不再只看“有没有发生 revolve cut”。

验证：
- `benchmark/runs/20260320_143220/L1_218` 已通过，`score = 1.0`。
- trace 中能看到 `add_rectangle_local_position_inferred_for_annular_groove_height_anchor` 这条 rewrite 证据。

### 三个 case 的最终根因与修复

#### 1. `L1_148`

#### 3. `L1_191`

最终根因分两层：
- 第一层：runtime 在 planner 返回后立刻把本轮 `query_topology` 清空，导致 `_rewrite_action_for_state` 拿不到 candidate sets，planner 即使看到了更强的 edge candidates，执行时也不会做收口。

  - 然后 `.edges("|Y and <Z").fillet(1)`
  - 也就是外侧底部两条与 Y 轴平行的边，不是 pocket 内底边。

- 新增 axis-parallel candidate sets：
  - `x/y/z_parallel_outer_edges`
  - `x/y/z_parallel_top_outer_edges`

- 修正 `L1_191` canonical override，使其与 audited executable reference / GT 一致。
- 修正 symmetric extrude 逻辑，保留 `symmetrically by 15 mm` 这种 per-side 语义，避免把 base block 做成半高。

## 补充结果更新（稳定基线提交前）

### provider 卡死根因与修复

现象：
- `benchmark/runs/20260320_144228` 在 `L1_122` 之前阶段暴露过长时间无输出。
- 问题不是 Kimi API 单纯慢，而是 timeout 无法真正中断 provider 调用。

根因：
- `CodeGenerator._complete_with_retries()` 外层用了 `asyncio.wait_for(...)`。
- 但 `OpenAICompatibleClient.complete()` 内部此前走的是 `asyncio.to_thread(self.client.invoke, ...)`。
- 这意味着超时只能取消协程外壳，不能停止底层阻塞的同步 `invoke`，所以 benchmark 会看起来“挂住”。

修复：
- 改为真正走异步 `ainvoke` 路径。
- 对 `kimi-k2.5-thinking` 保持无 completion cap，并给足 provider-aware timeout。

验证：
- `PYTHONPATH=src uv run --extra dev pytest tests/unit/test_openai_compatible_client.py tests/unit/llm/test_openai_compatible.py -q`
- 结果：`11 passed`
- 本次完整 benchmark `benchmark/runs/20260320_171141` 已确认不再在 `L1_122` 一类 case 上假性卡死。

### `L1_159` 重复打孔问题的最终修复

旧问题：

- 但 validator 仍保留 `feature_target_face_edit` blocker，导致 planner 误以为还没完成，追加了一个多余的 local sketch/hole 流程。
