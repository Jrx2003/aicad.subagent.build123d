# 执行-检测-验证模式详细实现指南

## 背景与问题定义

同事提出了三个核心需求：

1. **Check-Action-Validate 模式**：在执行动作前先 check，执行 action，执行后再 validate，每一步输出可供回溯的错误信息
2. **执行后检测信息作为 AI 证据**：将执行完动作后的检测输出作为 AI 判断的证据，与当前设计对齐
3. **Events 操作逻辑检查**：查看 JSON 文件中 events 代表的操作逻辑，在每个细分逻辑中添加检查和验证，配合更细粒度的 report

本文档详细分析这些需求在现有项目中的实现状态，并提供完整的代码位置指引。

---

## 一、Check-Action-Validate 模式

### 1.1 需求理解

这种模式要求在每个动作执行前后都设置明确的检查点：
- **Pre-check**：执行前验证前提条件是否满足
- **Action**：执行核心动作
- **Post-validate**：执行后验证结果是否符合预期

### 1.2 关键问题：验证是否使用 LLM？

**答案：`validate_requirement` 不使用 LLM，完全基于硬编码规则。**

#### 1.2.1 validate_requirement 的实现方式

**代码位置**：`src/sandbox_mcp_server/service.py` 第 768-818 行

```python
async def validate_requirement(
    self,
    request: ValidateRequirementInput
) -> ValidateRequirementOutput:
    """Validate requirement coverage against current session snapshot."""
    # ... 获取历史记录 ...

    # 构建检查列表（纯规则，无 LLM）
    checks = self._build_requirement_checks(
        snapshot=entry.result_snapshot,
        history=history,
        requirements=request.requirements,
        requirement_text=request.requirement_text,
    )

    # 判断是否有阻塞项
    blockers = [
        check.check_id
        for check in checks
        if check.blocking and check.status == RequirementCheckStatus.FAIL
    ]
    is_complete = len(blockers) == 0
```

**检查类型（全部硬编码）**：
- `solid_exists`：检查是否有实体（`snapshot.geometry.solids > 0`）
- `solid_positive_volume`：检查体积是否大于0
- `no_geometry_issues`：检查几何是否有问题
- 维度检查：对比 bbox 与需求中的尺寸规格
- 特征检查：如 `feature_inner_void_cutout`、`has_multi_plane_union`

**结论**：`validate_requirement` 是基于几何统计信息的规则引擎，**不涉及任何 LLM 调用**。

#### 1.2.2 哪里使用了 LLM 进行检验？

系统中确实有一个使用 LLM 的评估组件：**LLM Judge**（`src/sandbox_mcp_server/llm_judge.py`）

**代码位置**：`src/sandbox_mcp_server/llm_judge.py` 第 26-130 行

```python
class LLMJudgeEvaluator:
    """
    LLM Judge 用于评估生成结果与需求的语义匹配度
    """
    async def evaluate(self, evidence: EvidenceBundle) -> LLMJudgeEvaluationResult:
        # 构建包含渲染图的 prompt
        messages = self._build_messages(evidence=evidence, ...)

        # 调用 LLM 进行评估
        response = await self._llm_client.complete(
            messages=messages,
            temperature=0.0,
            max_tokens=self._max_response_tokens,
        )

        # 解析 LLM 返回的评分
        parsed = self._parse_judge_output(response.content)
        return LLMJudgeEvaluationResult(
            status="success",
            semantic_score=self._weighted_sum(parsed),
            judge_output=parsed,
        )
```

**使用场景**：
- 在 benchmark 评估阶段（非实时迭代）
- 评估模式：`EvaluationMode.LLM_JUDGE`（与 `GROUND_TRUTH` 并列）
- 默认配置：`sandbox_mcp_llm_judge_enabled: false`（默认关闭）

**配置位置**：`src/common/config.py` 第 157-160 行
```python
sandbox_mcp_llm_judge_enabled: bool = False
sandbox_mcp_llm_judge_rubric_path: str = "configs/eval/cad_llm_judge_v1.yaml"
```

**关键区别**：
| 组件 | 使用 LLM | 调用时机 | 目的 |
|-----|---------|---------|------|
| `validate_requirement` | ❌ 否 | 实时迭代中 | 快速几何健康检查 |
| `LLM Judge` | ✅ 是 | Benchmark 评估 | 语义相似度评分 |

    result = await self._codegen.generate_actions(
        ...
        previous_error=previous_error,
    )

    action_result = await self._sandbox.apply_action(...)

    # 如果失败，设置下一轮的错误信息

**关键设计**：

- **来源优先级**：`stderr` > `error_message`
- **结构化记录**：完整错误信息写入 `events.jsonl`，便于后续分析

## 五、LLM 使用全景图

### 5.1 哪些环节使用了 LLM？

| 环节 | 使用 LLM | 说明 |
|-----|---------|------|
| **Planner 决策** | ✅ 是 | 核心决策，生成下一步动作（`codegen.py`） |
| **validate_requirement** | ❌ 否 | 纯规则检查，无 LLM（`service.py` 硬编码规则） |
| **错误修复建议** | ✅ 是 | 将 `previous_error` 传入 prompt |
| **LLM Judge 评估** | ✅ 是（可选） | Benchmark 评估时使用（默认关闭） |
| **Report 生成** | ❌ 否 | 纯数据处理 |

### 5.2 错误信息的完整生命周期

```
动作执行失败
    ↓
Runner 捕获 error_message / stderr
    ↓
同时做两件事：
    1. 写入 trace/events.jsonl（留存）
    2. 设置 previous_error 变量
    ↓
下一轮 prompt 构建时
    ↓
_codegen.py 将 previous_error 格式化为 "## Previous Error"
    ↓
LLM 在下一轮看到错误信息，生成修复动作
    ↓
成功后 previous_error 被清除
```

### 5.3 总结与建议

#### 已实现的功能

1. ✅ **Check-Action-Validate 基础框架**：自动验证触发、验证检查构建
2. ✅ **证据收集系统**：几何查询、拓扑查询、验证结果收集
3. ✅ **证据压缩**：智能压缩以适应 LLM prompt 限制
4. ✅ **事件追踪**：完整的 events.jsonl 记录
5. ✅ **报告生成**：TSV 和 Markdown 格式的简要报告
6. ✅ **错误传递机制**：`previous_error` 单轮传递，成功清除

#### 需要补充的功能

1. ❌ **显式 Pre-check 阶段**：在 action 执行前进行前提条件检查
2. ❌ **细粒度中间事件**：action 执行过程中的子阶段记录
3. ❌ **版本回滚机制**：基于 checkpoint 的状态回滚
4. ❌ **时间线分析工具**：自动分析 events 并生成摘要
5. ❌ **几何变化追踪**：面级别的变化识别和记录

#### 优先级建议

| 优先级 | 功能 | 影响 |
|-------|------|------|
| P0 | 显式 Pre-check 阶段 | 提高执行可靠性，减少无效动作 |
| P1 | 几何变化追踪 | 增强 AI 对动作效果的理解 |
| P1 | 细粒度中间事件 | 便于问题定位和调试 |
| P2 | 时间线分析工具 | 提升开发效率 |
| P2 | 版本回滚机制 | 支持复杂修复策略 |
