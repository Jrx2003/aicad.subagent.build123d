## 需求澄清后的结论

今天最重要的不是写代码，而是先把概念彻底厘清。

### 1. 关系存在 vs 关系应该存在

这两件事必须分开：

1. `relation_base`
   - 当前模型里客观存在的关系
   - 例如：两个圆环是否同心、两个圆边是否共轴、两个路径段是否相切连续
2. `relation_focus`
   - 当前任务最该关注哪些关系
   - 这层需要 requirement / LLM / 阶段语义参与
3. `relation_eval`
   - “应该有的关系”和“当前观测到的关系”之间的偏差
   - 这里才适合谈误差、优先级、甚至分数

今天代码只做第 1 层，不再把第 1 层伪装成第 3 层。

### 2. `validate_requirement` 不再承担关系层职责

今天确认了一个很重要的边界：

1. `query_sketch`
   - 负责 pre-solid 路径/截面状态
   - 适合输出草图态 `relation_base`
2. `query_topology`
   - 负责实体后的 face/edge/axis/radius 等客观结构
   - 适合输出拓扑态 `relation_base`
3. `validate_requirement`
   - 继续负责语义 blocker
   - 当前不再输出 `relation_index`

也就是说：

> `relation_index` 现在是 `query_sketch/query_topology` 的客观关系底稿，不再是 `validate_requirement` 的另一套评分器。

## 今天确认的问题

## 1. `validate_requirement` 依然偏保守，尤其 sweep case

今天真实确认：

1. `validate_requirement` 现在没有 relation-base 输出
2. 对 bent pipe 这种 case，它仍然容易卡在：
   - `feature_profile_shape_alignment`

这不是今天要继续往里塞关系层的理由，反而说明：

> `relation_base` 和 `semantic validation` 分开是对的，因为它们现在本来就不是一套机制。

后面如果要补“rail 长度偏差”“应当共轴但没有做到”的判断，应该放在未来的 `relation_eval`，不是重新把 `validate_requirement` 变成关系层。

## 2. swept solids 的 wall faces 不一定是 `CYLINDER`

在 bent pipe 的真实 topology relation 文件里可以看到：

1. 一部分 wall face 的 `geom_type` 是：
   - `EXTRUSION`
   - `REVOLUTION`
2. 这会导致：
   - `annular_cylindrical_pair` 在这种 case 里不一定出现
   - 但 circular end edges 上的 `concentric/coaxial/equal_radius` 仍然可靠

这说明未来如果要做更稳的“空心 sweep 管壁关系”，不能只盯 `CYLINDER` face，还要考虑：

1. circular boundary edges
2. section-based grouping
3. 更一般的轴向截面抽象

## 3. annular solids 当前会出现重复 edge-pair group

以 `washer_annulus` 为例：

1. 顶面圆边和底面圆边都会形成候选 pair
2. 所以 `annular_edge_pair` 会出现多组“语义上相同、几何上分层”的组合

这不是错误，但会影响 relation-base 的压缩度。

后续可以考虑：

1. 先按截面层分组
2. 再做更高层的 section-level dedup

## 追加更新：`relation_focus / relation_eval` 已实现

上面那段“下一步”在今天后续工作里已经落地，不再只是计划。

### 1. 实现目的

这轮补的是两层新反馈，不再把所有问题往 `validate_requirement` 里塞：

1. `relation_focus`
   - 决定当前 round 最该看哪些相对关系
   - 例如：当前是 sweep rail、sweep profile，还是结果端面的环形拓扑
2. `relation_eval`
   - 把“需求里应有的关系”和“当前观测到的关系”做偏差对比
   - 这里才谈 `status / score / deviation / blocking`

### 2. 代码实现位置

新增核心模块：

- `src/sub_agent_runtime/relation_feedback.py`

主要职责：

1. `build_relation_feedback(...)`
   - 从 `requirements + action_history + query_sketch + query_topology + query_geometry` 生成：
     - `relation_focus`
     - `relation_eval`
2. 第一版已支持的 focus/eval 族：
   - `sweep_path_geometry`
   - `sweep_profile_section`
   - `sweep_result_annular_topology`
   - `annular_profile_section`
   - `annular_topology_core`

接入位置：

1. `src/sub_agent_runtime/runner.py`
   - 每轮生成 `relation_focus / relation_eval`
   - 单独落盘到 `queries/round_XX_relation_focus.json`
   - 单独落盘到 `queries/round_XX_relation_eval.json`
   - 同时写进 `prompts/round_XX_request_full.json`
2. `src/sub_agent/codegen.py`
   - 把两层新反馈压缩后喂给 planner
   - 新 prompt 规则明确要求：
     - 用 `relation_focus` 看当前该关注什么
     - 用 `relation_eval` 看预期与现实差在哪
3. `src/sub_agent_runtime/runner.py`
   - `_collect_latest_unresolved_blockers(...)` 现在会吸收 `relation_eval.blocking_eval_ids`
   - 对 sweep case，如果已经有更具体的 `relation_eval` blocker，就不再继续让宽泛的 `feature_profile_shape_alignment` 主导

### 3. 这轮修改真正解决了哪两个问题

#### 问题 A：`validate_requirement` 太保守，sweep case 容易卡死

现在的处理不是“再修 validator”，而是改 runtime 逻辑：

1. `validate_requirement`
   - 继续负责语义 blocker
2. `relation_eval`
   - 负责更具体的关系偏差诊断
3. `latest_unresolved_blockers`
   - 对 sweep 类需求优先采用 `relation_eval` 的具体 blocker

这让 planner 在真实 prompt 里看到的是：

- `eval:sweep_result_annular_topology`

而不是只看到：

- `feature_profile_shape_alignment`

#### 问题 B：bent pipe 的 wall faces 常常不是 `CYLINDER`

现在 `sweep_result_annular_topology` 的 eval 不再只依赖 wall face。

它明确优先使用：

1. `annular_edge_pair`
2. circular end edges 上的：
   - `coaxial`
   - `equal_radius`
   - `concentric`

也就是说，真实 B-Rep 里 wall face 即使表现成：

- `EXTRUSION`
- `REVOLUTION`

这层关系反馈仍然能工作。

## 今天最终结论

今天真正落地的不是“又多一层 JSON”，而是 runtime 里明确分出了 3 层职责：

1. `relation_base`
   - 客观关系底稿
   - 由 `query_sketch/query_topology` 提供
2. `relation_focus`
   - 当前 round 该看什么关系
   - 由 runtime 根据 requirement 和阶段生成
3. `relation_eval`
   - 预期关系和观测关系的偏差
   - 用于 planner 决策和 blocker 收敛

目前仍然没做完的地方也很明确：

1. `relation_focus / relation_eval` 只覆盖第一批 family
2. `annular_edge_pair` 还缺 section-level 去重
3. `L2_149` 证明“定位问题”已经比以前清楚，但“自动修复错误 sweep 结果”还需要下一轮策略设计
