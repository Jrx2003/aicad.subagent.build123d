## 2. 我对 `cad_xjm` 的理解

### 2.1 `analyze.py` 的核心价值：最小硬规则，不做过度语义推断

关键文件：

1. `~/code/cad_xjm/analyze.py:38`
2. `~/code/cad_xjm/analyze.py:68`

这份文件做的事情很小，但方向非常重要：

1. `wire_has_self_intersection(...)`
2. `wire_is_closed(...)`

它代表一种我认为正确的 CAD validator 边界：

1. 对 sketch/profile/path，只做最小、可验证、与几何本体直接相关的硬规则
2. 不把“planner 本该决定的高层语义”错误塞给底层 CAD 检查器

对我们项目的直接启发是：

1. `query_sketch` / barrier 应该优先吸收这类规则
2. 像 `closed`、`self_intersection`、`continuity` 这种信息应该结构化暴露给 planner
3. 不该继续用 prompt prose 让模型自己猜“这条 path 大概算不算闭合”

### 2.2 `modeling.py` 的核心价值：依赖图、脏传播、局部建模边界

关键文件：

1. `~/code/cad_xjm/modeling.py:71`
2. `~/code/cad_xjm/modeling.py:92`
3. `~/code/cad_xjm/modeling.py:195`
4. `~/code/cad_xjm/modeling.py:248`

这里真正值得借鉴的不是“把这套类原样搬过来”，而是三层思想：

1. `GraphNode` 把 sketch / feature 都放进依赖图
2. `mark_dirty()` 把脏传播机制做成显式系统行为
3. `SketchPlane -> ExtrusionNode` 这种关系说明“局部草图窗口”和“3D feature”之间应该有明确边界

我今天的判断是：

1. 你提到的“约束面”本质上就应该是这种依赖图思路在 runtime 层的 planner-facing 版本
2. 我们不一定要引入完整 feature graph，但必须把“当前轮允许在哪个局部面/草图窗口里工作”显式化

这就是今天 `active_surface` 改造的直接来源。

### 2.3 `cadnotes_20260327.md` 的核心价值：把 CAD 和 AI 的职责边界说清楚

关键文件：

1. `~/code/cad_xjm/cadnotes_20260327.md:8`
2. `~/code/cad_xjm/cadnotes_20260327.md:13`
3. `~/code/cad_xjm/cadnotes_20260327.md:36`
4. `~/code/cad_xjm/cadnotes_20260327.md:45`

这份笔记里最重要的一句话是：

1. 当输入足够精确时，CAD 应该给强校验
2. 当输入不够精确时，CAD 只能做基础 hard rules

这和我们当前系统的膨胀点完全对上：

1. 现在很多 runtime/validator/prompt 逻辑，其实是在替 planner 猜“用户大概想要什么”
2. 这种补丁越加越多，系统就越臃肿

所以今天的改动原则很明确：

1. 减少通用型泛化 blocker
2. 增强局部几何证据和局部工作面的显式 contract
3. 把专项 feature 家族问题留给 capability-like 局部规则，而不是继续把 `runner`/`validate_requirement` 变成大杂烩

## 3. 今天的关键改动

### 3.1 引入 `active_surface / surface_policy / expected_outcome / outcome_delta`

核心文件：

1. `src/sub_agent_runtime/active_surface.py:189`
2. `src/sub_agent_runtime/active_surface.py:329`
3. `src/sub_agent_runtime/active_surface.py:422`
4. `src/sub_agent_runtime/active_surface.py:547`
5. `src/sub_agent_runtime/active_surface.py:626`

做了什么：

1. 新增 `build_active_surface(...)`
2. 新增 `infer_expected_outcome(...)`
3. 新增 `build_outcome_delta(...)`
4. 新增 `normalize_expected_outcome_contract(...)`

这一步的意义不是“多加几个字段”，而是把 planner 每一轮面对的工作边界收紧成局部 surface：

1. `pre_solid_base_sketch`
2. `path_rail`
3. `path_profile`
4. `face_edit_window`
5. `edge_feature_window`
6. `post_solid_finish_window`

也就是说，planner 不再只面对“大量全局证据 + 一堆 blocker”，而是明确知道：

1. 现在该在哪个局部面工作
2. 可以做哪些动作
3. 这轮预期带来什么状态变化
4. 上一轮预期和实际 evidence 之间的偏差是什么

### 3.2 把新的 round contract 接到 `runner`

核心文件：

1. `src/sub_agent_runtime/runner.py:249`
2. `src/sub_agent_runtime/runner.py:259`
3. `src/sub_agent_runtime/runner.py:271`
4. `src/sub_agent_runtime/runner.py:323`
5. `src/sub_agent_runtime/runner.py:442`

做了什么：

1. 每轮在 runner 中构建 `active_surface`
2. 计算并写出 `surface_policy`
3. 把上一轮的 `expected_outcome` 变成当前轮的 `outcome_delta`
4. 将这些 artifact 写到 `queries/round_XX_*.json`

直接结果：

1. 新 round contract 是 inspectable 的
2. planner prompt、run artifact、debug trace 三者对齐

### 3.3 缩 prompt 噪声，让 planner 直接看到局部 contract

核心文件：

1. `src/sub_agent/codegen.py:125`
2. `src/sub_agent/codegen.py:276`
3. `src/sub_agent/codegen.py:325`
4. `src/sub_agent/codegen.py:662`
5. `src/sub_agent/codegen.py:696`

做了什么：

1. planner payload 新增：
   - `active_surface`
   - `surface_policy`
   - `expected_outcome`
   - `outcome_delta`
2. prompt 规则明确要求：
   - 默认在 `active_surface` 边界内行动
   - 如果 `outcome_delta` 显示上轮没达到预期，不要重复同一策略
3. generic `suggestions/completeness` 不再继续主导 compact payload

这一步的目标就是减少“AI 看一大堆无关提示后开始重新发明计划”的问题。

### 3.4 去掉 sweep 家族对其它 case 的串扰

核心文件：

1. `src/sub_agent_runtime/relation_feedback.py:32`
2. `src/sub_agent_runtime/relation_feedback.py:977`
3. `tests/unit/sub_agent_runtime/test_relation_feedback.py:1`

做了什么：

1. `relation_feedback` 里的 `_should_focus_sweep(...)` 不再因为“有 path”就自动进入 sweep family
2. 只有 requirement 真的提到 sweep，或历史里真的出现 `sweep`，才触发 sweep relation focus

这个修复直接解决了之前在非 sweep case 上出现的关系型串扰。

### 3.5 修正 post-solid face/edge 优先级，防止尾部 feature 抢走当前轮

核心文件：

1. `src/sub_agent_runtime/active_surface.py:626`
2. `tests/unit/sub_agent_runtime/test_active_surface.py:1`

做了什么：

1. 在 `active_surface` 选择逻辑中，未完成的 face-local edit 优先于 downstream edge finisher
2. 只有 face edit 真正确认完成，才切到 `edge_feature_window`

这个改动的效果是：

1. 避免了“明明该继续做面上 cut/notch，结果 planner 被 tail fillet 抢走”的问题

### 3.6 修正 barrier 与执行层对“闭合 path”的理解不一致

核心文件：

1. `src/sub_agent_runtime/runner.py:2019`
2. `src/sub_agent_runtime/runner.py:2022`
3. `src/sub_agent_runtime/runner.py:8221`

做了什么：

1. 对几何上首尾回到起点的 `add_path`，自动 rewrite 成 `closed=true`
2. barrier 不再只依赖 profile metadata，还会回退到闭合 path 的结构信息推断 polygon side-count

这一步解决的是一种非常典型的问题：

1. query 层知道这个 profile 已经闭合
2. 但 replay/execution 仍把它当成 open path

修完以后，像 `L1_191` 这种 post-solid triangle cut 就不再死在“没有 pending wires”。

### 3.7 修正对称拉伸的 per-side 语义

核心文件：

1. `src/sub_agent_runtime/runner.py:2942`
2. `src/sub_agent_runtime/runner.py:5821`

做了什么：

1. 对 requirement 中“symmetrically by N”这类语言，不再在缺少 sketch-plane context 时误把总距离砍半

这一步是 `L1_191` 从 0.6933 恢复到 `1.0` 的关键之一。

### 3.8 今天最后一刀：缩掉 `validate_requirement` 里最典型的伪 blocker

核心文件：

1. `src/sandbox_mcp_server/service.py:2595`
2. `src/sandbox_mcp_server/service.py:4036`
3. `src/sandbox_mcp_server/service.py:4082`
4. `src/sandbox_mcp_server/service.py:5073`
5. `src/sandbox_mcp_server/service.py:5106`
6. `src/sandbox_mcp_server/service.py:6832`
7. `tests/unit/sandbox_mcp_server/test_validate_requirement_contract.py:4623`

这是今天最符合“去臃肿”的一刀。

问题是什么：

1. `feature_profile_shape_alignment` 过于泛化
2. 在 `sweep` 已经成功、geometry/topology 都正确的情况下
3. 它仍然把“没有 post-solid 再画一次 circle sketch”当成 blocker

结果就是：

1. 最终实体已经对了
2. benchmark 分数已经 `1.0`
3. planner 仍然被逼着去端面上重画无效圆环

最后的修法不是简单删检查，而是收边界：

1. `feature_path_sweep_rail` 现在显式复用 `_relation_path_geometry_matches_requirement(...)`
2. 它会检查：
   - line lengths
   - arc radius
   - arc angle
3. `feature_profile_shape_alignment` 现在承认：
   - 对成功的 `sweep`，其前置 profile sketch 可以作为合法 shape evidence
   - 不再强迫 post-solid 再补一个无意义 sketch window

这一步非常重要，因为它说明我们不是“取消检查”，而是：

1. 去掉错误位置上的检查
2. 把真正该检查的东西收回 rail geometry 本身
