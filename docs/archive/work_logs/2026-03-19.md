### 多平面正交 union 的侧平面长短边歧义

在 2026-03-19 凌晨重新复跑 `L1_148` 时，我确认了一个新的共通问题：

1. requirement 明确是 `XY/XZ/YZ` 三个 datum plane 的 additive union。
2. planner 给出的动作表面上符合 requirement：
   - `create_sketch(XY) + add_rectangle(10x10) + extrude(both_sides)`
   - `create_sketch(XZ) + add_rectangle(10x80) + extrude(both_sides)`
   - `create_sketch(YZ) + add_rectangle(10x80) + extrude(both_sides)`
3. 但如果机械地把 `XZ/YZ` 的 rectangle `width/height` 直接映射到局部平面坐标，这两个后续实体会重复生成已经存在的杆，而不是新增正交 member。

这不是单个 case 的偶然问题，而是“自然语言里的 10x80 在侧平面上存在 axis-order 歧义，而 typed action IR 又把 `width/height` 固定化”带来的系统性缺口。

## 本日已落地的修正

1. 在 runtime rewrite 中新增了 multi-plane union 的 requirement-aware repair：
   - 当 requirement 明确是 multi-plane additive union，且当前侧平面 rectangle 的字面 `width/height` 会复现已有 span signature 时，runtime 会根据 requirement-derived signature options 重排 `width/height`。
2. 这个 repair 不是 case hard-code：
   - 触发前提是 requirement 语义明确；
   - rewrite 依据是 requirement 解析出的 signature options 与当前已覆盖 signature，而不是固定 case id。

## 下午新增推进

### `validate_requirement` 从“动作家族命中”推进到“语义 target 命中”

今天下午继续推进后，我确认上一版 validator 仍有一个根本缺口：

1. 只要 planner 用了 `face_ref` / `edge_refs`，旧逻辑基本就会把它当成“已经做了 pointer-aware edit”。
2. 这会导致一种危险的假完成：
   - 动作家族是对的；
   - local edit window 也是对的；
   - 但 face / edge target 实际上是错的；
   - validator 仍然可能判完成。

这不是单个 case 的问题，而是 face-edit / edge-target / groove / notch 这几类 requirement 共有的语义漏洞。

因此我在 service 里补了更严格的 ref 回溯校验：

1. `face_ref` / `edge_refs` 会先按 `face:<step>:...` / `edge:<step>:...` 回溯到被引用的 snapshot step。
2. validator 会在该 snapshot 的 topology index 上重新判断：
   - face 是否真的是 `top/front/back/left/right` 等目标面；
   - edge 是否真的是 `top_outer/bottom_outer/...` 这类目标边。
3. 所以“用了 ref”本身不再算语义命中，必须“ref 指向的实体真的属于 requirement 所要求的 target set”。

### nested inner cut / annular groove 的语义补强

除了 ref 回溯，我又补了两类通用语义约束：

1. nested centered inner cut：
   - 现在要求要么用 multi-profile bootstrap；
   - 要么是 post-solid axial end-face cut window；
   - 任意 side-face inner cut 不再会被当成“只是另一种等价建法”。
2. annular groove：
   - 不再只检查“有没有 subtractive revolve + local sketch window”；
   - 如果 requirement 明确给了 groove rectangle 尺寸和 axial height，就要求 groove window 和这些锚点对齐；
   - 否则判 `feature_revolved_groove_alignment` blocker。

### topology prefetch 扩张

之前 runtime 主动预取 topology 的条件偏保守，主要集中在：

1. invalid/stale ref
2. annular groove
3. nested cutout

现在已经扩到：

1. face-edit requirement
2. notch-like requirement
3. targeted-edge / fillet / chamfer requirement
4. 依旧保留 invalid/stale ref 与 annular groove / nested cutout

这样做的目标不是多查一次工具，而是减少 planner 在 topology-sensitive 任务里白白多绕一轮 inspection-only round。

## 晚间新增推进

### 1. face_ref 草图切除方向问题定位并修复

我进一步定位到一个更底层的执行问题：

1. `create_sketch(face_ref=...)` 过去虽然构造了稳定的局部坐标平面，但这个 workplane 脱离了实体；
2. 后续 `cut_extrude` 在 `cutBlind` 失败后会退化成“沿草图法向单边拉 cutter 再做 boolean cut”；
3. 对 front/back/left/right 这类面附着切除来说，这会让 cutter 向实体外侧生长，结果就是 notch / groove 看起来动作执行成功，但实体几何几乎没变。

本轮我做了两步修复：

1. sandbox helper 显式记录当前 sketch 是否来自 `face_ref`；
2. 当 `cut_extrude` 发现它是 `face_ref` 草图且无法直接 `cutBlind` 时，改为使用双向 cutter（`extrude(..., both=True)`）做减材，保证 cutter 穿过实体而不是只朝外侧长。

这不是只针对 `L1_191` 的补丁，而是对所有 face-attached subtractive edit 的通用执行修复。

### 2. query_topology 新增 inner-edge candidate sets

为了让模型能稳定命中“槽底边 / notch 底边 / interior feature edge”这类目标，我把 `query_topology` 的 requirement-aware candidate sets 继续扩展了：

1. `inner_edges`
2. `front_inner_edges` / `back_inner_edges` / `left_inner_edges` / `right_inner_edges`
3. `front_inner_bottom_edges` / `..._inner_top_edges` 等面内局部极值候选

这里的关键不是再堆一个特殊 hint，而是修正“outer / inner”的定义：

1. 不能只看 bbox 是否碰到外边界；
2. 必须结合 edge / face adjacency 来判断它到底是外轮廓边，还是局部切除后产生的 interior edge。

这样 `bottom of the v-shaped groove` 这类 requirement 才能拿到真正可用的 `edge_refs`。

### 3. validate_requirement 向 plane-specific 语义继续推进

`feature_multi_plane_additive_union` 过去更偏“最终 span/signature 覆盖”。

这会导致一个问题：

1. requirement 明确写了 `YZ plane + 10x80 rectangle + symmetric 10 extrusion`
2. planner 或 runtime 即使把 `width/height` 交换了，最终几何有时仍可能被宽松的 signature 逻辑放过

本轮我加了更严格的 plane-specific spec 校验：

1. 从 requirement 中提取按 plane 分段的 rectangle width / height / extrusion distance；
2. validator 逐 plane 检查 history 中对应 sketch window 是否保留了字面维度顺序；
3. 因此“显式 plane spec 被 runtime/LLM交换了轴”现在会被独立 blocker 抓住，而不会混在泛化的 union 几何检查里。

这一步对 `L1_148` 的意义很大，因为它能把问题明确归因到“plane-specific dim order 错了”，而不是笼统地说 shape 不像。

### 4. runtime 停止破坏显式 plane spec

我随后在真实 benchmark 中发现另一个更关键的问题：

1. 模型在 `L1_148` 的 round 2 已经给出正确的 `XZ width=10 height=80`
2. 但 runtime 旧的 `orthogonal_union_rectangle_axes_reordered_to_cover_unmet_span_signature` heuristic 又把它重写成 `80x10`

所以当前最大的噪声来源已经不是 LLM，而是旧 repair heuristic 自己在破坏显式 requirement。

本轮已修复：

1. 如果 requirement 已明确列出多个 datum plane 及对应 rectangle spec，则 runtime 不再允许这条 heuristic 擅自交换 `width/height`；
2. 只有在 requirement 没给出显式 plane spec、而又确实需要 signature-level repair 时，才允许该类 heuristic 介入。

### 5. symmetric extrusion 语义归一化

`L1_148` 还暴露了另一个通用问题：

1. 模型会把“symmetric extrusion 80mm”写成 `distance=80, both_sides=true`
2. 如果直接执行，实体总长会翻倍到 `160`

因此我又补了一层归一化：

1. 若 requirement 是 total-span 表述且要求 symmetric；
2. planner 又已经给了 `both_sides=true`；
3. runtime 会把总长参数规范成半长执行参数，即 `distance=total/2, both_sides=true`

这样可以避免不同模型对 “distance 是半边还是总长” 的理解差异直接污染几何。

## 深夜新增推进

### 7. 定位到 groove/notch 类任务的共通 translator 根因

我继续追 `L1_218` 时，终于把问题缩到一个明确且可复现的工具层缺陷：

1. 局部 profile 的 translator 长期使用：
   - `.center(...).rect(...).center(-...)`
   - `.center(...).circle(...).center(-...)`
2. 这类链式写法虽然能保留 pending wires，但会让 `_aicad_sketch.val()` 在某些阶段退化成 `Vector` 而不是可取 bbox 的 shape/wire。
3. 后续 `revolve(cut, use_cross_section=true)` 为了推断 groove 截面，会直接读：
   - `_aicad_sketch.val().BoundingBox()`
4. 一旦这里拿到的是 `Vector`，translator 就会掉进 fallback revolve 分支，最终把本应是“局部环槽 cut”的任务执行成错误的大范围切削。

这不是 `L1_218` 的单例问题，而是所有依赖“局部 sketch profile bbox -> 再转 cutter/revolve”流程的 groove / notch / face-local profile 任务的共通风险。

### 8. 本轮通用修复

我没有去针对某个 case 改 prompt，而是在 sandbox replay helper 层增加了稳定的 profile-shape 解析：

1. 新增 `_aicad_resolve_profile_shape(target)`：
   - 优先从 `ctx.pendingWires` 取真正的 pending profile；
   - 多 wire 时尝试合成 compound；
   - 再回退到 `vals()` / `val()` / 自身 `BoundingBox()`。
2. `revolve(cut, use_cross_section=true)` 现在不再直接信任 `_aicad_sketch.val()`，而是统一走这个 helper。
3. 这样 translator 不再因为 `Vector` 退化误判 sketch bbox，也就不会无意义地掉进 fallback revolve。

### 9. `validate_requirement` 继续向语义完成度推进

仅仅修 translator 还不够，因为 validator 之前仍可能把“动作和局部窗口都像 groove”误判为完成。

本轮新增了 `feature_revolved_groove_result`，它要求：

1. groove 执行后整体外轮廓 bbox 基本保留；
2. 体积确实下降，说明是减材；
3. 局部旋转特征面数量增长，说明不是把主体整体切坏。

因此像旧版那种“外包络被切成半边圆柱”的错误结果，现在即使 action history 看起来合理，也不会再被 validator 判成完成。

## 今晚收口与最终结果

### 12. same-shape nested section/frame 继续抽象成通用能力

在继续扫 case 时，我又确认了另一类共通问题：如果 requirement 是“同心同形状外轮廓 + 内轮廓”的 section/frame，planner 往往会分两步给：

1. 外层 `add_circle/add_rectangle/add_polygon`
2. 内层 `add_circle/add_rectangle/add_polygon`

这种写法在 typed action IR 里可以表达，但在跨轮 replay、stop-policy 和 bootstrap 验证上都偏脆弱。为此我没有去给单个 case 写提示词特判，而是把它收敛成工具面和 runtime 都可复用的通用机制：

1. `add_circle` 新增 `radius_inner`
2. `add_rectangle` 新增 `inner_width/inner_height`
3. `add_polygon` 继续支持 `radius_inner`
4. runtime 在 first-solid 之前，如果 requirement 语义明确是 same-shape nested section/frame，就会把相邻 outer/inner profile 融合成一个 profile action，并把被吸收的内层动作改写成 `snapshot`
5. `validate_requirement` 也会把这种 fused bootstrap 当成合法的“语义完成”证据，而不是错误地要求必须存在两个独立 profile action

这个改动的目标不是缩 prompt，而是让“同心同形状截面”从脆弱的多轮草图序列，变成一个更稳定、更可维护、更容易渐进式暴露给模型的能力单元。

### 17. `L1_148` 的最终定位

这是目前最重要的未收敛 case，我专门按你要求看了 evaluation 里的 iso 图，并对照 trace 做了归因。

1. 图像结论非常明确：
   - 生成结果是三轴正交 cross/star
   - 真值是两根 orthogonal bar
2. evaluator 数据也一致：
   - generated bbox: `[80, 80, 80]`
   - ground truth bbox: `[10, 80, 80]`
   - generated faces/edges 也显著偏高
3. 这不是“缺某个动作工具”：
   - planner 在 round 2 实际已经给出正确的 `XZ width=10 height=80`
   - 证据：`plans/round_02_response.json`
4. 真正的问题是流程层的 rewrite 仍在干扰显式语义：
   - `trace/events.jsonl` 里出现 `orthogonal_union_rectangle_axes_reordered_to_cover_unmet_span_signature`
   - runtime 把 planner 的 `10x80` 改写成了 `80x10`
5. 所以对 `L1_148` 的当前结论是：
   - 不是缺工具
   - 不是 planner 完全不会调用已有工具
   - 是 runtime 的 unmet-span heuristic 仍可能覆盖显式 plane-local dim order，而 dataset/requirement 对该 dim order 又更敏感

这也是为什么我在本轮没有去做 case-specific prompt 拟合，而是把它作为“显式 plane spec 与 repair heuristic 的冲突”记录成当前已知风险。
