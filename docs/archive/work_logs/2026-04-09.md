# 2026-04-09

## 目标

在 cleanup-first 收口完成后，开始 repair-oriented domain kernel push 的第一刀，但不直接跳到完整 typed helper 或全量 kernel patch compiler。  
本轮只做一个高价值、低风险的前移修复面：

1. 把 `execute_cadquery` 的已知 CadQuery API 幻觉前移到 deterministic preflight lint。
2. 让 lint 结果以结构化 failure summary + repair recipe 的形式进入下一轮，而不是只留下 raw stderr。
3. 验证这层修复面能否直接压缩 `L2_172` 一类高方差 case 的前段失败链。

## 本轮实现

### 1. `execute_cadquery` preflight lint

在 [tool_runtime.py](/Users/jerryx/code/aicad.subagent.iteration/src/sub_agent_runtime/tool_runtime.py) 的 `execute_cadquery` dispatch 前新增 deterministic lint：

- 拦截 `countersinkHole(...)`
- 拦截整个 `countersink*` 假方法族，例如 `countersinkSink(...)`
- 拦截 `cskHole(...)` 的错误 keyword，例如 `csinkDiameter=...`
- 拦截 `eachpoint(...).transformed(offset=loc.toTuple())` 这种高概率 `Location`/tuple 误用

lint 命中后不再进入 sandbox，而是直接返回：

- `failure_kind=execute_cadquery_api_lint_failure`
- `lint_hits`
- `repair_recipe`

### 2. 把 lint 结果接进 runtime repair surface

在 [context_manager.py](/Users/jerryx/code/aicad.subagent.iteration/src/sub_agent_runtime/context_manager.py) 中：

- `previous_tool_failure_summary` 现在会保留：
  - `lint_hits`
  - `repair_recipe`
- 新增 failure kind：
  - `execute_cadquery_api_lint_failure`
- 对应 recovery bias 与 next steps 改成：
  - 优先修 API surface
  - 不再把这类失败当 generic runtime failure
  - 对 countersunk hole array 会直接提示使用 `pushPoints(local_points) + cskHole(...)`

这一步的意义是把“模型写了不存在的方法名”从随机 stderr，变成一个结构化、可复用的 repair surface。

## 当前判断

这轮结果支持一个更清楚的方向判断：

1. 当前架构的主要瓶颈确实不是“模型完全不会写 CadQuery”
2. 也不是“有没有 FeatureGraph”
3. 而是 diagnosis / failure evidence 是否能被 runtime 压缩成低方差 repair surface

这次 preflight lint 证明：

- 即使还没走到完整 `FamilyRepairPacket -> typed helper`，
- 只要把 failure surface 结构化，
- token 和 round count 就能明显下降

## 下一步

## 追加：`L2_63` spherical recess validator gap 收口

在上一刀把 `FamilyRepairPacket + execute_repair_packet + preflight lint` 接上后，`L2_63` 的 bottleneck 发生了转移：

1. repair lane 已经能稳定选到 `spherical_recess`
2. runtime 也已经能执行 packet-backed sphere-array subtraction
3. 但 validator 仍把结果判成：
   - `feature_hole`
   - `feature_local_anchor_alignment`
   - `feature_profile_shape_alignment`
   - `feature_pattern`

### 根因

对照 [benchmark/runs/20260409_021811/L2_63/actions/round_02_execute_repair_packet.json](/Users/jerryx/code/aicad.subagent.iteration/benchmark/runs/20260409_021811/L2_63/actions/round_02_execute_repair_packet.json) 和 [geometry_info.json](/Users/jerryx/code/aicad.subagent.iteration/benchmark/runs/20260409_021811/L2_63/outputs/geometry_info.json)：

1. helper 产物已经包含：
   - 9 个 `SPHERE` faces
   - 9 个 top-plane `CIRCLE` edges
   - 正确的 3x3 centered centers
2. 但 snapshot 里的 `SPHERE` face `radius` 为空
3. 现有 `_snapshot_collect_subtractive_feature_centers()` 对 `SPHERE` 识别硬依赖 radius
4. 结果导致 `realized_centers=[]`，继而把 hole / anchor / profile / pattern 四个 blocker 一起带崩

### 本轮实现

在 [service.py](/Users/jerryx/code/aicad.subagent.iteration/src/sandbox_mcp_server/service.py) 中新增了两层 fallback：

1. `SPHERE` face `radius` 为空时，允许从 face bbox 推断球半径
2. 当 face 侧仍取不到中心集时，允许从 host-plane `CIRCLE` edges 恢复 local centers

新增的核心 helper：

1. `_snapshot_infer_spherical_feature_radius(...)`
2. `_snapshot_feature_edges(...)`
3. `_snapshot_edge_center_point(...)`
4. `_snapshot_edge_lies_on_host_plane(...)`
5. `_snapshot_collect_host_plane_circle_edge_centers(...)`

这让同一条 direct geometry 证据能够同时服务：

1. `feature_hole`
2. `feature_local_anchor_alignment`
3. `feature_profile_shape_alignment`
4. `feature_pattern`
