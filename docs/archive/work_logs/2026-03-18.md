4. 更新了规范文档，明确 validator 现在负责语义完成度，不再只是几何健康检查。

### 当前观察

- 旧版 validator 的主要问题是把“出现过某类 action”误当成“语义完成”。这会导致：
  1. `fillet/chamfer/revolve-cut` 发生 no-op 时仍可能被当作完成。
  2. 面向已有实体的编辑（top face/front face）即使没有形成有效的 local edit window，也无法被阻止。
  3. `groove/notch/U-shape` 这类任务里，planner 即使动作类型大体对了，也可能在错误的局部语义上结束。
- 这次改动的目标是让 validator 更像一个“流程/语义守门员”，而不是纯动作清单比对器。

## 进展更新：L1_101 暴露出的语义边界问题

### 发现的问题

在首次运行 L1 sampled benchmark 时，`L1_101`（top-face 四圆方阵凸台）暴露出一个典型的泛化问题：

- `validate_requirement` 把“four circles arranged in a square array”误判成了“nested inner cutout”。
- 触发原因不是单个 case 特殊，而是旧的 `nested_profile_cutout` 语义触发条件过宽：
  - 只要同时出现 `circle + square + centered + extrude` 一类词，就会被视为“同心内外轮廓截面”。
  - 这会把阵列、孔位分布、凸台布局等多实体布局语义误吸进去。

### 修正策略

- 收紧 `nested_profile_cutout` 触发条件：
  1. 若 requirement 明确出现 `array/pattern`，直接排除。
  2. 若 requirement 明确出现 `two/four/... circles|squares|rectangles` 这类多实例描述，直接排除。
- 这个修正是通用语义边界修正，不是对 `L1_101` 做 case 拟合。
- 修正后增加回归测试，确保 square-array 语义只会触发 `pattern` / `target-face edit` 一类 blocker，而不会再触发 `nested inner cutout`。

### 影响

- 该问题如果不修，会让 planner 在已有实体上错误进入“内腔/同心截面”修复路径，浪费 round budget。
- 修正后，planner 会更倾向于 top-face sketch + repeated circles/pattern 这条正确能力暴露路径。

### L1_101：连续暴露出三层共性问题

#### 第一层：pattern 语义被错判成 nested cutout

- 早期表现：`validate_requirement` 将四圆阵列误吸进 `nested_profile_cutout`。
- 修正：收紧 nested-profile 触发条件，明确排除 `array/pattern` 与 `four circles` 这类多实例语义。
- 这是语义边界修正，不是对 `L1_101` 定制答案。

#### 第二层：重复 profile 的工具表面不完整

- 复测目录：`benchmark/runs/20260318_183400/L1_101`
- 现象：planner 开始尝试更高层表达，使用了 `add_circle(centers=[...])` / 或在后续轮次拆成多个 `add_circle`。
- 问题：工具层原本没有把 `centers` 作为一等能力对待，translator 只能稳定处理单圆；validator 对 repeated-profile 的计数也只按 action 数，不按 profile 数。
- 通用修正：
  1. `add_circle` 明确支持 `centers` / `positions` 列表。
  2. repeated-profile 验证支持按 profile multiplicity 计数，不再把“一条 action 里多个圆”误看成一个 profile。
  3. 补充回归测试，确保 array/pattern 语义既可由显式 pattern action 满足，也可由 repeated-profile 窗口满足。

#### 第三层：attached-sketch additive flow 的执行语义错误

- 复测目录：`benchmark/runs/20260318_184100/L1_101`
- 现象：生成结果已经接近真值，但 `solid_count=2`、`score=0.7594235745830059`；iso 图能看出基体与凸台没有正确熔合。
- 根因一：`extrude` 在 attached sketch 上直接用 `_aicad_sketch.extrude(...)` 替换 `result`，没有把新增特征 union 回现有实体。
- 根因二：`add_circle`/`add_rectangle` 以前使用 `.center(...)`，这会把连续 profile 的位置当成相对位移累积；对于阵列类任务，局部坐标应被解释为绝对 sketch 坐标。
- 根因三：runner 的 centered-array normalize 启发式过宽，曾把本来正确的 `(4,4)` 改写成 `(-4,-4)`，属于“帮助过度”。
- 通用修正：
  1. `extrude` 在已有 solid + attached sketch 情况下，显式对新增特征执行 `union`，保证 pad/stud/boss 不覆盖基体。
  2. `add_circle` / `add_rectangle` 改为基于 `pushPoints(...)` 的绝对局部坐标语义，不再依赖累积 `center(...)`。
  3. centered-array 坐标 rewrite 改成 evidence-bound：只有当输入明显是 bbox-origin / edge-frame 坐标时才归一化；已经落在当前局部实体范围内的显式坐标不再改写。
- 最终复测目录：`benchmark/runs/20260318_185450/L1_101`
- 结果：`score=1.0`。

### 当前结论

- 今天新增修复的重点不是“让某个 case 过”，而是把三类通用缺口补成稳定契约：
  1. 工具参数必须真实生效：`cut_extrude.both_sides`。
  2. 附着草图上的局部加料/减料必须保留基体语义：`extrude` additive union。
  3. profile 坐标与 repeated-profile 表达必须是绝对、可组合、可验证的，而不是靠模糊启发式碰运气。
- 这些修复已经通过 focused benchmark 直接验证：

## 进展更新：L1_101 的真实根因进一步收敛

在继续复测时，`benchmark/runs/20260318_193500/L1_101` 再次掉回 `score=0.7594235745830059`。这次我重新看了 trace 和 evaluation 图，而不是沿用上一轮结论，得到更准确的共通根因：

### 真实根因

1. planner 在 `round_2/round_3` 输出的是 `add_circle(x=..., y=...)`，不是 `add_circle(position=[...])`。
2. translator 之前只认 `position/center`，不认独立的 `x/y/z` 字段。
3. 结果不是“planner 不会画四个圆”，而是 replay 时四个圆的局部坐标丢失，草图 profile 被错误地堆在默认位置。

这属于**动作参数兼容层缺口**，不是单 case 拟合，也不是模型能力不足。

### 通用修正

1. 在 registry 的 `normalize_action_params` 中，把 `x/y/z` 统一归一化到 `position`。
2. 这个兼容层同时适用于 `create_sketch / add_rectangle / add_circle` 这类局部坐标动作，而不是只修 `L1_101`。
3. 为 `validate_requirement` 新增 `feature_target_face_additive_merge`：
   - 对 `stud/boss/protrusion` 这类 target-face additive feature，不再只检查“有无 post-solid sketch window”
   - 还要求特征必须和原有实体保持 merged，不能变成断开的新 solid
4. 在 capability / output rules 里明确鼓励：
   - 当 requirement 已经给出 repeated circle centers 时，优先用 `add_circle.centers=[...]`
   - 这样可以把 face-attached repeated-circle sketch window 压缩成一轮完成

### 验证

1. 聚焦 benchmark：
   - `benchmark/runs/20260318_194100/L1_101` -> `score=1.0`
2. 聚焦单测：
   - `PYTHONPATH=src uv run --extra dev pytest tests/unit/sub_agent/test_codegen_aci.py tests/unit/sandbox_mcp_server/test_action_params_contract.py tests/unit/sandbox_mcp_server/test_validate_requirement_contract.py tests/unit/sub_agent_runtime/test_runner_contracts.py`
   - 结果：`90 passed`
3. 真实 live probe：
   - 目录：`test_runs/20260318_195200`
   - 结果：`converged=true`、`planner_rounds=2`、`validation_complete=true`
   - 关键证据：`plans/round_02_response.json` 中 planner 直接使用了 `add_circle.centers`

### 当前判断

- 对 repeated circles / studs / hole arrays 这类 face-attached pattern，当前最稳的共通路线是：
  1. `query_topology` 拿 face_ref
  2. `create_sketch(face_ref=...)`
  3. `add_circle(centers=[...])`
  4. `extrude` 或 `cut_extrude`
- 这条路线同时解决三件事：
  1. 减少尾部回合
  2. 避免 sketch state 累积歧义
  3. 让 repeated-profile intent 以一条更强的 typed action 暴露给模型

## 进展更新：face_ref 局部 frame 稳定化与 boundary-anchor 拓扑暴露

### 发现的问题

在继续看 `L1_191` 与 `L1_218` 的 evaluation 图时，我确认了一个更本质的共通问题：

1. 问题不是 planner 找不到正确 `face_ref`，而是 `face_ref` 附着草图后的**局部坐标系不稳定**。
2. 旧实现直接用 CadQuery 默认的 `face.workplane()`，导致同样的局部点集在不同侧面上会被解释成不同的朝向。
3. 具体表现：
   - `L1_191` 的 front-face V-notch 被解释到了侧面区域，而不是 front face 顶边中点对应的局部窗口。
   - `L1_218` 的 top-face centered square 在旧运行里也会被解释偏到一侧，进而把整个实体切成错误的半开形态。

这说明问题不在“有没有 `query_topology`”，而在 **topology ref 到 sketch local frame 的 contract 还不稳定**。

### 通用修正

这次没有加新建模动作，而是做了两层通用收敛：

1. `face_ref` 稳定局部 frame
   - 对 axis-aligned planar faces，不再直接继承 CadQuery 默认 `workplane()` 朝向。
   - runtime 现在为 `Top/Bottom/Front/Back/Left/Right` 六类 face_ref 构造稳定 local frame。
   - 关键目标不是“完全贴近 OCC 内部坐标”，而是让 planner 能可靠地把 `top edge midpoint / centered / height` 这类 requirement 解释成稳定的局部坐标。
2. `query_topology` boundary-anchor candidate sets
   - 新增：
     - `front_top_edges`
     - `back_top_edges`
     - `left_top_edges`
     - `right_top_edges`
     - `front_bottom_edges`
     - `back_bottom_edges`
     - `left_bottom_edges`
     - `right_bottom_edges`
   - 这些 candidate set 会附带：
     - `anchor_ref_id`
     - `anchor_point`
     - `anchor_role`
     - `sketch_plane`
     - `sketch_u_axis`
     - `sketch_v_axis`
3. `front/back` 语义统一
   - 之前 `create_sketch` 的 `FRONT -> >Y` 与 `query_topology.front_faces -> -Y` 是自相矛盾的。
   - 现在统一为：
     - `front` = `+Y`
     - `back` = `-Y`
4. candidate set 暴露上限提高
   - `_MAX_PROMPT_TOPOLOGY_CANDIDATE_SETS` 从 `6` 提到 `10`。
   - 目的不是放大 prompt，而是避免“正确候选集已经算出来，却被 token budget 截掉”。

### 验证

1. focused unit tests：
   - `PYTHONPATH=src uv run --extra dev pytest tests/unit/sandbox_mcp_server/test_query_topology_contract.py tests/unit/sandbox_mcp_server/test_action_params_contract.py tests/unit/sub_agent/test_codegen_aci.py tests/unit/sub_agent_runtime/test_runner_contracts.py`
   - 结果：`99 passed`
2. 新增 contract 覆盖：
   - `front_faces` 现在验证 `+Y` 语义与 `XZ / -X / +Z` sketch frame metadata
   - `front_top_edges` 验证 candidate id、anchor point、sketch plane metadata
3. 当前 benchmark 证据
   - 目录：`benchmark/runs/20260318_225103`
   - `L1_191` 的 `round_03_request.json` 已确认 planner prompt 中包含：
     - `front_faces`

     - `sketch_plane = XZ`

     - `sketch_v_axis = +Z`

### 当前判断

## 进展更新：L1_148 提前收敛的真正根因

### 新发现的问题

在新目录 `benchmark/runs/20260318_225103` 里，`L1_148` 第一轮只做了一个 `XY` 对称挤出，却被 runtime 直接判定完成，`score=0.1728333333333333`。

trace 证据显示：

1. round 1 planner 只生成第一根 centered post：
   - `create_sketch(XY)`
   - `add_rectangle(10x10)`
   - `extrude(distance=40, both_sides=true)`
2. `auto_validate_after_action` 直接把 `feature_multi_plane_additive_union` 判成 `PASS`
3. query evidence 中出现：
   - `required_signature_segments=3`
   - `covered_signature_segments=3`
   - 但 `additive_planes=['XY']`

### 根因分析

这个问题不是 planner 的锅，而是 validator 里 multi-plane union 的 signature 覆盖判定过宽：

1. requirement 虽然写了 `XY / XZ / YZ` 三个 setup planes；
2. 但由于三个 plane 在这个 case 中都可能投影到类似的最终 span signature，
3. 旧逻辑只要一个 signature 命中了所有 options，就会把三个 requirement segment 都判成“已覆盖”。

换句话说，旧逻辑是在用“最终形状像不像”替代“历史里有没有真的经过这些 setup planes”，这对 multi-plane union 是不成立的。

### 通用修正

1. 对显式写出 datum planes (`XY/XZ/YZ`) 的 requirement：
   - `feature_multi_plane_additive_union` 现在优先要求 **executed additive plane coverage**
   - 也就是历史里真的出现过这些 plane 的 additive stage
2. span signature 现在只作为辅助证据：
   - 当 requirement 没有显式 plane coverage 时才作为 fallback
3. 这保留了“不要把 plane-count 当作最终几何形态”的优点，
   同时避免“只做第一根杆就被误判完成”的早停。

   - 三 plane additive stage 都存在时继续通过

## 进展更新：nested-profile bootstrap compression 误删内轮廓

### 新发现的问题

在真实 probe `test_runs/20260318_230104` 中，我发现另一个典型流程错误：

1. requirement 明确要求在第一次挤出前保留 `circle + centered square` 这一整组 section profile。
2. planner round 2 实际已经给出了：
   - `add_circle`
   - `add_rectangle`
   - `extrude`
3. 但 runtime 的 `nested_profile_cutout_to_first_solid_compressed` 策略把第二个 profile (`add_rectangle`) 压掉了，只执行了：
   - `add_circle`
   - `extrude`

这说明当前系统里仍存在一种错误假设：

- “首实体前的第二个 profile 大概率是冗余噪声，可以为了节约 round budget 删掉。”

这个假设对 `notch` 之类某些 profile 分解任务可能成立，但对 `nested section / frame region / concentric profile` 是根本错误的。

### 通用修正

1. 对 `nested_profile_cutout` requirement：
   - runtime 不再压缩首实体前的多 profile sketch window。
   - `circle + square + extrude` 这种 section definition 会完整保留到 first solid builder。
2. 删除了旧的 rewrite：
   - 不再把 `nested_profile_cutout` 场景下的“第二个 profile”自动改写成 `snapshot`
3. 保留原有 `notch` 的 profile-skipping 启发式，因为 notch 仍可能是多 profile 拆分噪声；但 nested section 不再共用这条错误策略。

### 验证

1. focused unit tests：
   - `PYTHONPATH=src uv run --extra dev pytest tests/unit/sub_agent_runtime/test_runner_contracts.py tests/unit/sandbox_mcp_server/test_validate_requirement_contract.py`
   - 结果：`67 passed`
2. runner contract 现在明确要求：
   - `nested_profile_cutout_to_first_solid` 保留完整 section profile window
   - `secondary_profile_before_first_solid_skipped_for_nested_cutout_requirement` 不再出现
