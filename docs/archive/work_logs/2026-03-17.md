## 问题定位与解释

### 问题 1：Kimi K2.5 thinking 之前为什么“明明模型能答，却在我们这里不返回结果”

根因不是单一的“模型差”，而是组合问题：

1. 我们对 planner completion budget 设了上限。
2. Moonshot thinking 模型会优先把预算花在 `reasoning_content`。
3. 一旦 completion 提前耗尽，visible `content` 就可能为空。
4. 我们又把“空 content 但有 output_tokens”当成异常，因此 round 直接失败。

修正方向：

1. 对 `kimi-k2.5-thinking` 不再设置 planner completion 上限。
2. 给 provider request timeout 更宽松的 floor。
3. 保留空 final-content 的显式保护，但不再一开始就把模型截断。

### 问题 2：为什么还需要继续做 requirement-aware `query_topology`

因为 LLM 当前的主要难点不是“会不会写动作名”，而是：

1. 能不能稳定感知已有 face/edge；
2. 能不能把 requirement 里的“top / outer / top outer”映射到明确实体；
3. 能不能在局部编辑后避免 stale ref / broad selector 误伤。

论文的启发在这里非常直接：要给模型 pointer layer，而不是只给更多库文档。

### 问题 3：尾部回合为什么还会冗余

目前冗余主要来自两类情况：

1. 完成 candidate action 后没有立即验证，planner 又保守地重复 1 轮 inspection/编辑。
2. `validate_requirement` 还不够语义化，更多是在做几何完整性判断，无法完全替代人类对 requirement phrase 的语义检查。

本轮已经修复第一类中的一个明显分支：`revolve(cut)`。

## 晚间继续推进：从 case 过关转向通用流程加固

### 新诊断 1：`reconstructed_cadquery_code` 在 stateful edit 历史上会误导模型

我在检查 `benchmark/runs/20260317_180023/L1_218/prompts/round_03_request.json` 时发现：

1. `reconstructed_cadquery_code` 只重建了“圆柱 + 新建矩形草图”，没有表达已经完成的 `cut_extrude`。
2. 这不是单个案例 bug，而是一个通用问题：typed IR 的历史动作并不总能被安全地压缩成简化 CadQuery 代码。
3. 如果继续把这段近似代码当成权威上下文，就会让 planner 对“当前实体状态”产生错误感知。

### 新改进 1：引入权威的 indexed action trace

已修改：

1. `round_request` 新增 `reconstructed_action_trace`，按 step 编号给出权威动作历史摘要。
2. `reconstructed_cadquery_code` 只在“历史仍可被忠实压缩”的简单状态下保留。
3. 对于已经进入 `attach_to_solid / cut_extrude / revolve / complex plane placement` 的 stateful edit 历史，不再硬生成近似 CadQuery 代码。

这解决的是通用问题：

1. 给模型一个稳定、明确、可索引的动作历史表示。
2. 避免错误 replay summary 污染后续 planning。
3. 保留 progressive exposure，但把“哪份证据是权威的”讲清楚。

### 新诊断 2：`query_topology` 只有 ref list 还不够，缺少局部参考框架

仅有 `top_faces / outer_edges` 这类 ref set，仍然要求模型自己再推一遍：

1. 主轴是什么；
2. 外轮廓半径/半跨度是多少；
3. 轴向中点在哪里；
4. 哪些 sketch plane 更适合这次局部编辑。

这会在 groove、肩台、环形切口、侧面局部编辑等场景重复消耗模型的 reasoning budget。

### 新改进 2：`query_topology` candidate metadata + 更强的 primary candidate sets

已修改：

1. `candidate_sets[*]` 新增 `metadata`。
2. 现在可返回更强的 requirement-aware candidate sets：
   - `primary_outer_faces`
   - `primary_axis_outer_edges`
3. metadata 里会给出紧凑锚点：
   - `primary_axis`
   - `axis_midpoint`
   - `outer_half_span_estimate`
   - `suggested_sketch_planes`
   - `dominant_ref_id`

这不是对 `L1_218` 拟合答案，而是把“主轴/外轮廓/局部参考平面”提升成通用拓扑上下文。

### 新改进 3：runtime 在必要时自动预取 topology，而不是完全依赖模型主动 inspection

已修改：

1. 当 solid 已存在；
2. 当前 requirement/blocker 明显是 topology-sensitive；
3. 最新 step 还没有合适的 topology context；

runtime 会在进入下一轮 planner 前自动执行一次紧凑 `query_topology` 预取，并把结果直接放入下一轮 prompt。

这样做的目的：

1. 压缩尾部 inspection round；
2. 给模型提供更强的 indexed topology evidence；
3. 减少“明明该 query，却因为 planner 一时没提而继续盲猜”的情况。
