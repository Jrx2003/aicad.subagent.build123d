# 2026-04-03 工作记录

## 今日主题

延续 2026-04-02 的 ReAct 重构，但本轮不再继续往 `single_action` 方向生硬收紧，而是聚焦三个更底层、也更泛化的问题：

1. `validate_requirement` 被当成 post-solid 每轮默认必需证据，导致 token 膨胀、重复验证和 planner 超时。
2. `query_topology` 在 relation-heavy 需求下被过度自动抓取，很多时候只是把噪声和大 payload 继续喂给 planner。
3. planner 侧虽然已有 `active_surface / surface_policy`，但还缺少更清晰的 inspection 分区语义，导致模型容易“全查一遍再说”。

## 本轮结论

### 1. 证据调度问题的根因

根因不是“模型不够聪明”，而是 runtime 之前把证据策略写成了偏硬编码的保守模式：

1. post-solid 状态默认要求：
   - `query_snapshot`
   - `query_geometry`
   - `validate_requirement`
2. relation-heavy post-solid 场景又会默认补：
   - `query_topology`
3. 这样会形成一种坏循环：
   - 每轮动作推进一步
   - 旧 validation 立刻因为 step 变化而失效
   - 下一轮又被 runtime 预取回来
   - planner 再拿着新的大 payload 继续思考
   - token 和 latency 一起堆高

这不是单个 case 的问题，而是对所有 post-solid 多轮修复都会有副作用。

### 2. scripts/describe 的迁移评估（今天的更明确版本）

我重新对照了 `scripts/describe/experiment_1/describe.py` 和 README，理解同事的核心思路是：

1. 不要直接把 STEP/拓扑/关系原始表喂给模型。
2. 应该先抽成 feature-level 的对象：
   - hole
   - slot
   - boss
   - prism / annular / stack 等结构
3. 然后再让模型围绕“缺什么 feature、feature 数量/位置对不对”来思考。

我的判断：

1. 这个思路是对的，而且和我现在在做的 `planner_digest / compact relation payload` 是同方向。
2. 但它当前更适合作为：
   - planner-facing feature digest
   - failure diagnosis 辅助摘要
3. 它还不适合直接变成 runtime 的硬判官，因为：
   - describe 是独立实验脚本，不是正式 query contract
   - STEP 反推 feature 仍然有启发式不稳定性
   - 一旦直接进入执行闭环，误判成本会比“作为摘要参考”大得多

今天没有把它硬接入主协议，而是先把主协议改成更适合后续承接 feature digest 的形态：

1. 先压缩 evidence scheduler
2. 再把 inspection 分区稳定下来
3. 后续再决定是否把 describe 风格摘要升级成正式 runtime artifact

## Benchmark 启动记录

## 补充问题定位：L2_88 planner 超时并非单纯 prompt 过大

在新的 full L2 run 里，`L2_63` 已收敛，但 `L2_88` 长时间停在 round 1 planner 调用前后。进一步排查后确认：

1. `L2_88` 的 round 1 planner request 落盘体积并不夸张：
   - `round_01_request_full.json` 约 `6 KB`
   - `round_01_request.json` 约 `7.6 KB`
2. 因此它不是“单纯因为 prompt 太大导致的必然超时”。
3. 更深层的问题出在 `OpenAICompatibleClient.complete()` 的实现：
   - 之前把 `self.client.invoke(...)` 包在 `asyncio.to_thread(...)` 里
   - 再在外层套 `asyncio.wait_for(...)`
4. 这种写法表面上给了 planner 超时，但实际上：
   - 外层协程超时后只会取消等待
   - 被 `to_thread` 提交出去的阻塞线程不会被真正终止
   - case 进程会被悬挂中的 LLM 请求线程拖住，导致 benchmark 看起来像“planner 永远不返回”

这说明上一个阶段写进文档里的“thread-backed invoke 可以避免 hang”结论并不成立，真实效果反而会把 case 级 timeout 变成假超时、真悬挂。

## 补充问题定位：L2_130 / L2_192 暴露的两个更深层系统问题

### 1. L2_130 不是 planner 不会画，而是 runner 把明确几何意图改坏了

这次对 `L2_130` 的回看确认了一件事：planner 原始输出已经给出了明确的侧边法兰矩形，只是 runtime 里的 annular side flange rewrite 把它们强行改成了“包住环带厚度 + 外伸”的 overlap-shell slab。

更具体地说：

1. planner 原始矩形是明确的窄法兰：
   - `width = 2.0`
   - `height = 7.5`
   - 还带了 `corner_reference = top_right/top_left`
2. 这其实已经是“显式角点锚定 + 显式窄法兰”的描述。
3. 但旧 rewrite 逻辑把这种 payload 和“模糊 full-width overlap-shell dialect”混在一起处理，最终改写成：
   - `width = 9.5`
   - `height = 7.5`
4. 结果不是修复 planner，而是把 planner 原本正确的局部几何语义改坏了。

因此这轮修复的原则不是继续补特殊 case，而是把 rewrite 边界收紧：

1. 只有模糊的 overlap-shell 方言才允许 annular rewrite 介入。
2. 如果 planner 已经给出：
   - 显式 corner anchor
   - 一条边接近 wall thickness
   - 另一条边明显更小，表现为窄法兰外伸量
3. 那么 runtime 应该保留 planner 的原始几何，而不是再“自作聪明”地扩成环带包覆矩形。

### 2. L2_192 不是只有 evidence 多，而是 prompt budget 本身算错了

我继续拆 `L2_192` 的 prompt 后发现，上一个阶段虽然已经在做 prompt compaction，但 budget 判断和实际喂给模型的文本并不一致：

1. budget 之前统计的是 minified JSON 长度。
2. 实际写进 planner prompt 的却是 `indent=2` 的 pretty JSON。
3. 这意味着很多 payload 在“预算检查”时看起来没超，但真正渲染进 prompt 后会显著膨胀。

同时还发现第二个结构性问题：

1. `action_history` 里不断重复灌入完整 snapshot geometry：
   - `solids/faces/edges`
   - `volume/surface_area`
   - `bbox`
2. 这些内容本来就在当前 `query_snapshot/query_geometry` 里。
3. 所以 planner prompt 同时携带了：
   - 当前态 evidence
   - 历史态重复 geometry
   - reconstructed trace
4. 这就是典型的数据结构重复，而不是单纯“模型太慢”。

## 补充问题定位：prompt-only 的 `create_sketch` 约束不够，真正缺的是 runtime bootstrap continuation

在继续追 `L2_63` 的时候，又暴露出一个很典型、而且比继续堆 prompt 更底层的问题：

1. planner 仍然会在 pre-solid 第一轮直接给：
   - `add_rectangle`
   - `extrude`
2. 虽然 prompt 已经明确写了“如果当前没有 solid 且没有 sketch，要先显式 `create_sketch`”，但 Kimi 仍然会漏掉这一动作。
3. runtime 目前会把首个 sketch primitive 改写成 `create_sketch`，但旧逻辑只执行这个被改写后的动作，导致原来的：
   - `add_rectangle`
   - `extrude`
   被拆到后续轮次
4. 这会带来两个负面效果：
   - 白白增加 round 数
   - 后续又触发更多重复 evidence refresh

这说明：

1. 继续往 prompt 里塞“先 create_sketch”这类提醒，收益已经很有限。
2. 更合理的系统性修复应该是：
   - runtime 在 pre-solid bootstrap 时注入 `create_sketch`
   - 但保留同一 local window 里的后续 primitive / terminal solid 动作
3. 这才是真正泛化的修复，因为它解决的是“模型省略 setup 动作时，runtime 如何保持局部意图连续性”，而不是只修一个矩形 case。

## 本轮修复：bootstrap continuation + required-evidence 并行预取

修改文件：

2. `tests/unit/sub_agent_runtime/test_runner_contracts.py`
3. `docs/cad_iteration/ITERATION_PROTOCOL.md`

改动一：bootstrap continuation

1. 当 `one_action_per_round=true` 且 planner 仍然以：
   - `add_rectangle`
   - `add_circle`
   - `add_polygon`
   作为 no-solid 第一动作时，runtime 仍会按原有逻辑补出显式 `create_sketch(...)`。
2. 但现在不会再把这一轮压缩成“只执行 create_sketch”。
3. 新逻辑会把补出的 `create_sketch(...)` 和原始 local window 重新拼回同一轮：
   - 优先尝试恢复成完整的 `create_sketch + profile + terminal completion`

意义：

1. 减少 setup-only round。
2. 减少因为 setup-only round 额外触发的：
   - `query_snapshot`
   - `query_sketch`
   - 后续 planner 轮次
3. 这比继续给模型塞更多“请先 create_sketch”的提示更稳，因为它修的是 runtime 对局部意图的破坏。

改动二：required-evidence prefetch 并行化

1. `_maybe_prefetch_required_evidence(...)` 之前是串行执行：
   - `query_snapshot`
   - `query_sketch`
   - `query_geometry`
   - `query_topology`
   - `validate_requirement`
2. 这些工具在同一步上的 required-evidence 预取大多是独立的，所以现在改成并行 `asyncio.gather(...)`。
3. 查询结果仍然分别落盘，不会损失 inspectability。

意义：

1. 这是对 Claude Code “工具分区 + 独立查询并行化”思路的第一次真正实现，而不是只停留在 prompt 文案。
2. 它不直接减少 token，但能减少 round 之间的等待时间，特别是 relation-heavy / inspection-heavy case。

### 1. L2_130：annular 窄法兰误改写不只是 corner_reference

新的 live planner 方言里，两侧法兰不再总是写成 `corner_reference=top_right/top_left`，而是会直接给出：

1. `center=[-27,-25], width=2, height=7.5`
2. `center=[27,-25], width=2, height=7.5`

这类 payload 已经表达了“外侧窄法兰 + 壁厚高度”，不应该再被 runtime 扩成 overlap-shell slab。

本轮修复：

1. annular preserve 逻辑从“只保留 corner-anchored 明确窄法兰”扩展到：
   - 只要是明确的窄法兰尺寸
   - 且横向 side 候选已经落在 outer side 附近
   - 就保持 planner 原 payload
2. 同时避免把 `corner=[2,-25]` 这种 axis-swapped 模糊旧方言误判成显式法兰：
   - preserve 现在只看横向 outer-side 候选
   - 不再把 `baseline_y=-25` 这类纵向坐标误当成 side anchor

意义：

1. 这不是针对单个 case 的补丁，而是把 annular 法兰 dialect 分成了：
   - 模糊 overlap-shell 方言：允许 rewrite
   - 明确 outer-side 窄法兰方言：禁止 rewrite

### 2. L2_192：boss extrude 与 later hole family 被错误混入同一 face sketch

这次 `L2_192` 的 requirement 很清楚：

1. 顶面做 central cut + 6 孔阵列
2. 底面只做一个 50mm boss

1. 一个 primary 大圆
2. 一组更小、但中心落在 pitch circle 上的 secondary circles
3. secondary circles 与 primary boss profile 发生越界/重叠
4. additive extrude 生成退化/空实体

本轮修复：

1. 对 existing-solid face-attached additive circle window 新增保护：
   - 如果 requirement 同时明确包含 boss/stud additive 语义
   - 且也明确包含 later holes / pitch-circle / construction-circle 语义
   - 当前 sketch 已经有一个 primary boss circle
   - 新增的小圆 family 是 multi-center、明显更小、并且落在 primary boss 之外/边界外
2. runtime 会把这组 secondary circles 改写为 `construction=true`
3. 这样本轮 additive `extrude` 只消费 primary boss profile，不会把 later hole family 硬并进 additive profile

意义：

1. 这修的是“existing-solid face sketch 中 additive/subtractive family 混线”的底层问题。
2. 它对其他：
   - boss + bolt-circle holes
   - stud + pitch-circle drilling
   - primary additive pad + later patterned cuts
   这类 case 都有帮助。
