## 今日目标

1. 将 V2 从“只有 Claude-Code 风格外壳”推进到“具备语义状态内核”。
2. 引入 runtime-local `FeatureGraph`，降低每轮从 history / evidence blob 反推当前语义焦点的成本。
3. 维持外部 contract 与 MCP 工具面稳定，避免把这次重构直接扩散到 `service.py`。

## 当前判断

1. 现有 V2 已经有：
   - conversation-first loop
   - tool orchestration
   - compaction
   - probe-first policy
2. 现有 V2 仍然缺：
   - authoritative semantic state
   - 模型可直接查询/更新的统一语义对象
3. 所以当前系统仍然过度依赖：
   - latest write health
   - previous tool failure summary
   - validation blockers
   - runtime skill prose
   来拼出下一轮焦点。

## 本轮设计决策

1. 第一刀不做完整 CAD kernel feature tree。
2. 第一刀做轻量 `FeatureGraph`：
   - 节点：intent / body / feature / blocker / evidence / strategy
   - 边：requires / blocks / observes / targets / depends_on
   - 状态：planned / active / observed / satisfied / blocked / failed
3. 第一刀只落在 runtime：
   - `RunState`
   - `ToolRuntime`
   - `ContextManager`
   - `agent_loop_v2`
4. `service.py` 本轮不改，避免语义状态重构与几何工具规则重写互相污染。

## 追加进展：graph refresh policy + benchmark graph ingest

1. V2 runtime 新增两类 turn policy：
   - `graph_refresh_after_read_stall`
   - `graph_refresh_after_incomplete_finish`
2. 这两类 policy 的共同目标是切断“重复 broad read / 重复 finish”的空转：
   - 如果 latest write 之后已经连续两轮 inspection-only，且还没调用过 `query_graph_state`，下一轮会收窄到 graph-refresh 工具面
   - 如果 `finish_run` 之后 validation 仍不完整，下一轮也会先做 graph refresh，而不是立刻再次 finish 或继续泛读
3. `ContextManager` 现在会把这种恢复路径明确写到 prompt payload 中：
   - `stall_summary.recommended_bias`
   - `stall_summary.recommended_next_tools`
   - `objective_health.recommended_next_tools`
4. benchmark 侧已开始直接消费 `trace/feature_graph_round_*.json`：
   - `round_digest.json/md` 会显示 graph summary 与每轮 graph snapshot
   - `benchmark_analysis.json/md` 会显示 final graph digest、graph query count、blocked / unsatisfied nodes
   - `run_diagnostics.json/md` 会聚合 graph coverage / graph query / blocked / unsatisfied / requirement tags
   - `brief_report.md` 会显示 run-level graph coverage 和每个失败 case 的 final graph 摘要

## 这一步解决了什么

1. 之前 graph 已进入 runtime，但模型空转时并没有被真正用起来；现在 runtime 至少会在两个典型失败形态下强制一次 semantic refresh。
2. 之前 benchmark 虽然有 `feature_graph_round_*.json`，但聚合报告完全看不到；现在 graph 终于进入 case-level 和 run-level 分析链。
3. 这仍然不是 case patch：
   - 没有往 runner 里写 benchmark case if/else
   - 没有扩 `service.py`
   - 没有为某个 case 增加 blocker id

## L1_191 暴露出的新问题

1. 这次失败不是 read stall，也不是 finish incomplete。
2. 真正的问题是：
   - V2 先走了过长的 structured pre-solid 链

   - **code-first escape 触发得太晚**

## 追加进展：Phase 2-4 收口切片（legacy compat / kernel alias / runtime 边界）

1. Phase 2：legacy compat 入口已收口
   - `runner.run()` 现在只负责 mode dispatch
   - legacy 兼容模块 `sub_agent_runtime.legacy_runtime.run_legacy_iteration` 直接调用 `runner._run_legacy_inline(...)`
   - 删除了通过修改 `AICAD_RUNTIME_MODE` 再递归回 `runner.run()` 的 env-sentinel 兼容路径
   - 这一步的目的不是“改行为”，而是去掉并发时会污染全局环境变量的旧兼容惯性
2. Phase 3：kernel state query 面已前进一步
   - 新增 `query_kernel_state`，作为 `query_graph_state` 的 domain-kernel 命名别名
   - `query_graph_state` / `query_kernel_state` 默认返回 compact payload：
     - active / blocked / completed / unsatisfied features
     - evidence summary
     - kernel revision / binding counts
   - bindings 与 revision history 不再默认塞进 query payload，避免它们进入 prompt 主证据面时推高 token
   - 但 `trace/feature_graph_round_*.json` 仍保留完整 bindings 与 revision history，inspectability 没有丢
3. Phase 3：kernel binding 细节也收了一个底层 bug
   - 之前同一 round 内若同一 tool 连续回写两次，`binding_id` 会因为只按 `tool_name + round_no` 生成而发生覆盖
   - 现在 binding id 改为带 ordinal + payload digest 的稳定唯一键
   - 这保证 revision history 和 stale binding 标记不会被同轮覆盖掉
4. Phase 4：runtime/tool 边界先薄一层，不直接碰脏着的 `service.py`
   - 新增 `sub_agent_runtime.tool_adapters.KernelStateToolAdapter`
   - `query_graph_state` / `query_kernel_state` / `patch_feature_graph` 先从 `ToolRuntime` 的大 dispatch 中抽出
   - 这样 kernel-state tools 和 sandbox geometry tools 的边界开始明确，后续再拆 `SandboxMCPService` 不会把语义状态逻辑重新塞回 geometry service
