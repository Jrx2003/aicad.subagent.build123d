2. `relation_focus`

3. `relation_eval`
   - “应该有的关系”和“实际观察到的关系”之间的偏差

## 0.1 如果你现在只想先看懂 `relation_index.json`

可以先把一份 `relation_index.json` 当成下面这句话：

> 当前这一步里，系统看到了哪些 CAD 实体，这些实体之间有哪些客观关系，这些关系能不能再组成更高一层的局部结构。

它不是：

1. 不是“验证是否通过”的报告
2. 不是“下一步一定该干什么”的计划
3. 不是完整 CAD 历史树

它更像一张**关系底稿**。

### 最短读法

如果你打开一个 `relation_index.json`，只看这 5 个字段就够了：

1. `source_tool`
   - 这份关系底稿来自哪个工具
   - 常见值：`query_sketch` 或 `query_topology`
2. `focus_entity_ids`
   - 这份底稿最关心哪些对象
3. `entities`
   - 这些对象本身是什么
4. `relations`
   - 它们之间有什么关系
5. `relation_groups`
   - 多个对象能不能组成一个更完整的局部结构

### 一眼先忽略什么

第一次看时，你可以先忽略：

1. `evidence`
   - 这是简短解释，不是核心
2. `summary`
   - 只是统计
3. 很长的 `attributes`
   - 先看 `geom_type / radius / axis_direction / center` 这几个关键字段就行

### 把字段翻成人话

#### `entities`

它回答的是：

> “我现在把哪些东西当成可讨论的对象？”

例如：

1. 一个草图 profile
2. 一条 path
3. 一个圆形 loop
4. 一个 face
5. 一条 edge

#### `relations`

它回答的是：

> “这些对象之间客观上有什么关系？”

例如：

1. `concentric`
   - 同心
2. `coaxial`
   - 共轴
3. `equal_radius`
   - 等半径
4. `coplanar`
   - 共面
5. `connected`
   - 相连
6. `tangent`
   - 相切连续

#### `relation_groups`

它回答的是：

> “这些关系能不能再往上组成一个更像 CAD 语义的小结构？”

例如：

1. `annular_profile`
   - 一个环形截面
2. `annular_edge_pair`
   - 一对内外圆边
3. `annular_cylindrical_pair`
   - 一对内外圆柱壁
4. `sweep_profile_pair`
   - 一条 path 和一个截面 profile 的配对

### 推荐阅读顺序

第一次看任何 `relation_index.json`，建议用这个顺序：

1. 先看 `source_tool`
2. 再看 `focus_entity_ids`
3. 再看 `relation_groups`
4. 最后才回去看 `entities` 和 `relations`

原因很简单：

1. `relation_groups` 最接近“人能理解的 CAD 结构”
2. `entities + relations` 更像底层原料

也就是说：

> 先看“这里有没有形成一个环、一个配对、一个同轴内外壁”，再去追它到底是哪些 face/edge 组成的。

## 2. 先给结论

在 `L2_90` 里，当前系统的关系型反馈状态是：

1. `relation_base` 已经真实落地
   - 在 `query_sketch.relation_index`
   - 在 `query_topology.relation_index`
2. `relation_focus` 目前没有命中
   - 因为第一版 parser 主要面向 `sweep / annular`
3. `relation_eval` 目前也没有命中
   - 原因同上

所以这条 case 展现出来的不是“关系评估”，而是：

> 我们已经把 CAD 模型的一部分客观关系，稳定写成了新的结构化反馈；但任务相关的关系筛选与偏差评估，目前只覆盖更小的一类 case。

## 6. 再看 `query_topology.relation_index`

### 6.1 最值得看的真实文件

- [round_04_inspection_only.json](/Users/jerryx/code/aicad.subagent.iteration/benchmark/runs/20260327_084100/L2_90/queries/round_04_inspection_only.json)

这份文件同时有三样东西：

1. `resolved_query_topology_options`
   - runtime 实际请求了什么窗口
2. `candidate_sets`
   - 面向 planner 的“候选集”
3. `relation_index`
   - 面向关系层的“关系底稿”

这三样东西不要混淆。

### 6.2 `candidate_sets` 和 `relation_index` 的区别

`candidate_sets` 是：

1. 给 planner 做“选择”用的
2. 告诉它：
   - 哪些是 `top_faces`
   - 哪些是 `outer_edges`
   - 哪些是 `bottom_outer_edges`

`relation_index` 是：

1. 给 planner 做“理解”用的
2. 告诉它：
   - 哪些 face/edge 是实体
   - 这些实体之间有哪些关系

可以理解成：

1. `candidate_sets` 更像“候选目录”
2. `relation_index` 更像“关系图谱”

### 6.3 `L2_90` 里的 `relation_index` 字段是什么意思

还是看：

- [round_04_inspection_only.json](/Users/jerryx/code/aicad.subagent.iteration/benchmark/runs/20260327_084100/L2_90/queries/round_04_inspection_only.json)

重点字段如下。

#### `entities`

这里每一项都对应一个真实拓扑对象。

比如某个 face 会带：

1. `entity_id`
   - 一般就是 `face_ref`
2. `entity_type`
   - `topology_face`
3. `ref`
   - step-local ref
4. `attributes`
   - 真正用于关系计算的底层属性

在 `L2_90` 里，最重要的 `attributes` 通常是：

1. `geom_type`
2. `center`
3. `normal`
4. `axis_origin`
5. `axis_direction`
6. `radius`

这些字段不是“讲给人听的解释”，而是关系计算的原料。

#### `relations`

这里每一项是一条客观关系。

当前拓扑侧支持的基础关系主要来自：

- [service.py#L3546](/Users/jerryx/code/aicad.subagent.iteration/src/sandbox_mcp_server/service.py#L3546)

在这段代码里，系统会对返回窗口里的 face/edge 两两比较，提取：

1. `concentric`
2. `coaxial`
3. `equal_radius`
4. `coplanar`

#### `relation_groups`

这是比 pairwise relation 更像 CAD 语义的一层。

例如：

1. `annular_edge_pair`
2. `annular_cylindrical_pair`

`L2_90` 不一定会命中 annular 类 group，但这个字段非常重要，因为它代表：

> 我们不是只输出孤立的“这两条边共轴”，而是开始输出“这一组对象组成了一个更有语义的结构模式”。

## 8. 这份 case 对项目方向的意义

`L2_90` 这条 case 给出的结论很明确：

1. 关系层已经不再是文档概念

   - 已经进了 prompt
2. 但它目前主要完成的是“关系底稿”
3. 真正任务相关的关系筛选和偏差评估，还只覆盖更小的一组任务

所以这条 case 最适合帮助新人建立下面这个心智模型：

   - 偏目标选择
3. `query_topology.relation_index`
   - 偏模型结构理解
4. `relation_focus / relation_eval`
   - 偏任务条件下的关系筛选与偏差分析
