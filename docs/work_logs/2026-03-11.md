## Follow-up: Render Step Alignment + L1 Batch Run

### Context

Issue observed in benchmark artifacts: some `render_view` outputs showed workspace axes only while STEP artifact was valid.

### Root cause confirmed

In action rounds, planner-provided `inspection.render_view.step` could be stale (for example `step=1`) and was reused after later actions, causing renders to target pre-solid state.

### Runtime fix

File:

- `src/sub_agent_runtime/runner.py`

Change:

1. Added action-state-aware render step resolution:
   - `requested_step` is clamped to latest action snapshot step when behind current state.
2. Added observability fields in query record:
   - `resolved_render_view_options.requested_step`
   - `resolved_render_view_options.step`
   - `resolved_render_view_options.step_adjusted_to_latest`
3. Added prompt-safety handling for workspace-axis-only renders:
   - keep per-action PNG as evidence.
   - do not promote workspace-axis image to `outputs/latest_render_view.png`.
   - do not attach workspace-axis image to LLM prompt payload.

### Tests

Updated:

Added regression:
