# 2026-04-02 工作记录

## 今日目标

1. 将迭代循环收敛到更严格的单步 ReAct：每轮只执行 1 个 action，动作后立即基于反馈重规划。
2. 放宽预算与时限（轮次 / LLM token / timeout），避免“预算先耗尽、动作未完成”。
3. 降低 relation 反馈噪声：保留可解释性与可追溯性，但在 planner 侧优先提供高信号摘要。
4. 引入轻量 hooks 生命周期（pre/post round, pre/post action, failure），减少后续改动继续堆叠在 `runner.py`。
5. 每次关键改动后执行 sampled L2 全 10 case；重点分析失败样例，抽象可复用根因而非单 case 打补丁。

## 约束与原则

1. 不删除 inspectability：`prompts/plans/actions/queries/outputs/summary.json` 必须保持完整可审计。
2. 不以单案例过拟合为目标；修复必须能解释“为什么会提升随机 case”。
3. 卡住时先写清楚根因与失败边界，再继续改动。

## 对 scripts/describe 思路的迁移评估（阶段结论）

1. 可迁移价值：
   - 把“关系/几何大表”压缩成“feature-level 语义摘要”（例如孔阵列、槽、凸台、对称分布）再给 planner。
   - 在失败诊断时，优先比较“目标特征是否存在 / 数量是否正确 / 位置是否匹配”，而不是让模型反复遍历原始实体列表。
2. 不直接照搬的原因：
   - 当前 describe 脚本是独立实验代码，不是 runtime 协议的一部分。
   - 直接接入会引入额外耦合和不稳定输入面，短期应先在 `codegen` 侧做 relation 摘要策略，保持 contract 稳定。
3. 当前采用策略：
   - 先在现有 relation payload 上做“planner-facing 摘要压缩 + 关键 blocker 前置”；
   - 观察 L2 失败分布，再决定是否把 feature extractor 升级为正式 query surface。

### 19. 进一步定位到更底层的问题：`codegen` 在解析阶段提前截断 planner 动作窗口（本轮新增）

1. 新证据：
   - `L2_130` 旧失败工件中，`raw_content` 明确返回了 `add_path + add_rectangle`，但落盘后的 `actions` 只剩 `add_path`。
   - `L2_192` 旧失败工件中，`raw_content` 明确返回了 `create_sketch + add_circle + cut_extrude`，但 `actions` 只剩 `create_sketch`。
   - 这说明很多失败并不是先在 `runner` 中被拆散，而是更早在 `codegen` 的解析阶段就被强行“单动作化”了。

2. 根因：
   - `src/sub_agent/codegen.py` 中 `_MAX_ACTIONS_PER_ROUND = 1`，会把 planner 已经生成好的局部动作链硬裁成 1 条。
   - 这会直接破坏：
     - `L2_130` 这类 pre-solid 复合截面窗口（`add_path + add_rectangle`）
     - `L2_192` 这类 existing-solid 面编辑完成窗口（`create_sketch + add_circle + cut_extrude`）
   - 也就是说，之前新增的 runner 受控窗口升格能力，在很多真实失败 case 上根本拿不到完整输入，自然发挥不出来。

3. 本轮修复：
   - `src/sub_agent/codegen.py`
     - `_MAX_ACTIONS_PER_ROUND: 1 -> 3`
     - prompt 输出规则改为：
       - 默认仍偏向 1 个动作
       - 允许返回最多 3 个、属于同一个 local window 的动作链
       - 明确告知 runtime 可能只执行前缀，因此动作链必须保持局部连贯
   - `src/sub_agent/prompts/codegen_action.md`
     - 同步修改系统 prompt：最多 3 个动作，但必须属于一个 coherent local work window
   - `docs/cad_iteration/*`
     - 协议层从“plan one next step”修正为“plan next local repair window”
     - 明确 codegen 必须保留 planner 的局部动作窗口，不能在 runtime 选择之前就截成 1 条

4. 为什么这是可泛化提升而不是 case patch：
   - 这是 planner/runtime contract 的修复，不依赖具体几何模板。
   - 只要模型已经给出了正确的局部动作链，系统就不应该在 parser 这一层把它毁掉。
   - 这更接近 Claude Code 风格：模型保留决策窗口，runtime 再结合状态/预算选择可执行前缀，而不是 parser 先做生硬裁决。

### 19. 进一步定位到执行层 bug：`add_path` 弧段重写器错误假设了 planner 只会输出 `start_angle/end_angle`

2. 真正的断点在 `runner.py::_rewrite_add_path_center_angle_segments_to_endpoints(...)`：

   - planner 给的是合法弧段格式：

     - `end_angle`

   - `angle_degrees/arc_degrees`

### 21. 对 `scripts/describe/experiment_1` 的判断

我也同步看了同事在 `scripts/describe/experiment_1` 里的思路，结论如下：

1. **不建议直接照搬进主循环**：
   - 它偏离线 STEP feature detection；

2. **但思路很值得迁移**：
   - 重点不是“多做一个大分析器”，而是把几何关系反馈从原始 relation graph，压成对 planner 更有用的 feature digest。
3. 对当前项目最有价值的迁移点有两个：

   - “所在面 / 轴向 / 位置 / 是否贯穿”这种 planner 真正能直接使用的局部摘要
4. 这和 `2026-04-01_Claude_Code_Migration_Guide.md` 里强调的方向一致：
   - 减少大而杂的原始反馈
   - 增强 runtime 对局部工作窗口的识别
   - 让模型基于更紧凑、更高价值的局部状态做决策

所以我后续会优先走：

1. **relation / topology -> planner digest 化**
2. **feature-window 化**
3. **模型驱动窗口执行**

而不是把整个 `describe.py` 原样并进 runtime。

   - 不再错误降级成一个 `hole`
3. **显式 hole window 仍然保留 direct-hole 快径**
