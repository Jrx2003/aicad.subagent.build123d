## 今天实际定位到的根因链

### 第一层：证据生命周期错误

1. 旧逻辑里，query 结果在 planner 一轮结束后就会被清空。
2. 这会让 planner 下一轮即使刚查过 rail/profile/topology，也仍然像没有看过一样继续猜。
3. 现在这条链已经修复，证据会持续存在直到被新的 geometry/topology 变化失效。

### 第二层：执行参数和证据历史分叉

1. 真实 probe `test_runs/20260324_210000` 暴露出：
   - planner 第 3 轮给了 `create_sketch(path_ref=..., frame_mode=\"frenet\")`
   - 但 `query_sketch` 仍显示 profile sketch 是 `plane=XY, origin=[0,0,0]`
2. 根因不是 planner 不会调用 `path_ref`，而是系统只接受非常窄的 `frame_mode=\"normal_to_path_tangent\"`。
3. 当模型输出 `frenet`、`true`、`perpendicular` 这类语义近似但 schema 不严格的值时，执行层没有把 `path_ref` 解析成真正的 path-end frame。
4. 结果就是：
   - 执行层和证据层看到的不是同一个 sketch
   - LLM 继续在错误的 `XY@[0,0,0]` 上规划
   - 最终 `sweep` 只能得到零体积壳

### 第三层：annular profile 被误识别

1. 同心双圆 profile 原先在 `query_sketch` 中被记成：
   - `outer_loop_count=2`
   - `inner_loop_count=0`
2. 这意味着系统把它理解成两个独立 outer loops，而不是一个 annular profile。
3. 即使 planner 实际上已经画出了两个同心圆，validator 仍会认为 hollow sweep profile 不成立。
4. 现在已改为把同心多圆折叠成：
   - `outer_loop_count=1`
   - `inner_loop_count=1`
   - `nested_relationship=concentric_frame`

## 继续推进：Loft / Local Anchor / Pattern 能力补强

### 新增根因 3：registry 暴露了 pattern 动作，但 translator 实际上是空实现

1. 在继续检查 L2 失败簇时，我重新核对了 `feature_patterns` 相关链路。
2. 结果发现一个非常基础但影响很大的问题：
   - planner prompt 会看到 `pattern_linear` / `pattern_circular`
   - registry 也把它们当成合法 typed action
   - 但执行层此前没有真正把“上一个 additive seed feature”复制出去
   - 这意味着它们在产品能力上近似于“假动作”
3. 这不是某个 case 的特殊问题，而是系统对能力面的虚假暴露：
   - LLM 以为可以 pattern 一个 seed rib/tooth/boss
   - runtime 实际上没有对应的稳定构造能力
   - 最后只会出现执行空转、后续 validator 无法解释的 feature 缺失，或者 planner 被迫退回到更低效的重复手工构造

### 本轮通用修复：让 pattern 动作真正可执行

1. `pattern_linear` 现在会复制最近一次 additive seed feature，并按：
   - `count`
   - `spacing`
   - `direction`
   做线性平移阵列。
2. `pattern_circular` 现在会复制最近一次 additive seed feature，并按：
   - `count`
   - `center`
   - `axis`
   - `total_angle`
   做旋转阵列。
3. 为了支持这一点，执行层新增了 `_aicad_last_additive_feature` 跟踪：
   - additive `extrude`
   - additive `loft`
   - additive `sweep`
   - additive `revolve`
   都会在成功时更新最近 seed。
4. subtractive 或会破坏 seed 语义的动作会清空该 seed：
   - `cut_extrude`
   - `hole`
   - `sphere_recess`
   - subtractive `revolve`
5. 如果 planner 在没有 seed feature 的情况下调用 pattern，runtime 不再默默失败，而是上抛结构化 blocker：
   - `feature_pattern_seed`
   - warning code: `execution_warning_missing_pattern_seed`

### 为什么这条修复是通用的

1. 它解决的不是某个 benchmark prompt 的写法，而是系统能力面的一致性问题：
   - prompt 可见的动作
   - registry 声称支持的动作
   - runtime 真正可执行的动作
   三者现在是一致的。
2. 这会直接影响：
   - repeated boss / tooth / rib / fin
   - bolt-circle 上的 additive protrusion
   - linear/circular 阵列的局部 feature
3. 这类任务以前看起来像“LLM 不会选动作”，但更准确地说，是系统把一个并不存在的动作暴露给了 LLM。

## 继续推进：Loft / Local Anchor / Prompt Evidence Hygiene

### 新增根因 1：历史 step-local ref 污染 planner

1. `L2_172` 暴露出一个很典型的 evidence-hygiene 问题：planner 在第 6 轮继续复用早期步骤的 `face_ref`。
2. 关键证据：
   - `benchmark/runs/20260324_161918/L2_172/prompts/round_06_request.json`
   - `benchmark/runs/20260324_161918/L2_172/plans/round_06_response.json`
   - `benchmark/runs/20260324_161918/L2_172/trace/events.jsonl`
3. 问题不在于 runtime 不知道 ref 已失效，而在于 prompt 仍把旧 `face_ref` 放在 `action_history / reconstructed_action_trace` 里，模型会把这些历史 ref 当成当前可重用 target。
4. 已修复：
   - planner-visible `action_history` 和 `reconstructed_action_trace` 现在会对历史 step-local `face_ref/edge_ref/path_ref/profile_ref` 做 stale sanitize。
   - 只保留当前 step 的 ref 原样可见；更老的 ref 会被替换成 `<stale ... from step N; use current evidence>`。
5. 这样做不影响 runtime replay，因为运行时仍保留真实 action history；只修 prompt 暴露层。

## 新发现：`L2_90` 的 evidence lifecycle 根因

### 失败现象

1. 在 `benchmark/runs/20260324_161918/L2_90` 中，planner 到第 6 轮其实已经给出了合理动作：
   - `hole`
2. 但 runtime 又插入了 `query_topology` barrier，直接耗尽最后一轮，最终：
   - `converged=false`
   - `validation_complete=false`
   - benchmark `score=0.1796`

### 根本原因

1. 当前 evidence lifecycle 把 `query_topology` 的“是否 current”简单绑定到 `latest_step`。
2. 这在 face-attached sketch 任务里是错的，因为：
   - `create_sketch`
   - `add_circle`
   - `add_rectangle`
   这些步骤虽然会让 `step` 增长，但并没有改变 host solid 的 topology。
3. 在 `L2_90` 里：
   - step 4 已经有可用的 disk top-face topology
   - step 5/6/7 只是 face-attached sketch/profile 绘制
   - runtime 却把 step 4 的 topology 证据判成 stale
4. 结果是：
   - round 6 planner 已经准备执行 `hole`
   - runtime 仍返回 `targeted_edit_requires_current_query_topology`
   - 整个 case 因 barrier 耗尽回合

### 这说明什么

1. 证据 stale 判定必须区分：
   - latest step
   - latest topology-relevant/material step
2. 对 `query_geometry/query_topology/render_view` 这种依赖实体形状的证据，应该绑定到“最后一次真正改变实体”的 step，而不是绑定到最后一个 sketch step。
3. 对 `query_sketch` 才应该绑定到最新 step，因为 sketch state 确实在每次 `create_sketch/add_circle/...` 后变化。

### 已落地修复

1. runner 新增 `latest_topology_step` 概念：
   - 跳过 `create_sketch/add_circle/add_rectangle/add_polygon/add_path/snapshot`
   - 只把真正影响实体 topology 的动作当成 topology freshness 边界
2. `query_geometry/query_topology/render_view` 的 `current/stale` 判定已改为基于 `latest_topology_step`。
3. targeted face/edge edit barrier 也已改为基于 `latest_topology_step`，不再因为 sketch-only steps 而强制 re-query topology。
4. `feature_target_face_subtractive_merge` 也已经加入 runtime 的 repair-driven blocker reopen 集合。

## 新发现：local-anchor / revolve translator / face-hole translator 的执行层缺口

### 1. `L2_90` 不再被 topology barrier 打断，但暴露出更深一层的 direct-feature translator 问题

#### 已确认修复

1. round 4 planner 返回的 `hole(face_ref="F_...", centers=[...])` 不再被 runtime 以 `targeted_edit_requires_current_query_topology` 打断。
2. 这次已经能看到 runner 把 raw `face_id` 归一化为 step-local ref：
   - `face_ref_canonicalized_from_current_evidence`
3. 说明当前证据链已经允许 planner 直接消费 `query_geometry` 返回的 face id，不需要被迫再走一次 `query_topology` round-trip。

#### 新暴露的根因

1. `hole(face_ref=...)` translator 在 face-attached 分支中先调用：
   - `_aicad_workplane_from_face_id(...)`
   - 然后又 `.add(result.findSolid())`
2. 这会把 face workplane 与 solid 同时留在 CadQuery stack 上。
3. 在 `L2_90` 中，最终报错是：
   - `ValueError: If multiple objects selected, they all must be planar faces.`
4. 这不是 `L2_90` 特例，而是所有 direct `hole(face_ref=...)` 都可能踩到的 translator 结构性错误。

#### 第二个伴随问题

1. planner 在 `L2_90` round 4 返回的 hole centers 是世界坐标三元组：
   - `[x, y, z]`
2. 旧 hole translator 只截取前两个分量，等价于把世界坐标误当 face-local 2D 坐标。
3. 所以即便没有上面的 CadQuery workplane 报错，孔位也会错。
4. 这说明 direct face feature 当前缺少一个必要步骤：
   - `world-space feature centers -> resolved face workplane local 2D`

#### 已实施修复

1. runner 已支持：
   - raw `face_id` / `edge_id` 基于当前 `query_geometry/query_topology` 证据归一化为 step-local refs
2. hole translator 已改为：
   - face-attached 分支不再依赖 multi-object workplane stack
   - 对 world-space 3D centers 先通过 resolved face workplane 投影到 local 2D
   - 使用显式 cutter solid 做 subtractive cut，而不是继续调用脆弱的 `.hole()` face-stack 路径

### 2. `L2_96` 的问题不是 planner 没理解 taper，而是 translator 把 sloped revolve profile 扁平化了

#### 证据

1. prompt 明确要求：
   - 底边从 `X=12.5` 到 `X=20.0`
   - 竖直向上 `35.0`
   - 顶边回到 `X=10.75`
   - 再用一条斜边闭合
2. planner 实际给出的 polygon points 也确实包含斜边：
   - `[(12.5,0), (20,0), (20,35), (10.75,35)]`
3. 但最终 evaluator 显示生成体是：
   - `CYLINDER + CYLINDER`
   而 GT 是：
   - `CONE + CYLINDER`

#### 根因

1. `add_polygon` translator 会尝试从 polygon points 推断 `_aicad_stepped_profile['bands']`。
2. 这个 shortcut 本来适合 orthogonal band / stepped profile。
3. 但它错误地也吃掉了带斜边的 revolve profile。
4. 于是原本应该走真实 revolve 的 tapered/conical 几何，被提前降级成了 banded cylindrical approximation。

#### 已实施修复

1. `_infer_revolve_profile_bands_from_points(...)` 现在只允许 orthogonal band profile。
2. 一旦 profile 中存在真正的 sloped segment：
   - 不再推断 `bands`
   - 回退到精确的 sketch revolve 路径
3. 这条修复是通用的：
   - 解决所有“revolve profile 含斜边，却被 flatten 成 stepped cylinder”的 case

### 3. 当前对 L2 failure family 的重新分类

1. `evidence lifecycle / topology stale`：
   - 这一类已经明显收敛，`raw face_id -> explicit ref` 也打通了。
2. `direct face feature translator`：
   - 仍有执行层 bug，需要重点盯 `hole/sphere_recess/local cut window` 这类 face-attached direct feature。
3. `profile-shape semantics`：
   - revolve / loft / stacked profile 还会出现“planner 有效、translator 扁平化/validator 过早放行”的问题。
4. `planner blind guessing`：
   - 仍存在，但不再是当前最主要瓶颈；更多失败已经下沉到 translator/validator 语义层。

### 4. `L2_148` 继续深挖后的新根因：不是单纯 polygon phase，而是 regular polygon size semantics + benchmark face-type 误伤叠加

#### 4.1 已确认的事实

#### 4.3 继续深入后发现第二层问题：benchmark evaluator 在这类 polyhedron 上被 STEP face typing 误导

1. `tri_30_hex_30_apothem20` 的新 evaluator 手工重评结果：
   - `scores.final_score = 0.9343`
   - `difference_notes = ['topology-derived face/edge metrics suppressed due STEP face-type mismatch under otherwise close global geometry']`
2. 旧 evaluator 里，`L2_148` 被低分的直接原因不是 bbox/volume/area：

   - `volume_score = 1.0`
   - `surface_area_score ≈ 0.866`
3. 真正把分数打穿的是：
   - `face_score = 0`
   - `edge_score = 0`
   - `face_type_score ≈ 0.151`

5. 但 `L2_148` 的 prompt 和 GT 渲染图都说明它只是一个纯平面 polyhedron，不应该由这些 non-planar face-type 主导 anchor scoring。
6. 结论：
   - 当前 benchmark evaluator 把 STEP 导入后的 face decomposition 差异，当成了真实几何错误。

#### 4.4 已实施修复

2. `query_sketch` 现在会暴露：
   - `regular_polygon_size_mode`
   - `regular_polygon_circumradius`

3. prompt/card 也已补充：
   - 当 requirement 用 `center-to-side` / `line-offset from center` 表述 regular polygon 尺寸时，应显式输出 `size_mode="apothem"`

   - 则 suppress `face/edge/face-type/feature-anchor` 这组 topology-derived 误伤指标
5. 这是 benchmark 修复，不进入 planner/runtime 产品闭环，不会向 LLM 泄露 GT 信息。
