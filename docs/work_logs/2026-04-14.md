持续用当前 Build123d 仓库做 sampled L1/L2 真 benchmark sweep，并把原本容易被慢 case 卡死的串行全量改成“L1 分片 + 缺失 case 定向补跑 / L2 全量前半程 + 后半程补跑”的执行方式，确认当前项目主执行链是可运行的；截至暂停前，`benchmark/runs/20260414_140401`、`benchmark/runs/20260414_140402`、`benchmark/runs/20260414_140403` 已覆盖 L1 的 9 个 case，`benchmark/runs/20260414_140058` 已稳定落盘 L2 前半程 case，简单轴对称体与部分局部特征 case 可以 1-3 轮直接完成写模、验证和 evaluator 对比。

持续定位 Build123d 当前最主要的泛化短板，问题已经进一步从“API 不会用 / 代码根本跑不通”收敛成“validator evidence grounding 与 read-stall closure 不足”：L1 侧出现了多例 `VALIDATOR_MISMATCH`，包括 `L1_122`、`L1_157`、`L1_159`，其共性都是 evaluator 已判定 STEP 几何通过，但 runtime 仍因为 `insufficient evidence` 或重复 validation 无法收口；同时 `L1_148` 暴露出 1 例 runner exception，说明 orthogonal union family 仍有独立的 runtime 健壮性问题需要补。

持续把后续优化重点收敛到真正值得投的共享 family，而不是再堆零散 prompt 文案：当前最需要继续压的是 half-shell + directional hole、explicit-anchor hole / countersink、以及 validator 对“几何已正确但证据未闭环”这类场景的 completion closure。换句话说，Build123d 运行面已经证明可用，下一阶段的主要矛盾不再是“能不能生成”，而是“能不能稳定在有限轮次内判定自己已经做对”。
