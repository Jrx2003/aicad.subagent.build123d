## 今日追加进展（球形凹槽 / direct feature pattern）

### 4. 为 L2 的球形凹槽 / pattern 类需求补了通用 typed primitive，而不是 case 特判

本轮没有往 runtime 里塞 `if case_id == L2_63` 这类逻辑。
改的是三层通用能力：

1. 新增 typed action: `sphere_recess`
   - 语义：在目标面上切除一个或多个半球形凹坑。
   - 参数：
     - `radius` / `diameter`
     - `position` / `center`
     - `centers` / `positions`
     - `face_ref`

   - `spherical_face_edit`
   - `feature_patterns`

这里的关键点：
- rewrite 只依赖 requirement text、当前 action window、当前 state；
- 不依赖 benchmark case id；
- 不依赖 GT；

## 今日追加进展（三）：L2 path sweep 通用修复链

### 10. 先确认问题不是 GT 泄漏，而是 pre-solid sweep 这条通用链路本身不稳

针对 `L2_149`，今天没有去看 GT 反推动作序列，而是按 artifact 顺序排查：

1. 先看 `trace/events.jsonl`
2. 再看 `prompts/round_XX_request.json`
3. 再看 `actions/round_XX_action_YY_result.json`
4. 最后才看 benchmark evaluation

这样做的目的，是保证所有修正都建立在 runtime 自己暴露出来的执行证据上，而不是用 GT 倒推 prompt。

### 11. 修正一：pre-solid `create_sketch` 不能因为“还没有 solid”就被跳过

最早的 `L2_149` 失败里，round 2 的 profile `create_sketch(YZ)` 会被 runtime 重写成 `snapshot`。
根因是 `_extract_latest_sketch_context()` 只看 `create_sketch`，看不到前面的 `add_path` 已经建立了一个 rail sketch context。

修复：

1. `_extract_latest_sketch_context()` 现在会把 `add_path` 也视为 sketch-context carrier。
2. 因此在 no-solid 状态下，`add_path -> create_sketch(plane switch)` 不会再被误判成冗余。

这是通用修复，影响所有 pre-solid multi-sketch `sweep/loft` 场景。

### 12. 修正二：open-path rail 不能只从 `pendingWires` 恢复

随后排查发现，sweep rail 在 replay 时会丢失，因为 helper 只查 `pendingWires`。
但 open path 很常见地会存在于：

1. `pendingEdges`
2. `vals()`
3. `val()`

修复：

1. 新增 `_aicad_resolve_path_shape(...)`
2. `query_topology` / replay helper 现在会从 `pendingWires -> pendingEdges -> vals()/val()` 逐层恢复 rail
3. 如果存在多条 `pendingEdges`，会尝试 `cq.Wire.assembleEdges(...)`

这也是通用修复，不依赖 `L2_149`。

### 13. 修正三：`sweep` 不能依赖一个没有 pending wires 的临时 Workplane

再往后，`sweep` 从“找不到 path”变成了：

- 能找到 path
- 但 profile 消费方式不对

最开始的实现是：

1. 从 profile sketch 解析出一个 face
2. 再 `cq.Workplane(obj=face).sweep(path)`

这会在 CadQuery 内部走到 `popPendingWires()`，导致 `No pending wires present`。

修复分两步：

1. 先改成 shape-level sweep，不再依赖临时 Workplane 的 pending wires。
2. 再把 hollow profile 的 sweep 改成更稳的布尔流程：
   - outer wire sweep 成实心
   - inner wires 各自 sweep 成实心
   - 再做 boolean cut

这样能避免 annular profile 直接 face-sweep 时出现 shell/零体积的问题。

### 14. 修正四：`tangent_line` alias 会被静默丢弃，导致 rail 被截断

`L2_149` 继续查 trace 后发现另一个关键问题：

round 1 planner 产出的 rail 最后一段是：

- `type=tangent_line`

但 runtime helper 只认：

1. `line`
2. `tangent_arc`

结果是最后一段被静默丢弃，round 1 只保留了：

1. 第一段 line
2. 第二段 arc

随后 planner 才会在 round 2 又补一个 `add_path`，浪费 token，并继续把 profile / sweep 对齐搞乱。

修复：

1. registry normalization 现在把 `tangent_line` / `add_line` 统一归一化到 `line`
2. runtime path helper 也直接把 `tangent_line` 当 `line` 执行

这也是通用兼容修复。

## 今日追加进展（三）：`L2_90` 失败根因已定位

### 12. `L2_90` 不是工具缺失，而是 pattern 语义校验过宽

`L2_90` 已经跑完并失败：
- `benchmark/runs/20260323_105703/L2_90/evaluation/benchmark_eval.json`
- `final_score = 0.2237`

从 trace 看：
- planner 先正确做出带中心孔的圆盘；
- 然后主动请求 `query_topology`；
- 最后只做了一个 top-face seed hole：
  - `create_sketch(face_ref=top)`
  - `add_circle(center=[0,35], radius=7.5)`
  - `cut_extrude(through_all=true)`

也就是说：
- planner 没有完成 circular pattern；
- runtime 也没有把这个 seed feature 自动扩展成 6 个孔；
- 但 validator 仍然把 `feature_pattern` 判成 `pass`。

进一步拆 `validate_requirement` 后发现：
- `feature_pattern` 的 pass evidence 是 `found material pattern action`
- 但真实 history 里没有 `pattern_linear/pattern_circular`
- 也没有 repeated `hole/sphere_recess centers`

当前最可疑的通用 bug 是：
- `_history_has_explicit_repeated_profile_window(...)`
- 把“基体草图里的两个同心圆（外圆 + 中心孔）”误判成了 repeated pattern evidence

这属于通用语义错误，不是 case-fit：
- 任何“基体本身就包含多个轮廓”的 revolve/extrude prompt，都可能被错误当成 pattern 已完成

后续修正方向：
1. repeated-profile window 不能只看“有多个 profile”，还要区分：
   - nested section / concentric section
   - repeated same feature at distinct centers

## 今日追加进展（五）：新增 `L2_149` 专项根因排查报告

今天新增了一份单独的分析文档：

- `docs/work_logs/2026-03-23_L2_149根因排查报告.md`

这份报告不是简单记录“失败了”，而是把问题按证据链拆开：

1. requirement 真正要求什么
2. planner 在哪一步把 rail 写错
3. runtime 为什么没有把坏 rail 拦下来
4. validator 为什么会过早给出 `validation_complete=true`
5. 这到底属于系统设计问题、证据问题、还是动作语义问题
6. 下一步最有性价比的修改顺序是什么

当前结论已经比较明确：

- `L2_149` 不是缺少 `sweep` 动作
- 也不是经典 `face_ref / edge_ref` 选错
- 根因是 `path-sweep` 在 pre-solid 阶段缺少中间语义层，导致：
  - planner 可把 `line + arc + line` 写成 `line + arc + arc`
  - runtime 在 `DisconnectedWire` 下仍继续执行
  - validator 只看到了“有 hollow solid”，没真正验证“是不是沿正确 rail 扫出来的弯管”
