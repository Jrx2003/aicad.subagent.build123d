## 当前评估结论

### 1. 当前系统的真正问题不是 query 不够，而是中间表示不够稳

当前主链路的问题不是简单的“少几个 validator”：

1. `runner` 里同时承担了：
   - evidence prefetch
   - relation feedback 组装
   - barrier 检测
   - action rewrite
   - blocker 聚合
2. `codegen` 里同时承担了：
   - prompt payload 构造
   - prompt 压缩
   - 历史 sanitize
   - 大量策略性自然语言规则
3. `registry` 里同时承担了：
   - action schema / alias normalize
   - requirement semantics parser
   - capability bundle 选择
4. `validate_requirement` 继续扩张的话，会进一步和 relation / barrier / planner policy 重叠。

结果就是：

> planner 不能围绕一个稳定的局部目标推进，只能在大 prompt 和很多 patch 之间来回漂移。

### 2. `cad_xjm` 提供的可迁移价值

#### `analyze.py`

价值在于“最小硬规则”：

1. wire closed
2. self-intersection

这类检查应该进入 `query_sketch` / barrier，而不是继续放在 prompt 说明里让模型自己判断。

#### `modeling.py`

价值不在于直接搬一套类，而在于两层思想：

1. sketch / feature dependency graph
2. dirty propagation

这非常适合演化为：

1. `active_surface`
   - 当前轮只能在哪个局部面/草图窗口/路径端点附近工作
2. `edit_frontier`
   - 这一步之后哪些证据会失效，哪些 ref 会失效
3. `rollback_scope`
   - 如果这轮失败，最小回滚边界是什么

#### `cadnotes_20260327.md`

价值在于把边界讲清楚：

1. 输入明确时，CAD 应该能给强校验
2. 输入不明确时，CAD 只能给基础 hard rules
3. 所以系统不能无限把“需求语义推断”塞给 CAD kernel 一侧

这与本仓库当前的膨胀点完全一致。

## 目标架构

### 1. Surface-Bounded ReAct Loop

目标不是继续让 planner 直接面对整份全局状态，而是引入一个 planner-facing 的局部工作上下文：

1. `active_surface`
   - 当前最该编辑/检查的局部对象
   - 可能是：
     - `pre_solid_base_sketch`
     - `path_rail`
     - `path_profile`
     - `loft_profile_stack`
     - `face_edit_window`
     - `edge_feature_window`
     - `post_solid_finish_window`
2. `surface_policy`
   - 当前 surface 允许哪些 action
   - 当前 surface 依赖哪些 evidence
   - 当前 surface 失败后允许怎样 repair / rollback
3. `expected_outcome`
   - planner 需要明确说出：
     - 这轮动作后预计会出现什么状态变化
     - 哪些 blocker 应该消失
     - 下一轮还需要哪类 evidence
4. `outcome_delta`
   - runtime 对比“计划中的 expected_outcome”和“实际 evidence”，形成下一轮显式反馈

### 2. Capability Packs

把专项逻辑从全局 runtime 中拆出，按 feature family 组织：

1. `sweep_pack`
2. `loft_pack`
3. `face_edit_pack`
4. `edge_feature_pack`
5. `pattern_pack`
6. `groove_pack`
7. `trim_pack`

每个 pack 负责：

1. active surface 选择
2. barrier 规则
3. repair policy
4. validation pack
5. relation families

### 3. 该保留的东西

以下方向不改：

1. artifact inspectability
2. `query_sketch/query_topology` 作为客观 state/query surface
3. `relation_base` 的 objective relation 思路
4. benchmark / probe 驱动验证

### 9.7 `L2_130` 的首轮失败根因：首段 `tangent_arc` 在 CadQuery 里没有 previous edge

2. 触发条件很明确：
   - path 的第一段就是 `tangent_arc`
   - CadQuery 的 `tangentArcPoint(...)` 需要前一条 edge 来推切向
   - 但 `moveTo(...)` 后栈里只有当前位置向量，不是 edge

我没有直接拍脑袋改，而是先在 sandbox 里做 API 对照：

1. `dir(cq.Workplane('XY'))` 里确认可用接口包括：

所以这类 case 的正确修法不是再给 planner 加提示，而是 runtime 自己做容错：

1. `_aicad_apply_path_segments(...)`
   - 新增 `has_drawn_segment`
2. 新规则：
   - 如果当前 arc 之前已经有 edge，继续用 `tangentArcPoint(...)`
   - 如果当前 arc 是 path 的第一段，且给了 `to + radius`
     - 则自动降级到 `radiusArc((next_x, next_y), signed_radius)`
3. `signed_radius` 从这些信号恢复：
   - `turn`
   - `direction = counter_clockwise / clockwise`
   - `clockwise=true/false`

对应验证：
