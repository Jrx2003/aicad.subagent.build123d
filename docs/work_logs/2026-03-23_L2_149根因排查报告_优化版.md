# L2_149 根因排查报告（易读版）

## 1. 一句话结论

`L2_149` 失败不是因为系统缺少`sweep`（扫掠）动作，也不是"选错几何面"这种常见问题。

真正的原因是：**系统缺少对"路径扫掠"这类任务的中间语义保护**。简单说就是——AI 画错了轨道，但系统没发现，还继续往下做，最后又错误地判断成功了。

---

## 2. 什么是"路径扫掠"（Path Sweep）

### 2.1 这道题到底要做什么

这是一道"L 形弯管"建模题：

1. **先画轨道（rail）**：在正面画一个 L 形的轨迹线
   - 50mm 直线
   - 90度圆弧（半径 30mm）
   - 再 50mm 直线

2. **在轨道端点建立参考平面**：在 L 的末端建立一个垂直于轨道的平面

3. **在参考平面上画截面**：画两个同心圆（圆环）

4. **沿轨道扫掠**：让圆环沿着 L 形轨道扫过去，形成一个空心的弯管

### 2.2 用软件怎么理解

**在专业 CAD 软件（如 FreeCAD）里**：
- 有专门的"路径草图"（path sketch）
- 有"端点参考平面"（endpoint datum plane）
- 有"扫掠"（pipe/sweep）功能
- 每个步骤都是明确的对象

**在我们的系统里**：
- `create_sketch`（创建草图）
- `add_path`（添加路径）
- `create_sketch(origin, plane)`（再创建一个草图，需要手动指定原点和平面）
- `add_circle`（画圆）
- `sweep`（扫掠）

**问题就在这里**：我们的动作是"通用"的，不像 CAD 软件那样有明确的语义保护。AI 需要自己推断：
- 轨道端点在哪里
- 端点的切向是什么
- 参考平面应该怎么放
- 圆环怎么和端点对齐

这些推断对 AI 来说容易出错。

## 3. 实际出了什么问题

### 3.1 关键发现：两种情况，同一个根本问题

**排查中发现，不同 run 中 AI 的表现不一致：**

**情况 A：AI 生成正确，但执行失败（`20260323_143335` run）**
```json
{
  "segments": [
    {"to": [50, 0], "type": "line"},           // ✓ 正确
    {"angle": 90, "radius": 30, "type": "tangent_arc"},  // ✓ 正确
    {"length": 50, "type": "line"}            // ✓ 正确
  ]
}
```

**情况 B：AI 生成时就错了（`20260323_145756` run）**
```json
{
  "segments": [
    {"type": "line", "end": [50, 0]},         // ✓ 正确
    {"type": "tangent_arc", "end": [80, 30], "radius": 30},  // ✓ 正确
    {"type": "tangent_arc", "end": [80, 80], "radius": 30}   // ❌ 错误！应该是 "line"
  ]
}
```

**为什么 AI 会把 "line" 写成 "tangent_arc"？**

看 requirement 原文：
> "another 50.0mm **tangent straight line**"

这里的歧义：
- 正确理解："tangent" 修饰 "straight line"（与圆弧相切的**直线**）
- AI 误解：把 "tangent" 当成了 "tangent_arc"（切线**圆弧**）

**这是自然语言到结构化参数的映射歧义。**

### 3.2 更严重的问题：无论生成对错，AI 都不知道自己错了

**两种情况下的执行结果几乎一样：**

| 场景 | AI 生成的段类型 | 执行后的 edges | 系统返回 | AI 是否知道错了 |
|------|---------------|--------------|----------|----------------|
| A | line-arc-line（正确） | 1（应该3段） | `success: true` | ❌ 不知道 |
| B | line-arc-arc（错误） | 1（应该3段） | `success: true` | ❌ 不知道 |

**执行日志中出现警告**：
```
BRepBuilderAPI_MakeWire::Error()
BRepBuilderAPI_DisconnectedWire
```

但系统返回给 AI 的是：
```json
{
  "success": true,
  "issues": [],
  "geometry": {"edges": 1}
}
```

### 3.3 AI 的"认知盲区"：它"看"不到自己的错

**第二轮 AI 收到的反馈**（`round_02_request.json`）：
```json
{
  "action_history": [
    {
      "action_type": "add_path",
      "snapshot_summary": {
        "geometry": {"edges": 1, "bbox": [22.7, 0, 35.3]},
        "issues": [],
        "success": true
      }
    }
  ],
  "query_geometry": {},      // ❌ 空的
  "query_topology": {},      // ❌ 空的
  "previous_error": null     // ❌ 没有错误
}
```

**AI 能"看到"的：**
- ✅ `success: true` → "我成功了"
- ✅ `edges: 1` → "有1条边"
- ✅ `issues: []` → "没有问题"

**AI 不能"看到"的：**
- ❌ `DisconnectedWire` 警告被吞掉了
- ❌ 预期是3段但只有1条边（没有对比信息）
- ❌ 自己生成的段类型与 requirement 的映射是否正确
- ❌ bbox 范围说明轨道没有形成完整 L 形

**根本原因在 `service.py` 的 `_aicad_apply_path_segments` 函数（第 3894-3979 行）**：

```python
def _aicad_apply_path_segments(target, start_point, segments, closed=False):
    # ... 处理逻辑 ...
    for segment in segments:
        if segment_type in {'line', 'tangent_line'}:
            path = path.lineTo(next_x, next_y)
        elif segment_type in {'tangent_arc', 'arc'}:
            path = path.tangentArcPoint((next_x, next_y), relative=False)
```

**`tangentArcPoint` 的问题**：
- 虽然代码计算了圆弧终点
- 但 CadQuery 的 `tangentArcPoint` 在内部可能没有正确保持线段连通性
- 当 OCC 判定为 `DisconnectedWire` 时，只保留了部分边

**但最致命的是**：无论执行成败，系统都返回 `success: true`，AI 根本无法区分"真的成功了"和"看起来成功了但实际上断了"。

### 3.4 AI 继续往下做——因为它以为自己对了

**第二轮 AI 的响应**（`round_02_response.json`）：
```json
{
  "planner_note": "Create profile sketch on vertical plane at path endpoint...",
  "actions": [
    {"action_type": "create_sketch", "origin": [80, 0, 80]},
    {"action_type": "add_circle", "radius": 10},
    {"action_type": "add_circle", "radius": 8}
  ]
}
```

**AI 在继续往下做，因为它以为：**
- 轨道已经建好了（端点在 `[80, 0, 80]`）
- 现在该画圆环截面了
- 然后该扫掠了

**AI 不知道**：
- 轨道其实只有1条边，不是完整的 L 形
- 端点 `[80, 0, 80]` 可能根本不在正确的位置
- 基于这个错误轨道继续，最终扫掠出来的肯定是错的

### 3.5 验证器（Validator）误判成功

**最终结果**：
- `converged = true`（系统认为收敛了）
- `validation_complete = true`（验证完成了）

**但实际的评估数据显示**：
- 生成的包围盒：`[50, 20, 20]`
- 正确答案的包围盒：`[20, 90, 90]`
- 生成结果的面类型：`圆柱面 + 平面`
- 正确答案的面类型：`圆柱面 + 平面 + 环面`（TORUS 就是弯管部分的曲面）

**关键证据**：正确答案有"环面"（TORUS），说明有弯头；生成结果没有"环面"，说明根本没有弯管，只是一截短直管。

这就是第三个根因：**验证器只检查了"有没有空心扫掠"，没检查"是不是沿着正确的 L 形轨迹扫的"**。

## 4. 这四个问题组合起来就是灾难

### 场景 A：AI 生成正确，但执行失败

```
AI 生成正确的 line-arc-line
    ↓
Runtime 执行出错（tangentArcPoint 没连上形成连续 wire）
    ↓
系统没报错（DisconnectedWire 被当成 success: true）
    ↓
AI 看不到 "DisconnectedWire"，只看到 "edges: 1, success: true"
    ↓
AI 以为轨道建好了，继续画圆环、扫掠
    ↓
验证器误判成功（看到有实体就通过了）
    ↓
任务结束，但结果是一截短直管，不是 L 形弯管
```

### 场景 B：AI 生成就错了

```
AI 误解 "tangent straight line" 为 "tangent_arc"
    生成了 line-arc-arc（最后一段应该是 line）
    ↓
Runtime 同样执行失败（两段圆弧难以形成连续 wire）
    ↓
系统同样没报错（返回 success: true）
    ↓
AI 看不到自己的生成错误，也看不到执行错误
    ↓
AI 以为轨道建好了，继续画圆环、扫掠
    ↓
验证器误判成功
    ↓
任务结束，结果同样完全不对
```

### 关键问题：两种场景下 AI 都无法自知

| 阶段 | 场景 A | 场景 B |
|------|--------|--------|
| AI 生成 | ✅ 正确 (line-arc-line) | ❌ 错误 (line-arc-arc) |
| 执行结果 | ❌ wire 不连通 | ❌ wire 不连通 |
| 系统反馈 | ❌ success: true | ❌ success: true |
| AI 看到 | ❌ "我成功了" | ❌ "我成功了" |
| AI 知道吗 | ❌ 不知道 | ❌ 不知道 |

**根本问题**：
1. **AI 无法验证自己的生成**：系统没有暴露 "生成的段类型与预期是否匹配"
2. **AI 无法感知执行失败**：`DisconnectedWire` 被吞掉，返回 `success: true`
3. **AI 无法对比预期与实际**：不知道 "应该是3段但只有1条边"意味着错误

**这就好比**：
- 老师让学生画一个"带尾巴的大象"
- 学生画了一头"带耳朵的大象"（他误解了"尾巴"）
- 或者学生画对了但被橡皮擦掉了尾巴
- 老师检查时说"嗯，有东西，很好"（没有指出"不是尾巴"）
- 学生以为画对了，继续涂色
- 最后交上去的是一头没有尾巴的大象

## 5. 这不是哪些问题（排除法）

### 5.1 不是"缺少 sweep 动作"

系统已经有 `sweep` 动作了，而且确实执行了。

### 5.2 不是"选错几何面"

这个 case 的问题发生在生成实体之前。还没到"选哪个面来加工"的阶段。

### 5.3 不是"评估器算错"

评估数据是准确的——生成结果确实只是一截短直管，不是 L 形弯管。

### 5.4 不是"提前知道答案来硬凑"

整个分析都是基于系统生成的日志文件，没有针对这个 case 写特殊代码。

---

## 6. 根本原因拆解（三层）

### 第一层：动作表达能力"够用但太薄"

我们的动作可以完成任务，但不像 CAD 软件那样贴近用户的真实操作。

**CAD 软件的操作思路**：
1. 画轨道
2. 在轨道端点建立参考平面
3. 在参考平面上画圆环
4. 管道扫掠

**我们的动作**：
1. `add_path`（添加路径）
2. `create_sketch(origin=[...], plane=YZ)`（创建草图，需要手动算原点和平面）
3. `add_circle`（画圆）
4. `sweep`（扫掠）

AI 需要自己做很多推断，容易出错。

### 第二层：证据系统让 AI 成了"盲人"

**AI 无法"看见"自己的生成与预期的差距**。

当前系统给 AI 的反馈是**状态反馈**（当前是什么样）：
- 边的数量：1
- 包围盒：[22.7, 0, 35.3]
- 几何类型：LINE
- success: true

**但 AI 需要的是对比反馈**（预期 vs 实际）：
| 维度 | 预期 | 实际 | AI 是否知道 |
|------|------|------|------------|
| 段类型序列 | [line, arc, line] | [line, arc, line] 或 [line, arc, arc] | ❌ 不知道 |
| 边数量 | 2-3 | 1 | ❌ 不知道"1是错的" |
| 连通性 | CONNECTED | DISCONNECTED | ❌ 不知道（被吞了） |
| bbox 范围 | 80x80 | 22x35 | ❌ 不知道预期是多少 |

**这就好比**：
- 老师让学生画一个"带尾巴的大象"
- 学生画了一头"带耳朵的大象"
- 老师检查时说"嗯，有东西，很好"（没有指出"不是尾巴"）
- 学生以为画对了，继续涂色

**AI 在 L2_149 中的"盲人"表现**：
- 它不知道 "tangent straight line" 应该映射到 `line` 而非 `tangent_arc`
- 它不知道 `edges: 1` 意味着"前两段没连上"
- 它不知道 `success: true` 掩盖了 `DisconnectedWire`
- 它不知道自己继续往下做（画圆环、扫掠）是在错误的基础上堆错误

### 第三层：验证器的停止策略太早

### 第三层：验证器的停止策略太早

验证器现在对以下几类任务检查得比较好：
- 打孔（hole）
- 切槽（groove/notch）
- 面编辑
- 边选择

但对"路径扫掠"这类任务还缺这些检查：
- 轨道的段类型是否正确
- 轨道是否连通
- 弯头是否真的存在
- 端点参考平面是否正确
- 圆环是否和轨道对齐

所以系统会出现危险情况：
- AI 生成了一个有空心的实体
- 验证器看到"像扫掠成功了"
- 给出`validation_complete=true`
- 但几何形状离题目要求很远

## 7. 系统角度，这是哪一类问题

### 7.1 每轮动作太多或太少？

**不是主因**。

这个 case 的轮次设计其实合理：
- 第 1 轮：画轨道
- 第 2 轮：画圆环
- 第 3 轮：扫掠
- 第 4 轮：验证

问题是：**轨道建完后没有被强制检查一次**，系统允许"坏轨道 → 直接继续画圆环 → 直接扫掠"。

### 7.2 几何面选择机制不合理？

**不是这个 case 的主因**。

这个 case 还没到"选哪个面"的阶段，问题是：
- 轨道端点参考平面的选择太隐式
- 圆环平面和轨道的对齐关系不够明确

### 7.3 校验效率低 / 校验不对？

**是，而且是主因之一**。

验证器关注了：
- 有没有实体
- 有没有体积
- 大方向是不是空心扫掠

但没关注：
- 轨道是不是"直线+圆弧+直线"
- 弯头是否真的存在
- 圆环是否沿轨道正确挂接

### 7.4 证据不充足？

**是，而且是主因之一，核心问题是"AI 无法自知"**。

这是最值得补的一块。

**当前反馈（AI 只能"摸"到这些）**：
```
edges: 1
bbox: [22.7, 0, 35.3]
success: true
```

**AI 能推断**："我有1条边，系统说成功，那应该没问题"

**AI 无法推断**：
- "1条边" vs "我生成的3段" → "意味着只有1/3成功了"
- "bbox 22x35" vs "预期 80x80" → "意味着 L 形没形成"
- "success: true" vs "stderr 里的 DisconnectedWire" → "意味着系统掩盖了错误"
- "tangent straight line" 应该映射到 `line` 而非 `tangent_arc`

**如果系统在第 1 轮后就能明确告诉 AI**：
```json
{
  "rail_inspection": {
    "generated_segments": ["line", "tangent_arc", "tangent_arc"],
    "expected_segments": ["line", "tangent_arc", "line"],
    "segment_match": false,
    "expected_edges": "2-3",
    "actual_edges": 1,
    "connectivity": "DISCONNECTED",
    "status": "ERROR"
  }
}
```

**那 AI 就知道**：
- 自己生成错了（第三段应该是 line 不是 arc）
- 执行也出错了（wire 没连通）
- 不能继续往下做（必须先修复 rail）

**这就是"对比反馈" vs "状态反馈"的区别**——让 AI 能"看见"自己的对错，而不是做"盲人摸象"。

### 7.5 动作不足？

**不是"完全缺动作"，但确实缺高语义的动作**。

底层原子动作勉强够，但缺面向"路径扫掠"任务的专门动作。

---

## 8. 最合理的修改方向

不建议做"如果题目包含 sweep 和 arc 就怎样"这种硬编码，应该按能力层补充。

### 第一优先级：增加"草图/轨道"的可查询语义

建议新增查询功能，覆盖：
1. 当前轨道的段序列
2. 每段的类型、端点、长度/半径
3. 轨道是开放的还是闭合的
4. 轨道是否连通
5. 起点、终点、终点切向
6. 如果是截面草图，返回其平面/原点/内外环关系

这一步价值最大，因为同时能解决：
- L2_149 的轨道问题
- 类似的弯管、沿路径槽、截面扫掠等问题

### 第二优先级：给验证器增加"路径扫掠"的语义约束

验证器需要新增针对扫掠/管道的检查，至少包括：
1. 要求的轨道段序列
2. 轨道连通性
3. 轨道端点参考平面是否解析成功
4. 扫掠的弯头是否存在
5. 圆环数量是否匹配
6. 圆环是否挂在轨道端点上

对于 L2_149 这种任务，验证器应该能在以下情况下拒绝收敛：
- 轨道不是"直线+圆弧+直线"
- 轨道不连通
- 结果里没有弯头证据
- 包围盒显示是短直管而非 L 形弯管

### 第三优先级：给`add_path`/`sweep`增加更强的约束或辅助动作

当前`add_path`允许 AI 自由拼段列表，太容易在"直线/切线直线/切线圆弧"之间出错。

建议两种路线：

**路线 A：保留通用`add_path`，但增加约束**
- 段类型必须来自有限集合
- 当题目明确写了"直线 + 90度切线圆弧 + 直线"时，系统可以明确要求这一序列
- 如果出现"圆弧+圆弧"但题目没有第二段圆弧，应触发阻塞，而不是继续扫掠

**路线 B：补充高阶辅助动作**
- `create_reference_plane_at_path_endpoint`：在路径端点创建参考平面
- `add_pipe_path`：添加管道路径
- `sweep_along_path(profile_ref=?, path_ref=?, hollow=true)`：沿路径扫掠（显式绑定截面和路径）

这类动作不是为了偷懒，而是把 CAD 软件中本来存在的"对象级语义"暴露给 AI。

### 第四优先级：路径任务增加强制的中间检查点

对以下几类任务，可以考虑自动插入检查点：
- `path_sweep`（路径扫掠）
- `loft`（放样）
- `multi_profile_pre_solid`（多截面预实体）

具体到 L2_149，最直接的策略是：
- 轨道建完后强制一个"仅检查轮次"
- 必须先确认轨道段序列/端点/切向正确
- 才允许创建圆环和扫掠

这比盲目减少 token 更有价值，因为它避免"错误累积后再做无效后处理"。

### 第五优先级：不要把`DisconnectedWire`当成成功

当前最危险的一点是：
- 底层几何内核（OCC）已经明确提示"不连通的线框"（DisconnectedWire）
- 系统仍将动作标记为`success=true`

对路径轨道/扫掠/放样这类任务，这类警告应该被结构化上报，至少变成：
- `success=false`（失败）
- 或`success=true but semantic_blocker=disconnected_path`（成功但有语义阻塞）

否则 AI 会误以为轨道已经建好了。

---

## 9. 总结：系统设计问题 vs 动作不足问题

### 系统设计问题占主导（按重要性排序）

1. **pre-solid 证据不足**（生成实体前的证据太少）
2. **验证器对路径扫掠语义盲区**
3. **runtime 对线框警告过于宽容**
4. **轮次切分不合理**（缺中间检查点）
5. **planner 的轨道段约束太弱**

### 动作不足是次级问题

不是说"现在完全做不了扫掠"，而是：
- 当前动作集合更像底层原语（primitives）
- 不像 CAD 用户真正操作的对象级命令

所以如果后续要补动作，更推荐补：
- 参考平面相关的专门动作
- 路径端点框架辅助动作
- 扫掠的截面和路径显式绑定

而不是盲目增加更多底层草图原语。

---

## 10. 落地优先级建议

如果只按性价比排：

### P0（最高优先级）

1. `query_sketch`或等价的"pre-solid 轨道/截面"语义查询
2. `validate_requirement`新增路径扫掠语义检查
3. `DisconnectedWire`提升为结构化失败/阻塞项

### P1（中等优先级）

4. `path_sweep`任务类型增加强制的轨道检查点
5. `add_path`对题目指定的段序列做更强约束

### P2（较低优先级）

6. `create_reference_plane_at_path_endpoint(...)`或等价辅助动作
7. `sweep(profile_ref, path_ref)`显式绑定截面和路径，而不是依赖隐式当前上下文

## 11. 最终判断

`L2_149` 并不说明"这个系统连简单扫掠都不会"，它说明的是：

**当前系统对"pre-solid path-driven feature"（生成实体前的路径驱动特征）的表达和验证都还不够强。**

这类问题如果不补，会继续在下列任务上重复出现：
- 弯管/线框扫掠
- 沿路径槽
- 轨道上的截面扫掠
- 端点框架依赖的放样/过渡
- 一部分 FreeCAD 风格的参考平面+截面+路径特征

所以对这个 case 的最终判断是：

### 关于 AI 的表现

**AI 的表现存在两种可能**：

1. **部分情况下 AI 确实生成错误**：把 "tangent straight line" 误解为 "tangent_arc"，导致第三段类型错误（line-arc-arc 而非 line-arc-line）
2. **部分情况下 AI 生成正确**：但由于系统反馈缺失，AI 无法确认自己的生成是否正确

**但无论是哪种情况，AI 都无法"自知"**：
- AI **不知道** 自己生成的段类型是否符合预期
- AI **不知道** 执行后 wire 是否连通
- AI **不知道** 返回的 `success: true` 是否真的意味着成功

### 关于系统的问题

1. **不是单个动作缺失导致完全无法生成。**
2. **也不是典型的拓扑面选择错误。**
3. **根本问题是多层级的**：
   - **语义层**："tangent straight line" 到 `line` vs `tangent_arc` 的映射歧义
   - **执行层**：`tangentArcPoint` 可能无法保持 wire 连通性
   - **反馈层**：`DisconnectedWire` 被当成 `success: true`
   - **证据层**：AI 无法获取 "段类型序列对比"、"边数量对比"、"连通性状态" 等关键信息
4. **验证器缺少对 path 语义正确性的检查。**

### 核心结论

**L2_149 是一个"AI 无法自知"的典型案例。**

无论 AI 生成正确与否，当前的反馈系统都让 AI 成了"盲人"——它看不到自己的生成与预期的差距，也看不到执行结果与预期的差距。

后续真正值得做的，不是对`L2_149`硬拟合答案，而是建立一套**让 AI 能"看见"自己对错的语义反馈层**。

## 附录：排查过程的完整发现

### 为什么之前会误以为"AI 写错了"

最初的排查（`20260323_145756` run）中，从执行结果反推，看到轨道只有1条边，以为是 AI 生成了错误的段序列（把最后一段直线写成了圆弧）。

**但查看多个 run 后发现情况复杂**：
- **Run A (`20260323_143335`)**：AI 发送的参数完全正确（`line` → `tangent_arc` → `line`）
- **Run B (`20260323_145756`)**：AI 生成的第三段确实是 `tangent_arc` 而非 `line`

**这说明**：AI 的表现不稳定，有时是生成错误，有时是执行错误。

### 为什么 AI 会生成错误的段类型？

Requirement 原文：
> "another 50.0mm **tangent straight line**"

**自然语言歧义分析**：
- 正确理解："tangent" 修饰 "straight line" → 与圆弧相切的**直线**
- AI 误解："tangent" 被当成独立类型 → **tangent_arc**

**这反映了当前 prompt 的问题**：
- 没有明确说明 "tangent straight line" 对应 `line` 类型
- 没有给出段类型与 requirement 词汇的映射表
- AI 需要自己推断词汇到参数的映射

### 为什么 AI 无法"自知"错误？

**AI 在第二轮收到的反馈**（简化）：
```json
{
  "action_history": [
    {
      "action_type": "add_path",
      "success": true,
      "snapshot_summary": {
        "geometry": {"edges": 1}
      }
    }
  ],
  "previous_error": null,
  "query_geometry": {},
  "query_topology": {}
}
```

**AI 能推断的**：
- `success: true` → "执行没有报错"
- `edges: 1` → "有1条边存在"

**AI 不能推断的**：
- ❌ "1条边" vs "3段路径" 意味着"只有1/3成功"
- ❌ "success: true" vs "DisconnectedWire" 意味着"系统掩盖了错误"
- ❌ 自己生成的段类型是否符合预期

**根本原因**：系统反馈是**状态反馈**（当前是什么样），不是**对比反馈**（预期 vs 实际）。

### 经验教训

1. **不能仅从执行结果反推 AI 的错误**——要看 AI 实际发送了什么
2. **区分"AI 规划错误"、"Runtime 执行错误"、"系统反馈错误"**——三者需要不同的修复策略
3. **AI 无法"自知"的 case 最危险**——因为系统让 AI 成了"盲人"，它不知道自己错了
4. **需要建立"对比反馈"机制**——告诉 AI "预期3段，实际1条边"，而不是只说"有1条边"
5. **Warning 级别的信息不应该被忽略**——`DisconnectedWire` 应该被结构化上报
