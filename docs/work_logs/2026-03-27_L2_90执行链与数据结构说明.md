## 1. 一句话结论

`L2_90` 是一个很适合拿来“入门看懂系统”的 benchmark case，因为它同时覆盖了下面几件关键事情：

1. benchmark 载入 case 并应用 canonical override
2. planner 根据 prompt 生成 typed CAD actions
3. runtime 按“整段动作历史重放”的方式执行动作
4. `query_topology` 把“顶面”“外边”这类语义缩成一组真实 face/edge refs
5. `hole(face_ref=..., centers=[...])` 这种面附着直接特征是如何执行的
6. 虽然最终几何已经完全正确，但 `validate_requirement` 仍然可能偏保守，不一定把 `validation_complete` 置成 `true`

这也是为什么 `L2_90` 很适合帮助新人同时理解：

1. benchmark 数据结构
2. runtime artifact 数据结构
3. typed CAD action 的执行逻辑
4. 为什么我们后来会把“关系反馈”和“validator”拆开

## 2. 这个 case 到底是什么

先不要看代码，先看 benchmark case 文件：

- [`benchmark/runs/20260327_084100/L2_90/benchmark_case.json`](../../benchmark/runs/20260327_084100/L2_90/benchmark_case.json)

这份文件是理解整个 case 的入口。

里面最重要的字段有：

1. `case_id`
   - 这里是 `L2_90`
2. `prompt`
   - 这是最终真正送给 runtime 的 requirement 文本
3. `prompt_field`
   - 这里是 `canonical_override`
   - 说明这条 prompt 不是直接从 CSV 的默认字段拿的，而是用了数据集 override
4. `ground_truth_step`
   - 对应这条 benchmark 的 GT STEP 路径
5. `expected_generated_step`
   - 这次 run 预期生成的 STEP 输出位置
6. `reference_notes`
   - 解释为什么这条 case 使用了 override

### 2.1 为什么 `L2_90` 用的是 `canonical_override`

这个逻辑不在 runtime 里，而在 benchmark 层。

对应代码：

- [`benchmark/run_prompt_benchmark.py`](../../benchmark/run_prompt_benchmark.py)

关键逻辑：

1. `BenchmarkCase` 数据结构定义在第 23 到 32 行
2. `_load_case_map(...)` 在第 300 到 336 行
3. 第 315 到 334 行会检查 case override
4. 如果 override 里有 `prompt`，就用 override 的 prompt，并把 `prompt_field` 记成 `canonical_override`

也就是说，`L2_90` 的“需求文本来源”不是 runtime 猜出来的，而是 benchmark 在装 case 时就已经决定好了。

这份 override 清单在：

- [`benchmark/sampled_10_per_L/canonical_case_overrides.json`](../../benchmark/sampled_10_per_L/canonical_case_overrides.json)

`L2_90` 在这里的含义是：

1. 原始自然语言 prompt 与 GT STEP 不一致
2. benchmark 通过 audited override，把 prompt 改成与 GT 一致的版本
3. runtime 本身并不知道“这是 override case”
4. runtime 只会看到当前这条已经修正过的 requirement

### 4.2 benchmark 如何创建这次 case 目录

看：

- [`benchmark/run_prompt_benchmark.py`](../../benchmark/run_prompt_benchmark.py)

关键行：

1. 第 49 到 52 行

2. 第 417 到 443 行

3. 第 455 到 463 行

也就是说：

> benchmark 层并不直接“生成 CAD”，它负责的是 case 选择、prompt 决定、目录组织、以及最终评估。

### 4.3 benchmark 如何把 case 交给 runtime

调用脚本：

- [`scripts/run_aci_live_probe.sh`](../../scripts/run_aci_live_probe.sh)

关键行：

1. 第 26 到 29 行
   - 决定 runs root
2. 第 37 到 67 行
   - 把 requirement、provider、model、max_rounds 等参数装成 CLI 参数
3. 第 69 到 72 行
   - 真正调用 `aicad-iter-run`

所以：

1. benchmark 层负责“挑 case + 准备环境”
2. live probe 负责“把 case 变成 runtime 请求”
3. `aicad-iter-run` 负责真正的 iterative loop

### 4.4 runtime 主循环在哪里

核心文件：

- [`src/sub_agent_runtime/runner.py`](../../src/sub_agent_runtime/runner.py)

最重要的阅读起点是：

1. 第 146 行开始的 round loop
2. 第 215 到 238 行
   - 组装 `relation_focus / relation_eval / latest_unresolved_blockers`
3. 第 258 行开始
   - 组装 `request_full`
4. 第 841 到 872 行
   - 每次动作执行都不是“只执行当前动作”，而是：
   - `clear_session=True`
   - 重放 `已成功历史动作 + 当前动作`
5. 第 905 到 938 行
   - 把 STEP 和 `preview_iso.png` 落盘到 `outputs/`
6. 第 1425 到 1439 行
   - 写 `run_manifest.json` 和 `summary.json`

这个设计非常重要，必须理解：

> runtime 不是维护一个不可见的 CAD 内核状态，而是每次都用“动作前缀重放”来恢复状态。

这也是为什么：

1. 每个 action request 都能独立 replay
2. `actions/round_XX_action_YY_replay_command.txt` 很重要
3. rollback 如果要做实，就应该在“动作前缀管理”这层做

Round 1 的动作请求文件在：

2. `action_02_request.json` = 包含 `create_sketch + add_circle`
3. `action_03_request.json` = 包含 `create_sketch + add_circle + extrude`

这正是前面说的“动作前缀重放”。

### 5.2 Round 2：找顶面，然后打六个孔

Round 2 的 planner 输出在：

- [`benchmark/runs/20260327_084100/L2_90/plans/round_02_response.json`](../../benchmark/runs/20260327_084100/L2_90/plans/round_02_response.json)

这一轮的动作是：

1. `create_sketch(face_ref=...)`
2. `add_circle(construction=true, diameter=70.0)`
3. `hole(diameter=15.0, depth=15.0, centers=[...])`

这一步最值得看的是：**顶面不是模型自己“记住”的，而是 runtime 通过 `query_topology` 帮 planner 缩出来的。**

#### 顶面 ref 是从哪里来的

看 trace：

- [`benchmark/runs/20260327_084100/L2_90/trace/events.jsonl`](../../benchmark/runs/20260327_084100/L2_90/trace/events.jsonl)

Round 2 在 planner 之前有一个事件：

1. `auto_query_topology_prefetched`

这说明 runtime 在 planner 出手之前，先自动补了一份 `query_topology`。

然后去看：

- [`benchmark/runs/20260327_084100/L2_90/prompts/round_02_request_full.json`](../../benchmark/runs/20260327_084100/L2_90/prompts/round_02_request_full.json)

你会看到里面已经带了当前 topology evidence。  
因此 planner 能直接输出：

- `face_ref: "face:3:F_31a1df43a9d6"`

然后实际动作请求在：

- [`benchmark/runs/20260327_084100/L2_90/actions/round_02_action_01_request.json`](../../benchmark/runs/20260327_084100/L2_90/actions/round_02_action_01_request.json)

可以看到 runtime 还给这个 `create_sketch` 动作补了：

1. `_face_candidate_hint: "outer_faces"`

这不是给用户看的业务字段，而是 runtime 帮助 translator 更稳地解析 face 的辅助 hint。

#### `hole` 为什么没有真的做 `pattern_circular`

这是这个项目的一个关键设计点。

虽然 requirement 文本里说了“circular pattern”，但 planner 最后给的是：

1. 一个 `hole`
2. `centers=[6 个中心点]`

也就是直接把 pattern 降成显式坐标列表。

这是有意为之，因为：

1. 显式中心点更稳定
2. 少一层 pattern seed replay 风险
3. 对 benchmark 几何更直接

对应实现约束主要在：

- [`src/sandbox_mcp_server/registry.py`](../../src/sandbox_mcp_server/registry.py)

里面对 repeated features、hole、pattern 有明确的 planner 提示与 capability 暴露。

#### `hole(face_ref=..., centers=[...])` 真实怎么执行

关键代码：

- [`src/sandbox_mcp_server/service.py`](../../src/sandbox_mcp_server/service.py)

`HOLE` 分支在第 10300 行开始。

这段代码最值得理解的点有 4 个：

1. 第 10307 到 10314 行
   - 先解析 `face_ref`
2. 第 10315 到 10332 行
   - 规范化 `centers`
3. 第 10344 到 10389 行
   - 如果有显式 `face_ref`，就从该 face 建工作平面
4. 第 10352 到 10364 行
   - 如果给的是 3D 点，会尝试转成 face-local 2D 坐标

这就是为什么这个系统可以稳定做：

1. 先通过 `query_topology` 拿到 face ref
2. 再在该 face 上钻孔
3. 并且允许用局部点列来直接完成重复孔

这个 case 的一个典型现象是：

1. benchmark 几何评估已经 `score=1.0`
2. 但 runtime summary 里的 `validation_complete` 仍然是 `false`

真实文件：

- [`benchmark/runs/20260327_084100/L2_90/summary.json`](../../benchmark/runs/20260327_084100/L2_90/summary.json)
- [`benchmark/runs/20260327_084100/L2_90/evaluation/benchmark_eval.json`](../../benchmark/runs/20260327_084100/L2_90/evaluation/benchmark_eval.json)

也就是说：

1. 从 benchmark 的几何签名角度看，它已经正确
2. 但从 runtime 内部的 validator 角度，它还没有被判定“可以停”

这正是当前系统的一个真实现象，不是 bug 证据文件写错了。

这条 case 很适合拿来理解：

> `evaluation passed` 和 `validation_complete=true` 不是同一个概念。

## 6. 这几个核心 JSON 到底怎么读

这一节是给第一次看这些 artifact 的人准备的。

### 6.1 `benchmark_case.json`

它描述的是“这次 benchmark case 的外部身份”。

主要回答：

1. 这条 case 叫什么
2. 用的是哪条 prompt
3. GT STEP 在哪
4. 这次生成的 STEP 应该写到哪
5. 有没有 override

### 6.2 `prompts/round_XX_request_full.json`

它描述的是“某一轮 planner 真正收到的完整上下文”。

以 Round 1 为例：

- [`benchmark/runs/20260327_084100/L2_90/prompts/round_01_request_full.json`](../../benchmark/runs/20260327_084100/L2_90/prompts/round_01_request_full.json)

第一次阅读时建议先看：

1. `requirements`
2. `action_history`
3. `query_snapshot`
4. `query_geometry`
5. `query_topology`
6. `requirement_validation`
7. `relation_focus`
8. `relation_eval`
9. `latest_unresolved_blockers`
10. `evidence_status`

这些字段组合起来，构成“planner 当前 round 的世界状态”。

### 6.3 `plans/round_XX_response.json`

它描述的是“planner 决定做什么”。

典型字段：

1. `actions`
   - 当前 round 要执行的 typed actions
2. `inspection`
   - 如果不做动作，就做哪些查询
3. `planner_note`
   - 人可读总结
4. `usage`
   - 这轮 LLM token 使用量

### 6.4 `actions/round_XX_action_YY_request.json`

它描述的是“送给 sandbox 的真实执行请求”。

最重要的点不是字段，而是结构：

1. 它总是带完整的动作前缀
2. `clear_session=true`
3. 也就是“从头重放到当前动作”

### 6.5 `actions/round_XX_action_YY_result.json`

它描述的是“这次完整重放后的真实执行结果”。

里面最常用的字段：

1. `action_history`
   - 到当前为止已经成功执行的动作链
2. `snapshot`
   - 当前快照
3. `execution_diagnostics`
   - 有没有 no-op、step file 是哪个、step hash 是什么
4. `output_file_contents`
   - STEP / JSON / PNG 的二进制内容大小

### 6.6 `queries/*.json`

它们描述的是“某一轮 runtime 或 planner 主动拿到的查询反馈”。

以：

- [`benchmark/runs/20260327_084100/L2_90/queries/round_04_inspection_only.json`](../../benchmark/runs/20260327_084100/L2_90/queries/round_04_inspection_only.json)

为例，第一次要重点看：

1. `resolved_query_topology_options`
   - 最终真正发给 `query_topology` 的参数
2. `topology.candidate_sets`
   - 例如：
     - `outer_faces`
     - `outer_edges`
3. `topology.relation_index`
   - 当前窗口里抽出来的关系底稿
4. `matched_ref_ids`
   - 当前真正被窗口覆盖到的 ref

## 7. `query_topology.relation_index` 在这个 case 里怎么读

这份文件是新同学最容易看懵的地方。

在 `round_04_inspection_only.json` 里，`topology.relation_index` 不是“校验结果”，而是“关系底稿”。

它的数据结构定义在：

- [`src/sandbox_mcp_server/contracts.py`](../../src/sandbox_mcp_server/contracts.py)

关键类：

1. `TopologyFaceEntity`：第 418 行
2. `TopologyCandidateSet`：第 563 行
3. `RelationEntity`：第 595 行
4. `RelationFact`：第 616 行
5. `RelationGroup`：第 639 行
6. `RelationIndex`：第 684 行
7. `QueryTopologyOutput`：第 1361 行

### 7.1 `candidate_sets`

它表示“planner 不必自己从几十个 face/edge 里瞎找，系统先帮它收缩成几组有语义的候选”。

例如 `L2_90` 里你能看到：

1. `outer_faces`
2. `outer_edges`

这些不是 CAD 内核原生概念，而是 runtime 基于 requirement 和几何位置归纳出来的候选组。

### 7.2 `relation_index.entities`

这里列的是“关系图里有哪些节点”。

在这个 case 里，典型节点就是：

1. 外圆柱面
2. 6 个小孔的圆柱面
3. 顶/底平面
4. 对应的 circular edges

每个节点最值得看的 `attributes` 是：

1. `geom_type`
2. `center`
3. `axis_origin`
4. `axis_direction`
5. `radius`

### 7.3 `relation_index.relations`

这里列的是“两两之间客观存在的关系事实”。

例如当前系统已经会抽的：

1. `concentric`
2. `coaxial`
3. `equal_radius`
4. `coplanar`

重要的是：

> 它不是“通过/失败”，而是当前窗口里客观存在的关系事实。

### 7.4 `relation_index.relation_groups`

这是在 pairwise relations 之上再归纳出的更高层组合。

对新人来说，它通常比单条 relation 更容易读。

## 8. 为什么这个 case 的 `relation_focus / relation_eval` 是 `null`

真实文件：

- [`benchmark/runs/20260327_084100/L2_90/queries/round_02_relation_focus.json`](../../benchmark/runs/20260327_084100/L2_90/queries/round_02_relation_focus.json)
- [`benchmark/runs/20260327_084100/L2_90/queries/round_02_relation_eval.json`](../../benchmark/runs/20260327_084100/L2_90/queries/round_02_relation_eval.json)

这不是丢数据，也不是 bug。

当前原因很简单：

1. `relation_focus / relation_eval` 的第一版实现主要在：
   - [`src/sub_agent_runtime/relation_feedback.py`](../../src/sub_agent_runtime/relation_feedback.py)
2. 它现在只重点覆盖：
   - sweep rail
   - annular profile
   - annular topology
3. `L2_90` 这个 case 没命中当前这一批 family
4. 所以这两层反馈是 `null`

这条 case 正好能说明：

> 当前系统里已经存在三层不同语义的反馈：

1. `query_topology / query_sketch`：客观状态
2. `relation_focus / relation_eval`：任务化关系反馈
3. `validate_requirement`：最后保险的语义校验

并不是每条 case 都必须同时用到这三层。

## 9. 为什么 `L2_90` benchmark 通过了，但 runtime 还没“停得很漂亮”

真实结果：

- [`benchmark/runs/20260327_084100/L2_90/evaluation/benchmark_eval.json`](../../benchmark/runs/20260327_084100/L2_90/evaluation/benchmark_eval.json)
- [`benchmark/runs/20260327_084100/L2_90/summary.json`](../../benchmark/runs/20260327_084100/L2_90/summary.json)

你会看到：

1. `evaluation.passed = true`
2. `evaluation.score = 1.0`
3. 但 `summary.validation_complete = false`
4. 而且后面又多跑了几轮 inspection-only

这说明现在系统还有一个重要现实：

1. 几何上已经正确
2. 但 validator 仍然偏保守
3. 所以 planner 会继续检查，而不是马上收敛

这也是为什么你后面要求：

> validate 应该作为最后保险，而不是主导整个修复流程

这个要求是合理的。`L2_90` 这条 case 本身就证明了这个问题是真实存在的。

## 10. 如果你要改这个 case，最容易踩坑的文件在哪

### 10.1 benchmark 装载层

文件：

1. 改坏 override 逻辑，会导致 `prompt` 与 GT 不一致

### 10.2 prompt 装配层

文件：

- [`src/sub_agent/codegen.py`](../../src/sub_agent/codegen.py)

风险：

1. 改 compact 逻辑可能让 planner 看不到关键 evidence

### 10.3 runtime 循环层

文件：

- [`src/sub_agent_runtime/runner.py`](../../src/sub_agent_runtime/runner.py)

风险：

1. 动作前缀重放逻辑不能破
2. `outputs/` / `queries/` / `plans/` / `prompts/` 必须保持全量可检查

### 10.4 CAD action 翻译层

文件：

- [`src/sandbox_mcp_server/service.py`](../../src/sandbox_mcp_server/service.py)

风险：

## 12. 术语表

1. benchmark case
   - 一条标准测试样例，包含 prompt、GT STEP、评估结果
2. canonical override
   - benchmark 层对原始数据问题做的修正，不属于 runtime 智能
3. planner
   - 大模型，负责生成 `actions` 或 `inspection`
4. typed CAD action
   - 结构化 CAD 动作，例如 `create_sketch`、`hole`
5. action prefix replay
   - 每次执行当前动作时，把之前成功动作一起重放
6. candidate set
   - 例如 `top_faces`、`outer_edges` 这样的候选收缩结果
7. relation base
   - 当前模型里客观存在的实体关系底稿
8. relation focus
   - 当前 round 最该关注哪些关系
9. relation eval
   - 预期关系与观测关系的偏差
10. validation_complete
   - runtime 自己认为已经可以结束，不等价于 benchmark 几何评估通过
