# 2026-04-13 Build123d 迁移实验与演示说明

这份文档的目标不是宣称“迁移已完成”，而是给今天的汇报提供一套可以直接讲的主线、demo 与证据。

## 今日汇报主线

今天最适合讲的不是继续堆 benchmark 数，而是讲清三件事：

1. 为什么我们要从旧 CadQuery 契约切到 Build123d
2. Build123d 到底在哪些 runtime 场景里更适合我们
3. 现在哪些 family 已经有明确收益，哪些 family 仍然没有收口

## 一句话结论

Build123d 对这个仓库最重要的价值，不是“语法换了一套”，而是它更容易让 runtime 把几何建模过程收束成显式、可 lint、可验证、可复用的 contract。

## 三个推荐 demo

### 1. 局部坐标 countersink 板

- 文件：`demos/build123d_foundations/demo_local_frame_countersink.py`
- 关联真实证据：`benchmark/runs/20260413_102600/L2_172`

这组 demo 要讲的不是孔本身，而是：

1. requirement 给的是 corner-based sketch coordinates
2. host 本体往往是 centered Build123d solid
3. runtime 必须显式完成坐标系平移
4. `Locations` 让重复特征的 placement 很自然

这说明 Build123d 的局部坐标表达，对我们的 explicit-anchor family 是正向帮助。

### 2. 半壳体与定向打孔

- 文件：`demos/build123d_foundations/demo_half_shell_directional_holes.py`
- 关联真实证据：`benchmark/runs/20260413_142700/L2_130`

这组 demo 推荐讲两层：

1. 半壳体可以在同一个 builder 里靠 `Mode.SUBTRACT + Mode.INTERSECT` 完成
2. `Plane.XZ.offset(0)` 的局部坐标是 `(x, z)`，不是含糊的“随便 offset 到 hole_z”

这正是我们最近补 preflight 和 prompt contract 的原因。

### 3. 壳体本体与 lip-fit 盖子

- 文件：`demos/build123d_foundations/demo_enclosure_body_lid.py`
- 关联真实证据：`test_runs/20260413_094502`

这组 demo 不是说 enclosure 已经做完了，而是用最清楚的方式告诉大家：

1. body 和 lid 最好分成两个显式 solid 讲
2. cavity 和 lip 最好显式 staging，而不是依赖模糊 shell 副作用
3. `Mode.PRIVATE` 对 staged geometry 非常有价值

也就是说，即使外部 enclosure probe 还没完全通过，我们也已经知道更正确的 Build123d 组织方式是什么。

## 真实 run 建议怎么讲

### 1. 成功样例：`L2_172`

- 目录：`benchmark/runs/20260413_102600/L2_172`
- 适合讲：
  - `planner_rounds=1`
  - `first_write_tool=execute_build123d`
  - `validation_complete=true`
  - evaluator `passed=true`

这个 case 的关键不是“运气好一次过了”，而是它证明：

1. 局部坐标映射问题一旦被讲清楚
2. Build123d 的 placement surface 就足够直接
3. runtime 可以一轮写完整并完成验证

### 2. 健康修复样例：`L1_218`

- 目录：`benchmark/runs/20260413_141000/L1_218`

这个 case 适合讲：

1. Build123d 不是只适合一轮简单 case
2. 当修复发生时，contract 仍然可以足够清楚
3. annular groove 这种路径，如果收束回更显式的 Build123d 几何表达，就更容易被 runtime 理解和验证

### 3. 诚实边界样例：外部 enclosure probe

- 目录：`test_runs/20260413_094502`

这个 case 最适合讲“我们为什么没有自欺欺人”：

1. 它没通过
2. 但它也没有再退化回黑盒错误
3. 首轮直接被 preflight lint 截住，并给出 repair recipe

这说明系统已经开始具备“对未知 prompt 做结构化失败”的能力。

## 推荐汇报结构

1. 先说今天的目标只是迁移实验和基础验证，不是宣布大功告成
2. 再讲 Build123d 对 runtime 的三类帮助：
   - local frame
   - builder boolean
   - safe staging
3. 用三个 demo 把这三类帮助讲清
4. 用 `L2_172`、`L1_218`、外部 enclosure probe 三个 run 做正反证据
5. 最后明确边界：enclosure family 仍未完全收口，但方向已经更清晰

## 可直接引用的结论

可以直接这样讲：

> 这次切到 Build123d，最大的价值不是换了个 CAD 库，而是把一些原来很难显式表达的 runtime 约束，变成了更清晰的 workplane、builder 和 staging contract。  
> 这让我们不仅更容易做对，也更容易把失败变成可修复、可验证、可泛化的系统能力。
