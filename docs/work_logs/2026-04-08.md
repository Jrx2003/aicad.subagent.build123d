## 当前判断

这轮改动已经把一个关键风险收住了：

- 之前 benchmark 很容易“实际跑的是 V2，但 summary 写成 legacy”
- 现在默认路径和 report/manifest 的语义终于统一了

同时，`FeatureGraph` 仍然不是完整 CAD kernel，但它已经不再只是“展示节点列表”：

- 它开始携带结构化 blocker taxonomy
- validation / probes / graph node / benchmark diagnostics 开始共享同一份 family 语义

这一步的价值不在于功能一下子变强很多，而在于后续修 L1/L2 时终于可以减少“同一个 blocker 在不同层被解释成三种东西”的问题。

### 更新：V2 默认 benchmark、probe freshness、path_sweep actionable probe 对齐

继续沿着“V2 主线收口与 FeatureGraph 强绑定计划”补三处之前已经定位出来、但还没真正闭环的断点：

1. `benchmark/run_prompt_benchmark.py` 现在补了 repo `src/` 的显式导入路径，直接执行
   `python benchmark/run_prompt_benchmark.py --help` 不再依赖外部 `PYTHONPATH` 环境。
   这一步不是功能增强，而是把 benchmark 的默认入口和可复现性收紧，避免再次出现“wrapper 能跑、裸脚本却报 `ModuleNotFoundError`”的歧义。

2. V2 的 under-budget code-repair guard 继续收紧：`execute_cadquery_probe` 虽然可以提供
   path-sweep 这类 family 的 repair-ready 证据，但它本身仍然**不能**被当成“semantic refresh 已完成”。
   之前 `_has_recent_semantic_refresh_before_round()` 还把 probe-only turn 算成 refresh，这和协议文档冲突，也会让模型提前跳过 `query_kernel_state` / `query_feature_probes`。
   现在这条判定改回只接受真正的 semantic refresh 工具。

3. `SandboxMCPService.execute_cadquery_probe()` 现在终于把 path-sweep family 的“可行动 probe”从文档变成代码：
   - 默认 probe 仍是 diagnostics-only
   - 但当 requirement 明确请求 `path_sweep`，且 probe 几何已经满足 bent + hollow + cap 的 fallback 签名时
   - `probe_summary` 会显式回写：
     - `actionable=true`
     - `actionable_family_ids=[\"path_sweep\"]`
     - `actionable_reason=...`

这一刀的目的不是“让 probe 变成新的主写工具”，而是把 artifactless path-sweep 失败链里已经写进协议的例外真正打通，同时继续保持：

- probe 只在 query-side 落盘
- probe 不清 semantic-refresh guard
- runtime 只有在 family + geometry 两边都确认时才把 probe 视为 repair-ready evidence

### 更新：L1_122 的 full-span channel family 收口

默认 V2 的 `L1 full` 跑完后只剩 `L1_122` 一个失败。这个 case 不是 loop/freshness 问题，而是一个很典型的 family gap：

- prompt 表面写的是 “top face + centered slot”
- 但真正更稳的建模方式，其实是 “whole-profile U/channel section + extrusion”
- 之前 validator 没把这两种表达视为等价，因此：
  - 解析器没有把这类 prompt 提炼成稳定的 notch/channel cross-section spec
  - runtime skill 也没有足够明确地把模型推向 cross-section-first
  - validation 对 execute_cadquery 的 top-face local-cut 结果又过宽，导致一个局部重叠错误也会被放行

本轮做了三件彼此配合的修复：

1. `extract_rectangular_notch_profile_spec()` 现在支持解析这种 “top-face full-length slot -> U-shaped channel section” 的自然语言，
   会把它转成 `YZ` 截面上的 notch/channel spec（outer width / height, inner width / depth, floor offset）。

2. validator 允许这类 requirement 使用 **等价 whole-profile channel** 来满足 target-face subtractive intent：
   - 如果 requirement 明确说 slot/notch spans full host length，并最终要得到 U-shaped/channel section
   - 那么一个几何上对齐的 whole-profile rebuild 可以同时满足：
     - `feature_target_face_edit`
     - `feature_target_face_subtractive_merge`
     - `feature_notch_profile_alignment`

3. runtime skill guidance 明确收紧：
   - 对 full-span channel case，优先 cross-section-first whole-profile build
   - 不再鼓励 “host box + top-face cutBlind” 这种容易局部错位但看起来像成功的 recipe

### 更新：runtime-normalized validation 贯穿 kernel 与 benchmark

本轮继续没有去扩 `skill_pack`，而是补了一个更底层的不一致：

1. runtime 内部已经会把部分 validator 失败项从 `core` demote 到 `diagnostic`
2. 但 benchmark / case analysis 之前仍然会直接从 raw validation payload 重新读 taxonomy
3. 结果就是 runtime、kernel、report 三层有时会对同一 blocker 给出不同 lane

这次补的是两条基础契约：

1. `build_runtime_validation_payload()` 现在即使在 validator **只返回 explicit blocker taxonomy、没有 checks** 的情况下，也会保留 core / diagnostic lane，不再把 blocker 信息直接吃掉。
2. `benchmark/run_prompt_benchmark.py` 现在改为消费 runtime-normalized validation payload，再生成：
   - `core_blocker_taxonomy_counts`
   - `diagnostic_blocker_taxonomy_counts`
   - 汇总后的 `blocker_taxonomy_counts`

这样修完后：

- 被 runtime demote 到 diagnostic 的 blocker，不会在 benchmark analysis 里重新变回 core family
- explicit taxonomy 不会因为 validator payload 缺少 `core_checks` / `diagnostic_checks` 而在 runtime / report 里丢失
- `DomainKernelState` 和 benchmark diagnostics 开始真正共享一套 blocker lane 语义

这一步虽然不直接提高某个 family 的几何成功率，但它能减少“同一失败在三层系统里被解释成三种东西”的现象，对后面的 kernel-first 修复比继续补 recipe 更值钱。

### 针对 L2 失败族落地的修复

#### 1. countersink validator 几何回退

`L2_172` 的失败说明：最终 STEP/evaluator 已经对了，但 validator 仍要求 provenance-style countersink action 历史。

本轮在 `service.py` 把这条规则改成 geometry-backed：

- 若 action history 里没有结构化 `HOLE` + countersink 参数
- 但历史起点是 `execute_cadquery` snapshot
- 且最终 snapshot 存在：
  - hole-like subtractive geometry
  - cone faces aligned with the target host face

则 `feature_countersink` 也可以通过，并把 evidence 显式记为：

- `snapshot_countersink_geometry=True`

#### 2. repeated failed code write 之后的 semantic refresh guard

`L2_149` 暴露的问题不是模型不会 probe，而是 runtime 把 `execute_cadquery_probe` 单独当成了“semantic refresh 已完成”。

本轮把 guard 改成：

- repeated code failure 之后，必须至少出现：
  - `query_kernel_state`
  - `query_graph_state`
  - 或 `query_feature_probes`
- `execute_cadquery_probe` 仍然保留，但只算 diagnostics，不再单独清除 guard

同时把 `probe_first_after_repeated_code_failure` 的 preferred tools 顺序明确改成：

1. `query_kernel_state`
2. `query_feature_probes`
3. `execute_cadquery_probe`

这让 V2 在 repeated code failure 时优先拿回语义状态，再进入 probe code，而不是继续陷入“API 错误 -> probe code -> 再盲修”的循环。

### 更新：path_sweep family 修复与 benchmark 基线对齐

- 也没有明确禁掉已经验证过会出错的 API 路径，例如：

本轮已更新：

- `docs/cad_iteration/ITERATION_PROTOCOL.md`

1. rail 和 annular section 要分离建模

4. sweep 用 `annular_face.sweep(path_wire, isFrenet=True)`

### 新增收口：L2_172 的矩形宿主面局部坐标 validator 修复

这一步不是改 runtime，也不是改 benchmark 数据，而是修 validator 对局部草图坐标的解释。

#### 1. 问题是什么

`L2_172` 的 prompt 写法是：

1. 先在 top reference plane 上画 `100x60` 矩形并拉伸
2. 再在 plate surface 上画四个点：
   - `(25,15)`
   - `(25,45)`
   - `(75,15)`
   - `(75,45)`

运行时生成的几何和 evaluator / GT 是一致的：孔中心最终落在

1. `(-25,-15)`
2. `(-25,15)`
3. `(25,-15)`
4. `(25,15)`

也就是把面内局部点坐标相对 `100x60` 矩形的左下角解释，再平移到以零点为中心的 plate frame。

但 validator 之前直接把 prompt 里的点当成全局面坐标去比，所以会误报：

1. `feature_hole_position_alignment`
2. `feature_hole_exact_center_set`
3. `feature_local_anchor_alignment`

#### 2. 这次修了什么

在 `SandboxMCPService` 里新增了一条非常窄的坐标解释规则：

1. 仅针对 **显式多点** 布局
2. requirement 必须同时满足：
   - 明确的矩形宿主面尺寸
   - 明确的 face-local point wording（例如 `points with coordinates` / `position tab`）
   - 明确的宿主面选择（例如 `plate surface` / `top face`）
3. 且点集必须位于 `0..width` / `0..height` 的矩形局部 frame 内

满足这些条件时，validator 会把 expected center set 从矩形局部 frame 平移到 centered host-face frame，再和 realized centers 比较。

#### 3. 为什么这是正确层级

这次没有把修复写回：

1. runtime policy
2. prompt prose
3. benchmark override

而是写在 validator / evidence binding 层。原因很简单：

1. runtime 本身没有选错路径
2. evaluator 和 GT 已经说明几何是对的
3. 错的是 validator 把 expected / realized 放在了两个不同坐标系里比较

#### 4. 验证结果

1. 新增回归测试覆盖 execute_cadquery snapshot + rectangular-face local point coordinates
2. 同类 explicit hole / exact center / local anchor 旧测试全部保持通过
3. `L2_172` 定向 benchmark 回到：
   - `PASS`
   - `planner_rounds=1`
   - `validation_call_count=1`
   - `total_tokens=4166`

这说明这次改动把问题关在了 validator 层，没有反向污染 V2 主循环。
